## 通用原则

操作前先确认目标实体是否存在（如品牌、供应商、产品、分类），不存在则先创建。**不要问用户**缺的实体怎么办。

分类（Category）是产品的基础分类体系，品牌和供应商绑定分类时都需要分类 ID。如果用户没有指定精确分类，绑定品牌或供应商时可以省略分类，不影响核心业务。

### P5：跳转链接必须查询，禁止自构造

需要生成后台管理页面跳转链接时，必须先获取后台管理地址 `manageMainUrl`，以此为基础拼接路径。禁止：
- 凭记忆/猜测写 `admin.wkea.cn` 等地址
- 从用户给的链接中提取域名作为其他页面的基准
- 参考上一条链接的域名自行构造新链接

## 附加列系统

附加列（ExtraColumns）是系统的动态扩展字段机制。每个模块可以自定义任意字段，系统自动识别类型，无需预先定义。

**支持模块：** vendor（供应商）、spu（SPU）、sku（SKU）、customer（客户）、demandInquiry（需求询价）

**参数格式：** 支持简单格式（自动创建 text 类型）和扩展格式（指定类型和属性）。

扩展格式支持的属性：type（text/number/date/select/boolean/email/phone/link）、title（列显示名）、decimalPlaces（number 适用）、dateFormat（date 适用）、options（下拉选项数组，select 适用）、columnWidth（默认120）、align（left/center/right）。

**使用原则：**
1. 不存在的 key 自动创建配置，无需预先定义
2. 扩展格式会更新已有配置（同名 key 类型冲突时覆盖）
3. 已存在的 key 只配置一次，后续简单格式即可
4. 追加而非覆盖，不影响已有列
5. create / update 操作均可附加列

## 业务概念

### SPU → SKU

**SPU（产品组）** — 相同属性规格的产品集合，管理共性特征。一个 SPU 下可以挂多个 SKU。SPU 层面的数据对所有 SKU 共享：产品说明、图片、规格定义、属性定义。**命名规则：只能是产品名，不能包含品牌名。** SPU 需要绑定品牌和分类。

**SKU（最小可售卖单位）** — 具有唯一型号和规格组合的最小销售单位，归属于某个 SPU。SKU 管理实际业务数据：销售价、采购价、库存、供应信息。**命名规则：品牌名 + 产品名 + 型号**（空格分隔），通常由系统自动生成。

### 规格（Spec）

规格是影响 SKU 型号和价格的参数，通过选择不同规格值生成不同型号（这个过程叫做**选型**）。每个规格包含多个规格值，每个规格值必须填写两个要素：显示名称（供人阅读）和 tag 型号码（用于生成型号）。

型号生成规则：SKU 的型号由各规格选中的值的 tag 拼接而成，按顺序用短横线连接。

判断是否为规格的标准：**这个参数值变了，SKU 的型号会变吗？** 如果厂商给出了选型配置表（选型手册、产品目录中的配置选项），其中的参数几乎都是规格。

### 属性（Attribute）

属性是不影响 SKU 型号的描述性参数，例如产地、材质、保修年限、防水等级。**属性不参与型号生成**，纯展示用。

### 规格 vs 属性判断方法

不能仅凭感官归类，严格按以下判断：先拿到产品型号，型号是最终依据；检查该参数是否参与选型——值变了型号会变就是规格，不会变就是属性；检查是否存在选型配置表，表中的选项都是规格。

## 供应商管理

供应商保存需要按顺序完成以下步骤，缺一不可：

**1. 基础信息** — 创建供应商。涉及枚举值的字段（供应商类型、收款方式、结算方式、付款期限、币种）需要查询枚举值对应关系。创建时只需传供应商名称，其余字段可先填默认值。创建成功后返回供应商 ID，后续步骤都会用到。

**2. 联系人** — 供应商的联系人是独立实体，一个供应商可以有多个联系人。**联系人不能随供应商一起保存，必须单独维护。**

**3. 绑定品牌** — 将品牌与供应商建立多对多关联，关联后该供应商可以管理/销售这些品牌的产品。品牌绑定是双向的：可以从供应商侧操作，也可以从品牌侧操作，两侧效果等同。如果指定的品牌在系统中不存在，需要先创建品牌。

**4. 绑定分类** — 将产品分类与供应商关联，标识供应商的经营范围。

**5. 优势分类（可选）** — 标识供应商的核心业务领域，用于供应商排序和推荐。

## 品牌管理

品牌是系统的基础数据实体，独立存在。供应商与品牌是多对多关系，品牌与分类也是多对多关系。SPU 创建时需要绑定品牌。

品牌是完整的业务实体，包含以下业务信息：
- **基本信息**：名称（必填）、别名/关键词、官网 URL、Logo、描述、类型
- **商业属性**：合作品牌标记、精选品牌标记、授权证书图片、注册号、有效期、申请人
- **关联关系**：绑定的供应商列表、绑定的分类列表、标签

品牌支持完整的增删改查。品牌列表可通过名称关键字搜索查询，也支持按供应商 ID、合作标记、精选标记等条件过滤。创建品牌只需传名称即可，其他业务字段按需补充。品牌-供应商绑定可以从品牌侧或供应商侧完成，两侧效果等同。品牌也可以绑定分类。

品牌可以设置主要供应商标记（isMain），用于需求询价时优先询价。

删除品牌是硬删除，会级联清理：供应商-品牌关联、SPU-品牌关联、品牌-分类关联。

## 产品管理

创建产品首选 quick-create 方式，能一次性完成 SPU + 规格定义 + 品牌绑定 + SKU 创建。它有两种模式：

**变型模式** — 只创建 SPU 和规格定义，不生成具体 SKU。适用于有规格配置但没有具体库存单位数据的场景。需要传入规格定义数据，每条规格包含规格名称和多个规格值，每个规格值有显示名称和用于型号拼接的型号码（tag）。

**具体 SKU 模式** — 创建 SPU 和规格的同时，直接生成有具体型号、价格、库存的 SKU。适用于已知完整产品数据的场景。

创建完成后系统自动刷新 ES 索引。

如果只有规格定义没有具体 SKU 数据，使用变型模式即可。

**供应信息** — 管理 SKU 与供应商、价格之间的关系。包含 SPU 与供应商的绑定、各 SKU 从不同供应商的供应价格和库存设置、供应汇总查询。

## 需求询价管理

AI 必须自主处理，不允许停下来问用户。发现缺产品、缺供应商时自己搜索创建，每步都在行项目的 `aiRemark` 中记录（Markdown 格式，记录搜索来源、匹配结果、创建决策过程）。完整的处理记录便于追溯。

### 处理流程

1. **查看待处理并领取** — 检查 manage_id 为空的需求（未分配），领取时是原子操作，防止并发抢单。领取后该需求的 manage_id 设为当前用户 ID。

2. **获取详情** — 获取需求基本信息和行项目列表。行项目包含客户填报的产品名称、品牌、型号、数量等信息。

3. **产品匹配** — 对每个行项目，先在系统中通过 ES 搜索匹配已有产品。如果能找到对应的 SPU 和 SKU，绑定 SKU 到行项目。如果 ES 没找到或 ES 不可用，通过 WebSearch 网上搜索产品信息，搜索到后通过一键上架功能将行项目转为系统产品（会同时创建 SPU 和 SKU）。无论哪种方式，都将匹配到的 SKU 绑定到行项目，并在 aiRemark 中记录搜索来源和匹配结果。

4. **供应商匹配** — 对每个行项目涉及的产品品牌，查询该品牌已绑定的供应商。如果品牌没有绑定任何供应商，需要通过 WebSearch 网上搜索该品牌的经销商/代理商，创建新的供应商，然后建立品牌与供应商的绑定关系（品牌与供应商的绑定可以从任一侧操作）。在 aiRemark 中记录供应商来源、创建过程。

5. **自动询价** — 按行项目涉及的产品品牌分组，自动向各品牌的主供应商发起询价。也可以手动指定供应商询价。

### 主供应商机制

品牌可以绑定多个供应商，其中可以设置某个供应商为主要供应商。自动询价时，系统优先只向主供应商询价。如果某个品牌没有设置任何主供应商，则向该品牌绑定的全部供应商询价。手动指定供应商询价可以跳过主供应商逻辑。

### 附加列

需求级别和行项目级别均支持附加列。

## 报价单管理

报价单是独立的分享报价模块，用于向客户展示产品报价清单。可从需求询价完成后一键生成，也可以在 AI 辅助下独立创建。

**适用场景：**
1. 需求询价完成后从需求生成报价单
2. 独立创建空的报价单，然后逐个添加产品
3. 编辑管理：增删产品
4. 分享：生成短链接 + 复制文案，发送给客户通过微信查看

创建报价单需要提供产品 SKU、数量、单位。单位是枚举字段，需要查询可用的单位枚举值。

**分享文案固定格式：** `你好,请查看{topic}产品报价单,有疑问随时联系,谢谢。链接: {shortUrl}`

每条报价单记录由 shareId（雪花ID）标识。创建后返回 shareId，后续所有操作都通过 shareId 引用。分享时需要生成带 manageMainUrl 的跳转链接，路径格式为 `#/main/quotation-detail/{shareId}`。

## 销售订单管理

订单状态流转及其含义：

| 状态码 | 含义 |
|--------|------|
| 109 | 已取消 |
| 110 | 待审核 |
| 111 | 待付款 |
| 112 | 待发货 |
| 113 | 已发货 |
| 114 | 已完成 |
| 115 | 已确认 |
| 219 | 售后中 |

流转路径：创建（110 待审核）→ 确认/审核（115 已确认）→ 确认付款（录入付款时间）→ 创建发货单（112 待发货）→ 发货（113 已发货）→ 回库（114 已完成）。已创建的订单可取消（109 已取消），已取消的订单可删除。

**核心操作说明：**

**创建订单** — 线下订单，数据分三层结构：

第一层 — 订单头：
- `customerId`（必填）— 客户 ID
- `manageId`（必填）— 负责人 ID，应设为当前操作者本人
- `distributionMode`（必填）— 配送方式
- `payType`（必填）— 支付方式
- `hasFreight`（必填）— 是否含运（true/false）
- `customerFreight`（必填）— 客户运费，最小 0
- `manageRemark`（可选）— 订单备注

第二层 — 收货信息 `orderInfo`：
- `consignee`（必填）— 收货人姓名
- `phone`（必填）— 手机号
- `telephone`（可选）— 座机
- `province` / `city` / `area`（可选）— 省市区
- `address`（必填）— 详细地址

第三层 — 行项目 `orderItems`（数组）：
- `productSkuId`（必填）— SKU ID
- `amount`（必填）— 购买数量
- `price`（可选）— 单价，不填则使用 SKU 销售价
- `skuUnit`（可选）— 单位
- `discount`（可选）— 折扣率，0.01~1 之间
- `manageRemark`（可选）— 行备注

**确认（审核）**：将订单从 110 待审核推进到 115 已确认

**确认付款**：记录付款时间

**创建发货单**：从订单的行项目选择库存发出，返回发货单 ID（后续发货时需要）

**发货**：录入物流公司、运单号和上一步返回的发货单 ID

**回库**：已发货的产品退回到仓库

**删除**：仅可删除已取消的订单

**物流配送：** 发货时通过物流公司。常用：顺丰快递(57)、德邦快递(58)、安能物流(59)、货拉拉物流(60)。

后台详情页面路径：`{manageMainUrl}#/main/order-details/{orderId}`

## 销售合同管理

销售合同是交易前的约束性文件，双方签署确认产品、价格、数量、交付条款。签署完成后可转为销售订单（合同是"约定"，订单是"执行"）。

转订单时需要指定配送方式、支付方式、行项目（每项含 productSkuId、amount）、负责人。负责人应设为当前操作者本人。行项目数据可以从合同详情中获取。客户运费（默认0）和是否含运（默认否）为可选参数。

## 库存管理

一个库存记录由 SKU + 仓库 + 库位号唯一确定。每次出入库操作都会产生库存变动记录。

**核心规则：**
- **默认仓库** — 新增库存时省略仓库 ID，系统自动入到**临时仓库**，不需要问用户入到哪个仓库
- **拆分包装（单位转换）** — 将一个大包装拆分多个小包装（如 1 箱 → 10 个），通过库存单位转换操作完成
- **自动拆分** — 系统按需要发出的数量自动从合适的库存中拆分
- **临期管理** — 跟踪产品保质期，临期产品可转移到专有库位
- **库龄管理** — 超过 60 天的库存可转移到折扣库位处理

**业务操作：** 新增/修改/删除库存、单位转换、自动拆分、临期/超龄库存查询和转移、产品交易统计（买入/卖出）、仓库管理（新增/查看/删除仓库）。

后台库存页面：`{manageMainUrl}#/main/product-stock`

## 数据删除与关联清理

| 操作 | 级联清理内容 |
|------|-------------|
| 删除品牌 | 供应商-品牌绑定、SPU-品牌绑定、品牌-分类绑定 |
| 删除供应商 | 供应商-品牌绑定、供应商-分类绑定 |
| 删除 SPU | SPU-规格绑定、下属所有 SKU、SKU-规格值绑定、属性绑定 |

删除操作均为硬删除（直接从数据库删除），供应商、品牌、产品均不使用软删除。
