# aiRemark 记录模板

## 模板使用规则（强制）

1. **区域固定** — 8 个区域按以下顺序排列，不可调换、不可跳过
2. **标题固定** — 每个区域的标题名（`## xxx`）不可修改
3. **表格固定** — 标注"固定表格"的，列名和列数不可增减。数据再多也用同一个表格结构
4. **空区处理** — 某区域确实无内容时，写"无"或"暂无"，必须保留标题
5. **信息完整** — 所有搜索结果、链接、参数、思考过程全部填入，禁止删减。信息越多越好

---

## 区域结构

### 区域 1：风险评估（永远第一）

```
## 风险评估

**匹配置信度**：高 / 中 / 低
**关键差异**：
- （每个 ⚠️ 和 ❓ 项一行。全部 ✅ 则写"无"）
**下一步**：（一句话，可执行动作）
```

**语义规则：**
- 置信度判断依据：全部 ✅ → 高；有 ⚠️ 但可解释 → 中；有 ❓ 无法确认 → 低
- 置信度为"中"或"低"时，第一行必须写"**存在差异需确认**"，禁止写"规格匹配"或"确认生产"
- 关键差异每行一句话——客户说了什么、实际搜到什么、差异是什么
- 下一步明确到可执行动作。如"先向客户确认材质术语和品牌偏好，再继续供应商开发"

---

### 区域 2：二次确认（必须紧跟风险评估）

```
## 二次确认

### 已了解的信息
- 产品类型：（从官方来源确认的产品类型）
- 官方参考：（品牌、系列、型号结构/编码规则——仅供内部参考）
- 客户已给参数：（客户已明确的参数项）
- 客户未给参数：（规格表中存在多选项但客户没给的）
- 来源：
  - [品牌官网产品页](URL)
  - [datasheet](URL)

### 需确认的问题
（以下是可直接复制发送给客户的确认消息）

1. 简洁的问题，从业务员口吻出发
2. 一问一行，不加品牌名（客户指定品牌除外）
```

**语义规则：**
- 二次确认不是硬阻塞——无论客户是否回复，确认后都继续处理
- **已了解的信息中每条结论必须附带来源链接**，让业务人员自行验证
- **客户未指定品牌 → 提问中禁止透露任何品牌信息**。问参数就只问参数，不加"参考 SMC xxx"、"根据 Festo xxx"等字样。客户指定了品牌 → 可以带上品牌参考信息
- 从所有 ⚠️ 和 ❓ 中筛选出核心的、影响选型的缺失参数，控制在 1-3 条
- 序号列表，从业务员口吻出发，简洁有礼貌，可直接复制发送
- 全部无需确认 → 写"无需确认"
  - 官方规格表中不存在的参数 → 不问
  - 品牌偏好 → 不问
  - 不影响型号选型的参数 → 不问
- 序号列表，从业务员口吻出发，简洁直观有礼貌。一条一行，可直接复制发送
- 全部无需确认 → 写"无需确认"

---

### 区域 3：供应商确认清单（紧跟客户清单）

```
## 供应商确认清单

1. 问题一
2. 问题二
```

**语义规则：**
- 从所有 ⚠️ 和 ❓ 中筛选出**需要供应商回答的问题**
- 判断标准：以下情况归入供应商确认——
  - 搜索到的产品规格不完整或有矛盾（如不同来源的参数不一致）
  - 需要供应商提供技术文档（datasheet、选型手册）
  - 价格、交期、MOQ 等商务信息缺失
  - 不同品牌同型号产品的规格差异需要品牌方确认
- 格式：**序号列表，从业务员角度出发，一句话一个确认点。不加客套话。可直接复制发送给供应商。**
- 同一疑点涉及多个供应商，合并为一条
- 全部无需供应商确认 → 写"无需向供应商确认"

### 区域 4：客户原文

```
## 客户原文

| 字段 | 内容 |
|------|------|
| 产品名称 | |
| 品牌 | |
| 型号 | |
| 数量 | |
| 单位 | |
| 原始文本 | |
```

**语义规则：**
- 数据来源：`demand items` 返回的 `productName`、`productBrand`、`productModel`、`quantity`、`productUnit`、`originalText`
- 全部字段必填，没有的写"未提供"
- 原始文本放 `originalText`（客户最初提供的一段话），没有则写 parse_demand 的输入文本

### 区域 5：产品研究（详细内容，放在确认清单之后）

> 以下内容较长，业务人员先看完上面的风险评估和确认清单，需要了解细节再往下翻。

#### 5a. 品牌发现（固定表格）

```
### 品牌发现

**搜索轮次：**

| 轮次 | 搜索词 | 结果摘要 |
|------|--------|---------|
| 1 | | |
| 2 | | |

**找到的品牌：**

| # | 品牌名称 | 地区 | 代表型号 | 官网地址 | 参考价格 |
|---|---------|------|---------|---------|---------|
| 1 | | | | | |
```

**语义规则：**
- 搜索轮次表：每次搜索记录一条，记录实际用过的关键词和发现的结果
- 品牌表列出所有找到的品牌，不遗漏
- 至少搜索 2-3 轮不同关键词组合
- 官网地址必须是可以打开的完整 URL

#### 5b. 逐个品牌验证

```
### 逐个验证

#### {品牌名} {型号}
- 信息来源：（官网 / 1688 / 代理商 / B2B平台 / 其他）
- 页面链接：（完整 URL）
- 规格参数：
  - 频率：
  - 外壳材质：
  - 防护等级：
  - 电容：
  - 波束角：
  - 最大电压：
  - 工作温度：
  - 其他参数：
- 用途说明：（产品页上写的用途描述）
- 文档链接：（datasheet / 规格书 / 选型手册 URL，没有则写"未找到"）
```

**语义规则：**
- 每个品牌独立一节，按搜索发现顺序排列
- 规格参数列举产品页/文档中提取到的所有参数项。不同产品参数名不同，有什么列什么。没有提取到具体数值的参数项删除该行
- 链接必须完整可访问

#### 5c. 规格逐条对比（固定表格）

```
### 规格逐条对比

| # | 客户需求项 | 客户描述值 | 搜索结果值 | 标记 | 分析说明 |
|---|-----------|-----------|-----------|------|---------|
| 1 | | | | ✅ / ⚠️ / ❓ | |
```

**语义规则：**
- 客户说了什么就拆成一行——产品名、频率、材质、用途、尺寸……逐项
- 搜索结果值填多个品牌在该项上的取值（合并相似值，标注来源品牌）
- 标记规则：
  - ✅ — 字面一致或数值落在标称公差范围内
  - ⚠️ — 字面不一致（即使是同一事物用不同词表达），或数值差异超出公差
  - ❓ — 搜索结果中找不到该项信息
- 分析说明：✅ 简述原因；⚠️ 写差异解释 + 影响评估 + 判断依据；❓ 写"搜索结果未包含此项信息"

### 区域 6：供应商匹配

```
## 供应商匹配

### 系统已有供应商
| 品牌 | 供应商名称 | 供应商ID | 是否主供应商 | 联系方式 |
|------|-----------|---------|------------|---------|

### 新开发供应商
| 品牌 | 供应商名称 | 搜索来源 | 企查查核验结果 | 创建结果 |
|------|-----------|---------|-------------|---------|
```

**语义规则：**
- 系统已有：`demand vendors-by-brand` 查到的结果
- 新开发：按 vendor guide 流程搜索、核验、创建的全过程
- 企查查核验结果写明：工商状态、风险信息、资质标签（高新/专精特新等）
- 无供应商时两个表都写"无"

---

### 区域 7：询价状态

```
## 询价状态

| 供应商 | 是否已询价 | 询价时间 | 报价状态 | 备注 |
|--------|-----------|---------|---------|------|
```

**语义规则：**
- 数据来源：`demand get` 返回的 quotedVendors
- 已询价标"已发送"，未询价标"待发送"
- 无供应商时写"无供应商，未发起询价"

---

### 区域 8：处理时间线

```
## 处理时间线

| 时间 | 步骤 | 操作内容 |
|------|------|---------|
```

**语义规则：**
- 记录每个关键操作节点（搜索、创建、询价等）
- 时间格式：YYYY-MM-DD HH:mm
