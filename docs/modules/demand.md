# 需求询价管理

## 1. 业务概念

**需求询价** — B2B 采购的核心入口。客户提需求 → 内部匹配产品/供应商 → 向供应商询价 → 供应商报价 → 转订单。

### 实体关系
```
DemandQuotation（需求询价主表）
  ├── DemandQuotationItem（行项目 1..N）
  │     └── skuId → SKU（可选绑定）
  ├── DemandQuotationDocInfo（供应商报价记录）
  │     └── DemandQuotationDocInfoData（报价明细）
```

### 状态
| 状态码 | 含义 |
|--------|------|
| 274 | 待处理 |
| 275 | 处理中 |
| 276 | 已完成 |
| 291 | 已取消 |

### 行项目关键字段
- 客户填写（只读）：`productName`、`productBrand`、`productModel`
- 后台编辑：`manageProductName`、`manageProductBrand`、`manageProductModel`
- 绑定产品：`skuId`
- 业务数据：`quantity`、`expectPrice`、`finalSkuPrice`、`grossMargin`
- 记录：`aiRemark`（AI 处理记录，Markdown 格式）

---

## 2. 前置条件

- 产品匹配前：需求已被当前 AI 领取（领取由 AI 自动处理）
- 供应商匹配前：行项目必须已绑定 SKU
- 询价前：行项目涉及的品牌必须有绑定的供应商

---

## 3. 判断依据

- **ES 搜索可用**（线上） → 优先用 ES 搜索已有产品
- **ES 不可用**（开发环境返回 500） → 走 WebSearch
- **品牌有供应商** → 直接询价；**品牌无供应商** → 必须网上搜索创建，不能跳过

---

## 4. 操作流程

### 4.0 创建需求（客户提出需求时）

当客户提出需求（文字描述或图片/文件），先解析需求内容，再创建到系统，然后自动进入处理流程。

#### Step 0.1：解析需求

使用 `parse_demand` 工具或 `demand parse` 命令将客户的需求文本/附件解析为结构化行项目：

**方式 A：使用 MCP 工具 `parse_demand`**
- 将客户文字描述/附件 URL 传入，返回结构化行项目
- text — 客户文字描述或表格文件 URL；customerRemark — 需求额外备注；annex — 附件 URL（逗号分隔）

**方式 B：使用 CLI 命令 `demand parse`**
- 先 `--help` 查看参数，支持文本输入、附件链接、JSON 输出等

`parse_demand` 会返回解析结果，每个行项目包含：产品名称、品牌、型号、数量、单位等。

#### Step 0.2：创建需求到系统

将解析结果通过 `demand create` 创建到系统。创建时需要补充渠道来源和客户来源：

1. **渠道来源** — 从以下选项中选择：
   - `淘宝-亿日`、`淘宝-维嘉`、`1688`、`微信`、`邮箱`、`线下`、`其他`
   - 用户没提供 → 默认「其他」，不需要停下来问
2. **客户来源**（可选）— 客户姓名，用户提供了就填。没提供不填，**不要停下来问**

用 `demand create` 创建（先 `--help` 查看参数），关键字段：
- channel-source — 从上面选项中选择

- items — 解析结果 JSON

#### Step 0.3：进入全流程处理

创建成功后，按下方 4.1 全流程处理步骤继续（获取详情 → 创建进度 → 产品匹配 → 供应商匹配 → 询价）。

> **AI 必须全程自主处理，不允许停下来问用户。**
> 每步在行项目的 `aiRemark` 字段记录处理过程（Markdown 格式）。

### 4.1 全流程处理（逐项完成）

**推荐使用 sub-agent 分阶段执行**，避免长流程丢失上下文。每个阶段开一个 sub-agent，传入需求 ID 和当前进度，sub-agent 内有完整 skills 文档 + CLI 工具，自主完成该阶段并更新进度。

| 阶段 | sub-agent 任务 |
|------|---------------|
| 产品搜索 | 系统搜索 → 品牌发现 → 官网验证 → 技术文档 → 逐条对比 → 上架/绑定 → 写 aiRemark |
| 供应商开发 | 系统查询 → 网上搜索 → 企查查核验 → 创建绑定 → 写 aiRemark |
| 询价 | 对比已询价 → 发询价通知 → 记录 aiRemark |
| 生成报告 | 收集数据 → 读取 report-template.html → 填充占位符 → 写入 HTML 文件 → 告知用户 |

每个 sub-agent 完成后退回摘要，主 AI 检查后继续下一阶段。

#### 前置准备：获取详情 → 创建进度

**1. 获取需求详情**
`demand get` 获取详情（先 `--help` 查看参数）
获取后得到：需求主题、客户信息、行项目列表、SKU 详情等全部上下文。

**2. 创建进度**
按 P2 原则，严格按下述模板创建，**不得自行删减、合并或重排任务**。用 `progress create`（先 `--help` 查看参数）：

```
任务模板（名称固定，不得删减合并）：
  1. 获取详情
  2.1 SKU型号搜索
  2.2 ES搜索引擎搜索
  2.3 缩短型号SKU搜索
  2.4 网上搜索产品
  2.5 验证并绑定SKU
  3.1 查品牌已绑供应商
  3.2 网上搜索供应商（品牌官网→B2B→行业平台，收集公司名称）
  3.3 企查查核验供应商（工商信息→荣誉资质→联系方式，每家过）
  3.4 创建供应商并绑定（通过核验的全部创建，绑品牌/分类/联系人）
  4.1 对比已询价列表
  4.2 向新供应商发询价
  5. 生成报告并记录完成

relation-type: 需求
relation-id: {需求ID}
link: {manageMainUrl}#/main/demandInquiryDetails/{需求ID}
```

**3. 重命名会话**
将会话名改为以下格式，方便查阅：
```
需求：{需求ID}-{需求主题}
```

#### 进度更新规则

进度在开始时一次性建好全部任务。**每执行一个操作，就立即更新一次进度**，不能攒到一批再做：

```bash
# ⚠️ 正确做法：每完成一个小步骤就更新
产品匹配步骤 → 立即 progress step
下一个产品搜索 → 立即 progress step

# ❌ 错误做法：全部做完才更新
全部搜索完 → 一次性更新多个进度
```

更新格式：用 `progress step`（先 `--help` 查看参数）
- summary — 记录实际操作中的思考和结果，即使没有也要如实记录（想了什么办法、查了什么、为什么没找到）
- links — 相关来源链接，多个逗号分隔

---

**Step 1：产品搜索与信息采集**

> 目标：确认产品真实存在、获取完整规格信息、决定是否绑定。

**在开始搜索前，先运行 `product guide` 了解系统的产品定义**（SPU/SKU 结构、规格系统、属性、快速创建等）。这对后续判断规格数据至关重要。

#### 搜索策略（按优先级，换方式不等于不存在，多种都要尝试）

**一、系统内搜索（先查有没有现成的）**

- **A. SKU 型号搜索** — 用 `product sku list`（先 `--help` 查看参数），用客户型号搜 model 字段
- **B. ES 搜索（线上环境）** — 用 `product spu es-search`（先 `--help` 查看参数），搜型号/品牌+型号。仅线上可用
- **C. 缩短型号再试** — "CVS100N" → "CVS100"，"RA/8125/50" → "RA/8125"

**系统搜不到 → 开始网上搜索。**

#### 搜索关键词方法论（重要）

**不要一次把客户所有描述堆进搜索框。** 拆成短关键词组合，每次搜 2-3 个核心词，多轮尝试。

以客户说"超声波换能器 用于气体，压敏陶瓷做的，频率 200K"为例：

**第一轮（核心词）：**
- `超声波换能器 200K` — 产品名 + 关键规格
- `200KHz ultrasonic transducer` — 英文更精准
- `超声波换能器 气体` — 产品名 + 用途

**第二轮（换角度，第一轮结果不够时）：**
- `200KHz 换能器 厂家` — 找生产商
- `ultrasonic air transducer 200KHz` — 英文换表达
- `压电陶瓷 超声波 气体` — 材质+产品+用途

**第三轮（搜到品牌后，针对品牌深挖）：**
- `聚德电子 DYA-200 datasheet` — 品牌+系列+文档
- `Unictron A200M7 spec` — 品牌+型号+规格

**原则：短词优先，多轮尝试，搜不到就换词。** 中文搜不到换英文，英文搜不到换拼音。不要一次丢 5 个关键词进去。

**二、先确定品牌（客户没给品牌时必须先做这一步）**

客户没指定品牌 → **不能跳过品牌直接搜产品**。必须先找到这个产品由哪些品牌生产：

- 搜 `{产品名} {型号} 厂家` / `{产品名} {型号} 品牌` / `{产品名} {型号} 生产商`
- 搜英文：`{产品名} {型号} manufacturer` / `{产品名} {型号} brand`
- 用 `web_fetch` 打开找到的页面，列出所有生产这个产品的品牌
- **至少找到 2-3 个品牌后，再进入下一步（逐个品牌去官网验证）**

**三、品牌官网产品页（逐个品牌验证）**

对上面找到的每个品牌，去官网确认产品是否存在：

- 搜 `{品牌} 官网` → 打开官网 → 找 **"产品中心"、"型号查询"、"Product Finder"、"产品目录"、"Downloads"**
- 搜 `{品牌} {型号}` → 直接定位产品详情页
- 从产品页提取：完整型号、规格参数表、技术参数、尺寸图、产品图片
- 官网列的所有规格信息**全部记录**
- **如果官网上找不到这个型号 → 说明这个品牌可能不生产这个产品，标记为"待确认"**

**四、技术文档 / 规格书 / 选型手册（产品搜索阶段就要关注）**

- 搜 `{品牌} {型号} datasheet pdf` / `{型号} 规格书` / `{型号} 选型手册`
- 搜 `{品牌} {系列名} catalog` / `{品牌} 产品目录 pdf`
- 搜 `{品牌} {型号} 说明书` / `{品牌} {型号} 安装手册`
- 提取：规格表、尺寸图、性能曲线、选型配置表（有规格变化时的全部选项）

> 找到 datasheet/选型手册时，不只确认产品存在，**必须提取规格数据**。有规格数据才能上架完整系列（参考 `product guide`），没有就只能上架单个 SKU。aiRemark 中专门记录找到的规格信息。

**五、B2B 平台 + 搜索引擎交叉验证**

- 1688/百度爱采购：搜 `{品牌} {型号}` → 看详情页参数和实物图
- icspec.com / allchips.com：电子元器件专用
- 搜索引擎：`{品牌} {型号} 参数` / `{型号} 替代` / `{品牌} {型号} 技术参数`

**搜索词多换：** 完整型号 → 缩短型号 → 品牌+产品名 → 型号单独 → 英文 → 拼音 → datasheet → catalog → 选型 → 参数 → 替代 → 说明书

**每次搜到结果必须用 web_fetch 实际打开页面查看内容，不能只看搜索摘要。**

#### 交叉验证与判断

找到产品信息后，**必须逐一对比验证**，不能凭感觉下结论：

1. **客户需求 vs 搜索结果**：客户说的每一项（产品名、品牌、型号、规格、材质、用途、频率……）逐条对比
2. **品牌确认**：找到的品牌是官方品牌还是经销商自己标的？→ 以官网为准
3. **至少 2-3 个独立来源**确认后，才能下结论

**在 aiRemark 中必须写明验证推理过程：**
- 客户要求什么 → 实际搜到了什么 → 每一项是否匹配 → 为什么确认/排除
- 不能只贴链接，要写清楚"为什么这个链接里的产品就是客户要的"

#### 匹配度标记规则（每条规格必须标记）

**客户说的和搜到的每一项都要打标记，不能跳过。**

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| ✅ 一致 | 完全一致或等价 | 数值相同、名称一致、规格匹配 |
| ⚠️ 有差异 | 不完全一致，需解释 | 用词不同可能同义、数值偏差、术语不统一 |
| ❓ 无法确认 | 搜不到对应信息 | 找不到规格、官网未列出此项 |

**⚠️ 标记的铁律：客户说的和搜到的不一样——哪怕只差一个字——也必须标 ⚠️，绝对不能标 ✅。**

工业品参数差一个字可能就是完全不同的产品。常见必须标 ⚠️ 的情况：

- 客户说"压敏陶瓷"，搜到"压电陶瓷" → ⚠️ 完全不同的物理效应和材料！压敏=电压→电阻变化（电路保护），压电=机械力↔电压转换（超声波换能器）——**虽然超声波换能器确实用压电陶瓷，但术语不同必须标注并解释**
- 客户说"不锈钢304"，搜到"不锈钢316" → ⚠️ 不同牌号，耐腐蚀性不同
- 客户说"200KHz"，搜到"200KHz±5%" → ✅ 数值落在公差范围内，但需注明公差
- 客户说"气体用"，搜到"液体/气体两用" → ✅ 涵盖气体用途，但需注明也适用液体

**每个 ⚠️ 标记必须附带三要素：**
1. **差异大白话解释** — 让不懂技术的业务人员也能看懂区别
2. **影响评估** — 这个差异会导致什么后果（不影响使用？可能不是同一个产品？需要确认？）
3. **判断依据** — 为什么 AI 认为可以接受 / 不能接受

**示例（正确）：**
```
- 材质：客户说"压敏陶瓷" → 搜索到的产品均使用压电陶瓷(PZT) ⚠️ 术语有差异
  - 解释：压敏陶瓷是压敏电阻材料，用于电路保护；压电陶瓷是PZT，用于超声波换能器。
    两者是完全不同的物理效应（压阻效应 vs 压电效应）。
  - 影响：超声波换能器的工作原理决定必须用压电陶瓷，客户大概率混淆了术语
  - 判断：功能上合理（换能器=压电不是压敏），但需向客户/供应商确认
```

**示例（错误，绝不允许）：**
```
- 压敏陶瓷 → 官网说明采用压电陶瓷(PZT) → ✅ 匹配
  ↑ 这完全跳过了术语差异！业务人员看不懂为什么"压敏"="压电"！
```

#### 确认标准

| 情况 | 操作 |
|------|------|
| 同品牌 + 同型号完全一致 | **必须用 `demand update-item` 直接绑定**，不能只记 aiRemark |
| 客户没指定品牌，搜到 WKEA 同型号 | 绑定 |
| 客户没指定品牌，搜到其他品牌（Festo/SMC等） | **不绑定**。只记 aiRemark，上架时按客户原文创建，品牌默认 WKEA |
| 品牌或型号不完全一致 | 只记 aiRemark，不绑定 |
| 搜不到 | 如实记录，继续供应商匹配 |

> 客户给了品牌且网上验证一致 → **绑定是强制的，不能跳过。**

#### 供应商确认清单（每次必须生成）

完成产品搜索后，将所有标记为 ⚠️ 或 ❓ 的差异点整理成**供应商确认清单**，方便业务人员直接复制发送给供应商确认。

**格式要求：**
- 每个问题一句话，直截了当
- 一个问题只问一件事
- 不加"您好"、"请问"等客套话——纯干货，可直接复制发送

**生成规则：**
- 每个 ⚠️ 差异 → 至少 1 个确认问题
- 每个 ❓ 无法确认 → 1 个询问问题
- 全部 ✅ → 写"无需确认，所有参数完全匹配"

**格式（追加到 aiRemark 结论后面）：**
```markdown
## 供应商确认清单

| # | 疑点 | 可直接复制发给供应商 |
|---|------|---------------------|
| 1 | 客户说压敏陶瓷，搜索显示超声波换能器用压电陶瓷 | 请确认该产品是否使用压电陶瓷（PZT）材料？客户描述为压敏陶瓷，请说明材料类型 |
| 2 | 频率200KHz，不同品牌标注±5%公差 | 200KHz频率的公差范围是多少？是否有不同精度等级可选？ |
```

#### 上架并绑定

**系统已有匹配 SKU** — 用 `demand update-item`（先 `--help` 查看参数）绑定

**系统没有** — 先看 `product guide` 判断有没有规格数据：
- 搜索中找到了选型手册/datasheet 里有规格变化（如缸径32/40/50/63）→ 用 `product quick-create --specs` 建完整系列
- 只有单个型号无规格信息 → 用 `demand simple-create-product` 一键上架
- 客户指定了品牌且验证一致 → 创建后立即用 `demand update-item` 绑定

#### aiRemark 记录规范（强制）

**每搜索一次就记录一次，不能攒到最后一起写。** 业务人员可能不熟悉产品领域，AI 必须详细解释推理过程，让人看得懂"为什么这个产品就是客户要的"。

每个行项目的 aiRemark 必须包含以下部分。**最前面必须先写结论**，让业务人员一眼看到关键信息再往下看细节：

**⓪ 结论（最前面，先写这个）**
- 最终确认的产品名、型号、品牌
- 客户未指定品牌 → 标注"系统默认 WKEA"
- 参考价格范围（如有）
- 信息来源（列出最关键的产品页 URL）
- 一句话总结：产品是否存在、下一步做什么

**① 品牌发现过程**
- 搜了什么关键词找品牌（厂家 / 品牌 / manufacturer）
- 找到了哪些品牌生产这个产品（至少列出 2-3 个）
- 每个品牌的官网地址
- 为什么认为这些品牌是相关的

**② 逐个品牌验证过程**
- 对每个品牌，在官网上搜了什么、找到了什么产品页
- 产品页原文 URL（必须附上）
- 产品页上的完整规格参数
- 是否有 datasheet / 选型手册 / 产品目录 / PDF（附链接）

**③ 客户需求 vs 搜索结果逐条对比（核心）**

### 4.2 供应商报价与价格管理

> 以下为业务参考，不属于全流程处理的固定步骤。AI 根据实际场景按需查阅使用。

#### 查看报价详情
- 用 `demand vendor-quotes`（先 `--help` 查看参数）

#### 报价评估与建议

当业务人员说"看看报价情况"时，AI 应主动整理报价数据并给出建议：

**1. 展示报价对比**
- 用 `demand vendor-quotes` 拉取全部供应商报价
- 将数据整理为对比表格：供应商名称 | 资质 | 单价 | 交期 | 库存 | 发货地 | 报价总金额 | 是否完成
- 同时展示供应商资质（tags）、报价有效期等辅助信息

**2. 给出采纳建议**
- 评估标准（按优先级）：价格低 > 交期短 > 已有采购记录 > 供应商资质（专精特新/高新企业等优先）
- 综合排序后推荐 1-2 家最合适的供应商，说明理由
- 明确告知：**所有报价均会保存，业务员只需决定采纳谁**

**3. 等待业务员决策**
- 问"你想采纳哪家供应商？"或直接指定供应商 ID
- 业务员确定后，执行以下操作：

**4. 保存全部报价 + 设置采纳**
- 所有供应商报价 → 用 `demand save-price` 逐条保存（保留全部报价记录）
- 采纳的供应商 → 用 `product supply set-master` 设为主供应商价格

#### 保存供应商报价

**记录报价（不改售价）** — 用 `demand save-price`（先 `--help` 查看参数），isMaster=false

**设为主供应商价格** — 用 `product supply set-master`（先 `--help` 查看参数）
- 改写 SKU 默认售价、采购价、交期，同步更新维嘉替代品折扣
- **智能重定向：** 如果该 SKU 已绑定完全替代品（isFullReplace），价格自动设到替代品上

#### 替代品与停产

以下工具各自独立，按需组合。命令均需先 `--help` 查看参数。

**替代关系** — `product sku replace`
- `add` / `list` / `remove` 子命令。完全替代加 full-replace 参数。只管两个 SKU 关系，不管价格和停产。

**创建替代产品** — `product quick-create` 或 `spu create` + `sku create`

**停产标记** — `product spu update` 的停产参数。值 0=停产无替代，SPU ID=停产有替代。

**操作示例：**
- 供应商说"第N个产品停产，用xxx替代" → items 定位 itemId/skuId → quick-create 建替代品 → sku replace add 绑关系 → set-master 设价（自动走到替代品）→ spu update 停产参数
- 供应商说"可用xxx替代，没停产" → sku replace add 绑关系 → set-master 设价
- 供应商说"停产，没替代" → spu update 停产参数值 0

## 5. 边界情况

- **ES 不可用**（开发环境返回 500）→ 必须走 WebSearch，不能跳过产品匹配
- **品牌无供应商** → 必须网上搜索创建，不能报错退出
- **供应商没有邮箱/电话** → 不能创建，继续深挖找到联系方式
- **多个供应商匹配** → 优先选有授权证书、工商状态正常、有官网的
- **供应商异步报价** → AI 发出询价后等待供应商响应（异步），需轮询检查报价状态
- **一键上架** — 内部自动查重（按品牌+型号），重复则直接绑定
- **主供应商机制** — 品牌可设置主要供应商，自动询价时优先询价
