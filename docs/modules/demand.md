# 需求询价管理

## 1. 业务概念

**需求询价** — B2B 采购的核心入口。客户提需求 → 内部匹配产品/供应商 → 向供应商询价 → 供应商报价 → 转订单。

### 实体关系
```
DemandQuotation（需求询价主表）
  ├── DemandQuotationItem（行项目 1..N）
  │     └── skuId → SKU（可选绑定）
  ├── DemandQuotationDocInfo（供应商报价记录）
  │     └── DemandQuotationDocInfoData（报价明细）
```

### 状态
| 状态码 | 含义 |
|--------|------|
| 274 | 待处理 |
| 275 | 处理中 |
| 276 | 已完成 |
| 291 | 已取消 |

### 行项目关键字段
- 客户填写（只读）：`productName`、`productBrand`、`productModel`
- 后台编辑：`manageProductName`、`manageProductBrand`、`manageProductModel`
- 绑定产品：`skuId`
- 业务数据：`quantity`、`expectPrice`、`finalSkuPrice`、`grossMargin`
- 记录：`aiRemark`（AI 处理记录，Markdown 格式）

---

## 2. 前置条件

- 产品匹配前：需求已被当前 AI 领取（领取由 AI 自动处理）
- 供应商匹配前：行项目必须已绑定 SKU
- 询价前：行项目涉及的品牌必须有绑定的供应商

---

## 3. 判断依据

- **ES 搜索可用**（线上） → 优先用 ES 搜索已有产品
- **ES 不可用**（开发环境返回 500） → 走 WebSearch
- **品牌有供应商** → 直接询价；**品牌无供应商** → 必须网上搜索创建，不能跳过

---

## 4. 操作流程

### 4.0 创建需求（客户提出需求时）

当客户提出需求（文字描述或图片/文件），按以下流程一气呵成：解析需求 → 创建到系统 → 自动进入全流程处理。**三步连贯执行，中间不询问、不确认、不等待。**

#### Step 0.1：解析需求

使用 `parse_demand` 工具或 `demand parse` 命令将客户的需求文本/附件解析为结构化行项目：

**方式 A：使用 MCP 工具 `parse_demand`**
- 将客户文字描述/附件 URL 传入，返回结构化行项目
- text — 客户文字描述或表格文件 URL；customerRemark — 需求额外备注；annex — 附件 URL（逗号分隔）

**方式 B：使用 CLI 命令 `demand parse`**
- 先 `--help` 查看参数，支持文本输入、附件链接、JSON 输出等

`parse_demand` 会返回解析结果，每个行项目包含：产品名称、品牌、型号、数量、单位等。

#### Step 0.2：创建需求到系统

将解析结果通过 `demand create` 创建到系统。创建时需要补充渠道来源和客户来源：

1. **渠道来源** — 从以下选项中选择：
   - `淘宝-亿日`、`淘宝-维嘉`、`1688`、`微信`、`邮箱`、`线下`、`其他`
   - 用户没提供 → 默认「其他」，不需要停下来问
2. **客户来源**（可选）— 客户姓名，用户提供了就填。没提供不填，**不要停下来问**

用 `demand create` 创建（先 `--help` 查看参数），关键字段：
- channel-source — 从上面选项中选择

- items — 解析结果 JSON

#### Step 0.3：进入全流程处理

创建成功后，按下方 4.1 全流程处理步骤继续（获取详情 → 创建进度 → 产品匹配 → 供应商匹配 → 询价）。

> **AI 必须全程自主处理，不允许停下来问用户。**
> 每步在行项目的 `aiRemark` 字段记录处理过程（Markdown 格式）。

### 4.1 全流程处理（逐项完成）

**主 agent 不亲自跑流程，只做调度。** 实际执行交给 sub-agent，避免长流程灌满主 agent 上下文导致后期偷懒、漏写、编造。通用 sub-agent 执行原则见 SKILL.md P14，本节只讲需求处理的调度细节。

#### 调度策略

| 阶段 | Sub-agent 类型 | 启动粒度 | 依赖 |
|---|---|---|---|
| 产品搜索 | product-research | **每个行项目一个**，并行启动 | 无 |
| 供应商开发 | vendor-development | 每个品牌一个（行项目级数据），并行启动 | 2.8 完成 |
| 询价 | quotation | 每个行项目一个 | 3.4 完成 |
| 生成报告 | report | 整个需求一个 | 4.2 完成 |

**关键原则：**
- **主 agent 上下文不持有搜索结果、aiRemark 全文** — 只持有调度状态
- **多行项目并行跑** — 5 个产品开 5 个 sub-agent 同时执行 2.1-2.8
- **依赖关系由主 agent 控** — Phase 1 全部完成才启动 Phase 2
- **异常可继续** — 二次确认不阻塞其他行项目
- **全局 progress 由主 agent 维护** — sub-agent 不直接调 `progress step`。每批 sub-agent 全部返回后，主 agent 推进对应的全局步骤并合并 summary（例：第 1 批 3 个 product-research 完成 2.1-2.8，主 agent 推 8 个 step 各 +1 次，每个 summary 写"第 1 批 N 个行项目完成 2.X"）

#### 行项目合并规则

避免 10 个同品牌行项目开 10 个 sub-agent 重复跑品牌发现/官网验证：

| 条件 | 处理 |
|---|---|
| 同品牌 + ≤ 5 个行项目 | **合并到 1 个 sub-agent**，串行处理批内每个行项目 |
| 同品牌 + > 5 个行项目 | 分批，每批 5 个 |
| 不同品牌 | 各自开 1 个 sub-agent |
| 涉及二次确认 | 不合并，独立 sub-agent 单独处理 |

#### Sub-agent 启动 Prompt 模板

主 agent 用以下模板启动 product-research sub-agent（参数化即可）：

```
你是一个 WKEA 需求处理的 sub-agent。任务：对单个行项目执行产品研究流程。

【必读文档】
- 通用 sub-agent 原则：SKILL.md P14
- 业务：docs/modules/demand.md 的 2.1-2.8 章节
- aiRemark 模板：docs/modules/demand-aiRemark-template.md

【任务参数】
- 任务类型：product-research
- 需求 ID：{demandId}
- 行项目 ID：{lineItemId}
- 进度 ID：{progressId}
- 起始步骤：{currentStep}（重试时为第一个未完成步骤）
- 重试次数：{retryCount}
- 上次错误：{lastError}

【输入数据】
{demandGet + items 拉到的该行项目完整数据，含 productName/brand/model/quantity}

【执行规则】
1. 按 SKILL.md P14 返回结构化结果
2. 每完成一步立即 progress step
3. 写 aiRemark 前先读模板
4. 严格遵守 SKILL.md P13（禁止用训练知识编造）
5. 二次确认按 P14 异常处理：写 aiRemark + 标记 abnormal + 继续往下能做的部分
```

**vendor-development / quotation / report 的模板结构相同，参数和文档链接不同。**

#### 主 agent 调度伪代码

```
# Phase 1: 产品研究（按行项目并行，每批最多 3 个）
demand = demand_get(demand_id)
items = demand_items(demand_id)
progress = progress_create(demand_id, items)

# 分批：每批 3 个行项目，跑完一批再起下一批（详见 SKILL.md P14）
for batch in chunks(items, 3):
    agents = [spawn_subagent(item, batch) for item in batch]
    await_all_with_retry(agents, max_retry=3)

# Phase 2: 供应商开发（依赖 Phase 1）
对需要新供应商的行项目 → spawn vendor-development sub-agent
等待返回

# Phase 3: 询价 + 报告（串行）
spawn quotation sub-agent
spawn report sub-agent
```

> 多个行项目时，每个行项目的搜索、验证、aiRemark 写入必须独立完成。禁止"第一个验证过了，其余的跳过"。**主 agent 通过 progress get 监督，不读 aiRemark 全文。**

#### 前置准备：获取详情 → 创建进度

**1. 获取需求详情**
`demand get` 获取详情（先 `--help` 查看参数）
获取后得到：需求主题、客户信息、行项目列表、SKU 详情等全部上下文。

**2. 创建进度**
按 P2 原则，严格按下述模板创建，**不得自行删减、合并或重排任务**。用 `progress create`（先 `--help` 查看参数）：

```
任务模板（名称固定，不得删减合并）：
  1. 获取需求详情
  2.1 系统内搜索产品
  2.2 网上搜索生产商品牌
  2.3 逐个品牌官网验证
  2.4 查找技术文档和规格书
  2.5 B2B平台交叉验证
  2.6 规格对比并记录研究结果
  2.7 检查客户疑点（有疑点则暂停通知）
  2.8 产品上架或绑定
  3.1 查询品牌已有供应商
  3.2 网上搜索新供应商
  3.3 企查查核验供应商资质
  3.4 创建供应商并绑定品牌
  4.1 对比已询价供应商
  4.2 向新供应商发送询价
  5. 生成处理报告

relation-type: 需求
relation-id: {需求ID}
link: {manageMainUrl}#/main/demandInquiryDetails/{需求ID}
```

**3. 重命名会话**
将会话名改为以下格式，方便查阅：
```
需求：{需求ID}-{需求主题}
```

#### 进度查询与恢复

**查看当前进度：**

```bash
# 查看进度详情（含所有步骤完成状态、每个步骤的 summary 和 links）
progress get --id {进度ID}
```

输出包含每个步骤的状态：待完成 / 已完成 / 异常。步骤 2.7（检查客户疑点）标记为异常时，说明进行了二次确认通知，但不阻塞流程——无论客户是否回复，都继续下一步。

**恢复处理（客户回复后或直接继续）：**

1. `progress get --id {进度ID}` — 查看当前进度
2. 客户有新信息 → 更新研究记录，从 2.8 继续
3. 客户无回复 → 按已有信息从 2.8 继续

> AI 在恢复处理时，必须先 `progress get` 了解当前状态，再继续操作。

#### 进度更新规则

进度在开始时一次性建好全部任务。**每执行一个操作，就立即更新一次进度**，不能攒到一批再做：

```bash
# ⚠️ 正确做法：每完成一个小步骤就更新
产品匹配步骤 → 立即 progress step
下一个产品搜索 → 立即 progress step

# ❌ 错误做法：全部做完才更新
全部搜索完 → 一次性更新多个进度
```

更新格式：用 `progress step`（先 `--help` 查看参数）
- summary — 必须包含**验证证据**：搜到了什么（具体型号/参数）、搜到了几个（数字）、来源 URL（P15 规则：来自工具返回）、关键参数值。**禁止空泛描述**如"找到多个来源""确认产品存在""搜索完成""已核验"
- links — 所有相关来源链接，多个逗号分隔。每条搜索结论必须有对应链接。**链接必须来自搜索工具或 web_fetch 的实际返回（P15），不能拼造**

**progress summary 写作标准：** 想象另一个 AI 读到这个 summary，能否仅凭 summary 内容判断步骤是否真正完成？如果不能 → 补充具体数据。

**Step 1：产品搜索与信息采集**

> 目标：确认产品真实存在、获取完整规格信息、决定是否绑定。

**在开始搜索前，先运行 `product guide` 了解系统的产品定义**（SPU/SKU 结构、规格系统、属性、快速创建等）。这对后续判断规格数据至关重要。

#### 搜索策略（按优先级，换方式不等于不存在，多种都要尝试）

**一、系统内搜索产品**

- **A. SKU 型号搜索** — 用 `product sku list`（先 `--help` 查看参数），用客户型号搜 model 字段
- **B. ES 搜索（线上环境）** — 用 `product spu es-search`（先 `--help` 查看参数），搜型号/品牌+型号。仅线上可用
- **C. 缩短型号再试** — "CVS100N" → "CVS100"，"RA/8125/50" → "RA/8125"

**系统搜不到 → 开始网上搜索。**

产品信息来源优先品牌官网和行业龙头网站，其次 datasheet/选型手册，最后才是 B2B 平台交叉验证。B2B 搜到产品后必须追溯到官网确认。每个验证结论需标注来源类型。

#### 搜索关键词方法论（重要）

**不要一次把客户所有描述堆进搜索框。** 拆成短关键词组合，每次搜 2-3 个核心词，多轮尝试。

以客户说"超声波换能器 用于气体，压敏陶瓷做的，频率 200K"为例：

**第一轮（核心词）：**
- `超声波换能器 200K` — 产品名 + 关键规格
- `200KHz ultrasonic transducer` — 英文更精准
- `超声波换能器 气体` — 产品名 + 用途

**第二轮（换角度，第一轮结果不够时）：**
- `200KHz 换能器 厂家` — 找生产商
- `ultrasonic air transducer 200KHz` — 英文换表达
- `压电陶瓷 超声波 气体` — 材质+产品+用途

**第三轮（搜到品牌后，针对品牌深挖）：**
- `聚德电子 DYA-200 datasheet` — 品牌+系列+文档
- `Unictron A200M7 spec` — 品牌+型号+规格

**原则：短词优先，多轮尝试，搜不到就换词。** 中文搜不到换英文，英文搜不到换拼音。不要一次丢 5 个关键词进去。

> **品牌搜索必须中英文双关键词。** 中文名和英文名覆盖度差异很大——搜"乐可利"可能只找到 3 条记录，但搜"LEGRIS"能找到 59 条。涉及外资/合资品牌时，优先用英文名搜索，中文名作补充。

**二、网上搜索生产商品牌**

客户没指定品牌 → **不能跳过品牌直接搜产品**。必须先找到这个产品由哪些品牌生产：

- 搜 `{产品名} {型号} 厂家` / `{产品名} {型号} 品牌` / `{产品名} {型号} 生产商`
- 搜英文：`{产品名} {型号} manufacturer` / `{产品名} {型号} brand`
- 用 `web_fetch` 打开找到的页面，列出所有生产这个产品的品牌
- **至少找到 2-3 个品牌后，再进入下一步（逐个品牌去官网验证）**

**三、逐个品牌官网验证**

对上面找到的每个品牌，去官网确认产品是否存在：

- 搜 `{品牌} 官网` → 打开官网 → 找 **"产品中心"、"型号查询"、"Product Finder"、"产品目录"、"Downloads"**
- 搜 `{品牌} {型号}` → 直接定位产品详情页
- 从产品页提取：完整型号、规格参数表、技术参数、尺寸图、产品图片
- 官网列的所有规格信息**全部记录**
- **如果官网上找不到这个型号 → 说明这个品牌可能不生产这个产品，标记为"待确认"**

**四、查找技术文档和规格书**

- 搜 `{品牌} {型号} datasheet pdf` / `{型号} 规格书` / `{型号} 选型手册`
- 搜 `{品牌} {系列名} catalog` / `{品牌} 产品目录 pdf`
- 搜 `{品牌} {型号} 说明书` / `{品牌} {型号} 安装手册`
- 提取：规格表、尺寸图、性能曲线、选型配置表（有规格变化时的全部选项）

> 找到 datasheet/选型手册时，不只确认产品存在，**必须提取规格数据**。有规格数据才能上架完整系列（参考 `product guide`），没有就只能上架单个 SKU。aiRemark 中专门记录找到的规格信息。

**五、B2B平台交叉验证**

- 1688/百度爱采购：搜 `{品牌} {型号}` → 看详情页参数和实物图
- icspec.com / allchips.com：电子元器件专用
- 搜索引擎：`{品牌} {型号} 参数` / `{型号} 替代` / `{品牌} {型号} 技术参数`

**搜索词多换：** 完整型号 → 缩短型号 → 品牌+产品名 → 型号单独 → 英文 → 拼音 → datasheet → catalog → 选型 → 参数 → 替代 → 说明书

**每次搜到结果必须用 web_fetch 实际打开页面查看内容，不能只看搜索摘要。**

#### 搜索深度自检（2.6 完成前必须逐项确认）

| 检查项 | 要求 | 不满足则 |
|--------|------|---------|
| 每条搜索结论是否有对应的 `web_fetch` 打开记录 | 搜到结果 → 打开页面 → 提取内容，不能只看搜索摘要 | 标 ❓，不能标 ✅ |
| 是否至少 2-3 个独立来源 | 官网 + B2B + datasheet 等，至少 2 个不同网站 | 继续搜索，不进入下一步 |
| URL 是否来自工具返回结果 | 每个引用的 URL 必须是搜索/web_fetch 实际返回的（P15） | 删除该 URL，不能写入 aiRemark |
| 搜到的页面内容是否与客户需求逐条匹配 | 每项规格都要打标记（✅/⚠️/❓） | 补充标记，不能跳过 |

**未通过自检 → 补充搜索后再进入 2.6。**

#### 交叉验证与判断

找到产品信息后，**必须逐一对比验证**，不能凭感觉下结论：

1. **客户需求 vs 搜索结果**：客户说的每一项（产品名、品牌、型号、规格、材质、用途、频率……）逐条对比
2. **品牌确认**：找到的品牌是官方品牌还是经销商自己标的？→ 以官网为准
3. **至少 2-3 个独立来源**确认后，才能下结论

**在 aiRemark 中必须写明验证推理过程：**
- 客户要求什么 → 实际搜到了什么 → 每一项是否匹配 → 为什么确认/排除
- 不能只贴链接，要写清楚"为什么这个链接里的产品就是客户要的"

#### 匹配度标记规则（每条规格必须标记）

**客户说的和搜到的每一项都要打标记，不能跳过。**

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| ✅ 一致 | 完全一致或等价 | 数值相同、名称一致、规格匹配 |
| ⚠️ 有差异 | 不完全一致，需解释 | 用词不同可能同义、数值偏差、术语不统一 |
| ❓ 无法确认 | 搜不到对应信息 | 找不到规格、官网未列出此项 |

**⚠️ 标记的铁律：客户说的和搜到的不一样——哪怕只差一个字——也必须标 ⚠️，绝对不能标 ✅。**

工业品参数差一个字可能就是完全不同的产品。常见必须标 ⚠️ 的情况：

- 客户说"压敏陶瓷"，搜到"压电陶瓷" → ⚠️ 完全不同的物理效应和材料！压敏=电压→电阻变化（电路保护），压电=机械力↔电压转换（超声波换能器）——**虽然超声波换能器确实用压电陶瓷，但术语不同必须标注并解释**
- 客户说"不锈钢304"，搜到"不锈钢316" → ⚠️ 不同牌号，耐腐蚀性不同
- 客户说"200KHz"，搜到"200KHz±5%" → ✅ 数值落在公差范围内，但需注明公差
- 客户说"气体用"，搜到"液体/气体两用" → ✅ 涵盖气体用途，但需注明也适用液体

**每个 ⚠️ 标记必须附带三要素：**
1. **差异大白话解释** — 让不懂技术的业务人员也能看懂区别
2. **影响评估** — 这个差异会导致什么后果（不影响使用？可能不是同一个产品？需要确认？）
3. **判断依据** — 为什么 AI 认为可以接受 / 不能接受

**示例（正确）：**
```
- 材质：客户说"压敏陶瓷" → 搜索到的产品均使用压电陶瓷(PZT) ⚠️ 术语有差异
  - 解释：压敏陶瓷是压敏电阻材料，用于电路保护；压电陶瓷是PZT，用于超声波换能器。
    两者是完全不同的物理效应（压阻效应 vs 压电效应）。
  - 影响：超声波换能器的工作原理决定必须用压电陶瓷，客户大概率混淆了术语
  - 判断：功能上合理（换能器=压电不是压敏），但需向客户/供应商确认
```

**示例（错误，绝不允许）：**
```
- 压敏陶瓷 → 官网说明采用压电陶瓷(PZT) → ✅ 匹配
  ↑ 这完全跳过了术语差异！业务人员看不懂为什么"压敏"="压电"！
```

#### 确认标准

| 情况 | 操作 |
|------|------|
| 同品牌 + 同型号完全一致 | **必须用 `demand update-item` 直接绑定**，不能只记 aiRemark |
| 客户没指定品牌，搜到 WKEA 同型号 | 绑定 |
| 客户没指定品牌，搜到其他品牌（Festo/SMC等） | **不绑定**。只记 aiRemark，上架时按客户原文创建，品牌默认 WKEA |
| 品牌或型号不完全一致 | 只记 aiRemark，不绑定 |
| 搜不到 | 如实记录，继续供应商匹配 |

> 客户给了品牌且网上验证一致 → **绑定是强制的，不能跳过。**
> 知名品牌（SMC/Festo/Bosch Rexroth 等）不例外——客户写的型号仍可能写错系列号、混淆版本。必须逐一验证每个型号参数，不能以"大牌不会错"为由跳过。

#### 提问规则——基于官方规格表

生成客户确认问题前，**必须先找到官方品牌的同类产品页面或 datasheet**。提问只能基于官方规格表中明确存在的可选参数。

**流程：**
1. 搜索官方品牌同类产品（品牌官网 → datasheet → 选型手册）
2. 从官方来源提取规格表或型号编码规则，明确哪些参数存在多个选项
3. 对照客户已提供的信息：
   - 客户已给 → 不提问
   - 客户没给 + 规格表中有多个选项 → 可提问
   - 客户没给 + 规格表中不存在该参数 → 不提问
4. 每个问题**必须附带来源**：从哪个品牌的产品页/datasheet 确认该参数是可选的

**找不到官方规格表 → 不生成确认问题。** 标注"未找到官方规格信息"即可，不能凭猜测提问。

**示例：** 客户说"4-M6"，搜到 SMC KQ2 系列管接头，官方 datasheet 显示型号结构为 `KQ2[接头类型][管径]-[螺纹]`，接头类型有 H(直通)/L(弯头)/T(三通) 可选。客户已给管径 4mm、螺纹 M6 → 不提问。客户没给接头类型 → 规格表中存在多选项 → 提问："4-M6 需要哪种接头类型？参考 SMC KQ2 系列：H=直通、L=弯头、T=三通"

#### aiRemark 写入检查（2.6 完成前必须执行）

2.6（规格对比并记录研究结果）不是读完数据就完成。**必须实际执行 `demand update-item` 命令将研究结果写入每个行项目的 aiRemark 字段。** 2.6 的 progress summary 中写："已为 N 个行项目写入 aiRemark"。未写入 = 2.6 未完成，不得标记。

#### 产品搜索完成后的流程判断

Step 1（产品搜索）完成后，**检查客户确认清单是否为空**：

**需要客户确认 → 二次确认，不硬阻塞：**

> 二次确认的目的是在继续之前和客户对一下核心信息。但**不是硬阻塞——无论客户回不回新信息，确认后都继续往下走。** 客户给了新信息 → 更新后继续；客户没回 → 按已有信息继续。

1. **标记进度异常并立即停止** — 用 `progress step --abnormal` 标记 2.7，summary 粘贴 aiRemark 中"二次确认"的完整内容。**标记后立即停止，禁止操作任何后续步骤。** 步骤 2.8-5 必须保持待完成状态，不得标记。
2. **通知企业微信** — 发送和进度 summary 完全一致的内容（需求 ID + 主题 + 二次确认全文 + 管理后台链接），不写两套
3. **输出给当前对话** — 同上

**无需确认 → 继续：**

- 产品信息完整无歧义 → 直接进入 Step 2（供应商匹配）

> **判断标准**：不是看搜到多少品牌，而是看有没有官方规格表中存在多选项但客户没给的参数。找不到官方规格表 → 不生成确认问题，直接继续。

- 只有单个型号无规格信息 → 用 `demand simple-create-product` 一键上架
- 客户指定了品牌且验证一致 → 创建后立即用 `demand update-item` 绑定

**上架/绑定后立即执行验证（P9）：**

# simple-create 创建后

#### aiRemark 记录规范（强制）

**每搜索一次就记录一次，不能攒到最后一起写。**

aiRemark 必须使用 **固定模板** 记录。模板文件：`docs/modules/demand-aiRemark-template.md`。

**在使用 aiRemark 之前，必须先用 Read 工具读取模板文件，按模板定义的区域结构、标题名、表格列名填写。不可自行修改结构。**

**核心规则：**

- **8 个固定区域**按顺序排列，不可调换：风险评估 → 二次确认 → 供应商确认清单 → 客户原文 → 产品研究 → 供应商匹配 → 询价状态 → 处理时间线
- **风险评估永远第一**，业务人员一眼看到置信度和关键差异
- **二次确认紧跟风险评估**，包含已了解的信息（带来源链接）+ 可直接复制发送给客户的确认问题（序号列表，简洁有礼貌）。**客户未指定品牌 → 提问中禁止透露任何品牌信息，只问参数。** 品牌未指定默认 WKEA，二次确认不是硬阻塞。
- **供应商确认清单紧跟二次确认**，规格不全/文档缺失/价格交期 → 问供应商
- **客户原文必填**，所有字段都要填写
- 每个区域即使无内容也要保留标题，写"无"
- 信息越多越好，禁止删减、概括、改写搜索到的内容

**结论（风险评估）的语义规则：**
- 置信度由 ⚠️/❓ 数量决定，不由"搜到多少品牌"决定
- 全部 ✅ → 置信度"高"，可写"规格完全匹配"
- 有 ⚠️ 但可解释 → 置信度"中"，第一行必须写"**存在差异需确认**"，禁止写"规格匹配"
- 有 ❓ 无法确认 → 置信度"低"，第一行必须写"**关键信息缺失需确认**"
- 关键差异用一句话列出每个 ⚠️ 和 ❓：客户说了什么、实际搜到什么、差异是什么

**格式参考：** 见 `docs/modules/demand-aiRemark-template.md`，包含完整区域结构、固定表格列名、语义规则说明。

### 4.2 供应商报价与价格管理

> 以下为业务参考，不属于全流程处理的固定步骤。AI 根据实际场景按需查阅使用。

#### 查看报价详情
- 用 `demand vendor-quotes`（先 `--help` 查看参数）

#### 报价评估与建议

当业务人员说"看看报价情况"时，AI 应主动整理报价数据并给出建议：

**1. 展示报价对比**
- 用 `demand vendor-quotes` 拉取全部供应商报价
- 将数据整理为对比表格：供应商名称 | 资质 | 单价 | 交期 | 库存 | 发货地 | 报价总金额 | 是否完成
- 同时展示供应商资质（tags）、报价有效期等辅助信息

**2. 给出采纳建议**
- 评估标准（按优先级）：价格低 > 交期短 > 已有采购记录 > 供应商资质（专精特新/高新企业等优先）
- 综合排序后推荐 1-2 家最合适的供应商，说明理由
- 明确告知：**所有报价均会保存，业务员只需决定采纳谁**

**3. 等待业务员决策**
- 问"你想采纳哪家供应商？"或直接指定供应商 ID
- 业务员确定后，执行以下操作：

**4. 保存全部报价 + 设置采纳**
- 所有供应商报价 → 用 `demand save-price` 逐条保存（保留全部报价记录）
- 采纳的供应商 → 用 `product supply set-master` 设为主供应商价格

#### 保存供应商报价

**记录报价（不改售价）** — 用 `demand save-price`（先 `--help` 查看参数），isMaster=false

**设为主供应商价格** — 用 `product supply set-master`（先 `--help` 查看参数）
- 改写 SKU 默认售价、采购价、交期，同步更新维嘉替代品折扣
- **智能重定向：** 如果该 SKU 已绑定完全替代品（isFullReplace），价格自动设到替代品上

#### 替代品与停产

以下工具各自独立，按需组合。命令均需先 `--help` 查看参数。

**替代关系** — `product sku replace`
- `add` / `list` / `remove` 子命令。完全替代加 full-replace 参数。只管两个 SKU 关系，不管价格和停产。

**创建替代产品** — `product quick-create` 或 `spu create` + `sku create`

**停产标记** — `product spu update` 的停产参数。值 0=停产无替代，SPU ID=停产有替代。

**操作示例：**
- 供应商说"第N个产品停产，用xxx替代" → items 定位 itemId/skuId → quick-create 建替代品 → sku replace add 绑关系 → set-master 设价（自动走到替代品）→ spu update 停产参数
- 供应商说"可用xxx替代，没停产" → sku replace add 绑关系 → set-master 设价
- 供应商说"停产，没替代" → spu update 停产参数值 0

### 4.3 执行记录规范

所有处理记录写入行项目的 `aiRemark` 字段。**必须使用固定模板** — 先读取 `docs/modules/demand-aiRemark-template.md`，按模板定义的区域结构、标题名、表格列名填写。

核心要求：

- 8 个区域按固定顺序排列，不可调换
- 信息越详细越好，禁止删减

## 5. 边界情况

- **ES 不可用**（开发环境返回 500）→ 必须走 WebSearch，不能跳过产品匹配
- **品牌无供应商** → 必须网上搜索创建，不能报错退出
- **供应商没有邮箱/电话** → 不能创建，继续深挖找到联系方式
- **多个供应商匹配** → 优先选有授权证书、工商状态正常、有官网的
- **供应商异步报价** → AI 发出询价后等待供应商响应（异步），需轮询检查报价状态
- **一键上架** — 内部自动查重（按品牌+型号），重复则直接绑定
- **主供应商机制** — 品牌可设置主要供应商，自动询价时优先询价
