# 需求询价管理

## 1. 业务概念

**需求询价** — B2B 采购的核心入口。客户提需求 → 内部匹配产品/供应商 → 向供应商询价 → 供应商报价 → 转订单。

### 实体关系
```
DemandQuotation（需求询价主表）
  ├── DemandQuotationItem（行项目 1..N）
  │     └── skuId → SKU（可选绑定）
  ├── DemandQuotationDocInfo（供应商报价记录）
  │     └── DemandQuotationDocInfoData（报价明细）
```

### 状态
| 状态码 | 含义 |
|--------|------|
| 274 | 待处理 |
| 275 | 处理中 |
| 276 | 已完成 |
| 291 | 已取消 |

### 行项目关键字段
- 客户填写（只读）：`productName`、`productBrand`、`productModel`
- 后台编辑：`manageProductName`、`manageProductBrand`、`manageProductModel`
- 绑定产品：`skuId`
- 业务数据：`quantity`、`expectPrice`、`finalSkuPrice`、`grossMargin`
- 记录：`aiRemark`（AI 处理记录，Markdown 格式）

---

## 2. 前置条件

- 产品匹配前：需求已被当前 AI 领取（领取由 AI 自动处理）
- 供应商匹配前：行项目必须已绑定 SKU
- 询价前：行项目涉及的品牌必须有绑定的供应商

---

## 3. 判断依据

- **ES 搜索可用**（线上） → 优先用 ES 搜索已有产品
- **ES 不可用**（开发环境返回 500） → 走 WebSearch
- **品牌有供应商** → 直接询价；**品牌无供应商** → 必须网上搜索创建，不能跳过

---

## 4. 操作流程

### 4.0 创建需求（客户提出需求时）

当客户提出需求（文字描述或图片/文件），先解析需求内容，再创建到系统，然后自动进入处理流程。

#### Step 0.1：解析需求

使用 `parse_demand` 工具或 `demand parse` 命令将客户的需求文本/附件解析为结构化行项目：

**方式 A：使用 MCP 工具 `parse_demand`**
- 将客户文字描述/附件 URL 传入，返回结构化行项目
- text — 客户文字描述或表格文件 URL；customerRemark — 需求额外备注；annex — 附件 URL（逗号分隔）

**方式 B：使用 CLI 命令 `demand parse`**
- 先 `--help` 查看参数，支持文本输入、附件链接、JSON 输出等

`parse_demand` 会返回解析结果，每个行项目包含：产品名称、品牌、型号、数量、单位等。

#### Step 0.2：创建需求到系统

将解析结果通过 `demand create` 创建到系统。创建时需要补充渠道来源和客户来源：

1. **渠道来源**（必填）— 客户从哪个渠道来的，从以下选项中选择：
   - `淘宝-亿日`、`淘宝-维嘉`、`1688`、`微信`、`邮箱`、`线下`、`其他`
   - 如果用户没提供，**必须主动询问**，不能自行猜测
2. **客户来源**（可选）— 客户姓名，用户提供了就填，没提供就询问一下，不填也可以

用 `demand create` 创建（先 `--help` 查看参数），关键字段：
- channel-source — 从上面选项中选择
- customer-source — 客户姓名（可选）
- items — 解析结果 JSON

#### Step 0.3：进入全流程处理

创建成功后，按下方 4.1 全流程处理步骤继续（获取详情 → 创建进度 → 产品匹配 → 供应商匹配 → 询价）。

> **AI 必须全程自主处理，不允许停下来问用户。**
> 每步在行项目的 `aiRemark` 字段记录处理过程（Markdown 格式）。

### 4.1 全流程处理（逐项完成）

#### 前置准备：获取详情 → 创建进度 → 重命名会话

**1. 获取需求详情**
`demand get` 获取详情（先 `--help` 查看参数）
获取后得到：需求主题、客户信息、行项目列表、SKU 详情等全部上下文。

**2. 创建进度**
按 P2 原则，严格按下述模板创建，**不得自行删减、合并或重排任务**。用 `progress create`（先 `--help` 查看参数）：

```
任务模板（名称固定，不得删减合并）：
  1. 获取详情
  2.1 SKU型号搜索
  2.2 ES搜索引擎搜索
  2.3 缩短型号SKU搜索
  2.4 网上搜索产品
  2.5 验证并绑定SKU
  3.1 查品牌已绑供应商
  3.2 网上搜索供应商
  3.3 创建并绑定供应商
  4.1 对比已询价列表
  4.2 向新供应商发询价
  5. 记录完成

relation-type: 需求
relation-id: {需求ID}
link: {manageMainUrl}#/main/demandInquiryDetails/{需求ID}
```

**3. 重命名会话**
将会话名改为以下格式，方便查阅：
```
需求：{需求ID}-{需求主题}
```

#### 进度更新规则

进度在开始时一次性建好全部任务。**每执行一个操作，就立即更新一次进度**，不能攒到一批再做：

```bash
# ⚠️ 正确做法：每完成一个小步骤就更新
产品匹配步骤 → 立即 progress step
下一个产品搜索 → 立即 progress step

# ❌ 错误做法：全部做完才更新
全部搜索完 → 一次性更新多个进度
```

更新格式：用 `progress step`（先 `--help` 查看参数）
- summary — 记录实际操作中的思考和结果，即使没有也要如实记录（想了什么办法、查了什么、为什么没找到）
- links — 相关来源链接，多个逗号分隔

**Step 1：产品匹配（为每个行项目找到或创建 SKU）**

> 目标：为每个行项目绑定一个具体的 SKU（系统产品）。匹配到完全一致的型号才能上架。

#### 搜索已有产品（按顺序尝试，换方式不等于不存在）

**方式 A：按 SKU 型号搜索** — 用 `product sku list`（先 `--help` 查看参数），用客户型号搜索，匹配 model 字段

**方式 B：ES 搜索引擎搜索（线上环境优先）** — 用 `product spu es-search`（先 `--help` 查看参数），优先按型号搜索，也可搜产品名、品牌+型号等。仅线上可用，开发环境走到方式 C/D

**方式 C：按 SKU 型号搜索（缩短型号再试）** — 同方式 A，完整型号搜不到则缩短再试（"CVS100N" → "CVS100"）

**方式 B：网上搜索**（系统找不到时必须上网搜，不能跳过）

搜索产品时，**必须主动点击搜索结果中的链接进去查看实际内容**，不能只看摘要片段就下结论。

**搜索渠道（多管齐下）：**
| 渠道 | 搜索方式 | 搜索词示例 |
|------|---------|-----------|
| 通用搜索引擎 | `web_search` | "SMC 电磁阀 VXZ-1/4-F"、"3M 电动钻 600W 型号" |
| B2B 平台 | 搜索引擎搜 + `web_fetch` 进详情页 | "1688 SMC VXZ系列 电磁阀"、"淘宝 3M 电动工具" |
| 品牌官网 | `web_fetch` 打开官方产品页 | 搜索品牌官网的"产品中心"或"型号查询" |
| 行业网站 | 用产品系列名搜 | "AMSX-X2044 选型"、"VXZ 系列 规格书" |

**验证搜索结果（必须多来源交叉验证）：**

**当找到一个可能的匹配后，不能只看一个来源就下结论。** 必须从至少 **2-3 个独立来源**交叉验证：

1. **品牌官网** — 打开官方产品页，确认型号、规格参数
2. **B2B 平台** — 1688/淘宝等平台同型号产品页，看参数和实物图
3. **行业网站/技术文档** — 产品数据手册、选型指南等
4. **多个供应商页面** — 不同供应商的同型号产品信息对比

**确认标准：**
- **完全一致**（品牌+型号+规格从多个来源确认一致）→ 上架并绑定 SKU，记录 aiRemark 时附上所有来源链接
- **部分一致**（品牌对但型号有出入/参数不完全匹配）→ **不能上架**，只记录 aiRemark：写了什么信息对得上、什么对不上、来源链接
- **完全不一致** → 如实记录搜索过程和结果

**不同搜索词多试几次：**
- 用产品全称搜 → 搜不到换简称
- 用型号搜 → 用系列名搜
- 用品牌+产品名搜 → 用品牌+型号搜
- 用中文搜 → 用英文/数字型号搜

#### 验证产品匹配

找到可能的匹配后，必须验证以下条件**全部满足**才算完全匹配：
1. **品牌一致** — 客户要求的品牌与系统品牌一致
2. **型号完全一致** — 客户要求的型号与系统型号必须完全匹配
3. **规格参数一致** — 如有规格要求，逐一比对

> **验证通过**（品牌+型号完全一致）→ 绑定 SKU，进入下一步。
> **近似匹配**（品牌或型号有细微差异，如少字母、拼写接近）→ **不能直接当完全匹配处理**。必须先验证：
>   - 网上搜索确认：用搜索引擎搜该品牌/产品，打开官网或产品页确认是否正确
>   - 多渠道交叉验证：至少 2-3 个独立来源确认信息一致
>   - 验证后确认一致的 → 按完全匹配处理，在 aiRemark 中记录验证过程
>   - 验证后仍不确定的 → 不能强行绑定，继续尝试其他搜索方式
> **验证不通过**（品牌或型号对不上）→ 不能强行绑定，继续尝试其他搜索方式。
> **所有方式都搜完仍无匹配** → 不需要停在这里！即使没找到 SKU，**也要继续往下走**：
>   - 记录当前搜索情况（尝试了哪些方式、关键词、结果）
>   - 供应商匹配阶段可以按**品牌名/产品名**继续查找供应商
>   - 供应商报价后可以确认产品确实存在，届时再一键上架创建 SKU

#### 3.3 上架并绑定

**情况 A：系统已有完全匹配的 SKU** — 用 `demand update-item`（先 `--help` 查看参数）将 SKU 绑定到行项目。绑定后 `demand get` 返回的行项目中会自动嵌入 SKU 详情（价格、库存、品牌、分类、规格值等），无需额外调用 `sku get`。

**情况 B：系统没有，需要新建** — 用 `demand simple-create-product` 一键上架（自动创建 SPU+SKU 并绑定），或手动用 `product spu create` + `product sku create`（各自先 `--help` 查看参数）

**Step 2：供应商匹配（必须为每个行项目涉及的品牌找到有联系方式供应商）**

> 目标：**尽可能多地找到合格供应商**，不能只找一家就了事。每家供应商必须通过企查查核验，经营异常或已注销的排除，有荣誉资质的优先。

#### 搜索供应商（多渠道、多找几家）

**方式 A：系统查询** — 用 `demand vendors-by-brand`（先 `--help` 查看参数），默认只返回主要供应商

**方式 B：网上搜索**（系统里没找到足够供应商时）
品牌官网查找经销商、搜索引擎、B2B 平台等多渠道搜索，找到潜在供应商的公司名称。

**找到后，对每家潜在供应商都执行以下企查查核验流程：**

---

#### 企查查核验流程（每家供应商都必须做）

**第1步：查询企业工商信息** — 确认经营状态
```
get_company_registration_info(searchKey: "公司全称")
```
- 确认 **登记状态** 是否为 `存续（在营、开业、在册）`
- 如果是 **注销/吊销** → 这家供应商直接排除，不需要继续查询
- 记录企业基本信息（法人、注册资本、成立日期、参保人数等）

**第2步：查询荣誉资质** — 优先选有荣誉的
```
get_honor_info(searchKey: "公司全称")
```
- 查看是否有 **专精特新**、**高新技术企业**、**科技型中小企业** 等荣誉
- **有荣誉的供应商优先选择**
- 记录荣誉列表，用于后续对比决策

**第3步：查询联系方式** — 获取电话、邮箱、官网
```
get_contact_info(searchKey: "公司全称")
```
- 获取电话、邮箱、官网地址
- **必须至少有一个邮箱或电话，否则无法发送询价通知**
- 如果没找到联系方式，尝试其他渠道补充，实在没有则排除

---

#### 对比决策

当所有潜在供应商都完成企查查核验后，汇总对比：

| 条件 | 处理 |
|------|------|
| 已注销/吊销 | ❌ **排除**，不保存不询价 |
| 存续，无联系方式 | ❌ **排除**，发不了询价 |
| 存续，有联系方式，无荣誉 | ✅ **全部创建并询价** |
| 存续，有联系方式，有荣誉 | ✅ **优先，全部创建并询价** |

> **不要只留一家供应商**。只要是通过核验的供应商，都保存到系统并向其发送询价。供应商越多，报价竞争越充分。

3. `vendor bind-brand` — 绑定品牌（先 `--help` 查看参数）

  1. ✏️ demand vendors-by-brand --brand-id 3M → S00123(主要), S00456
  2. ✏️ 对比已询价列表 → S00123未询价
- 来源：
  - [供应商详情] https://admin.wkea.cn/#/main/supplier-detail/S00123
  - [企查查] https://agent.qcc.com/query?company=3M中国 → 经营存续，无风险
- 验证结果：官网有联系方式，企查查工商状态正常
- 结果：已向S00123发送询价
```

**注意：** `aiRemark` 要写得像项目笔记——记录思考过程、每一步操作、每个来源链接和网址。不要只写结论不写过程。

**Step 2：供应商匹配（必须为每个行项目涉及的品牌找到有联系方式供应商）**

> 目标：找到至少一个**有邮箱或手机号**的供应商。没有联系方式的供应商无法发送询价通知，等于没用。

#### 4.1 搜索方式（多维切换，换维度不等于不存在）

**方式 A：按品牌查已绑供应商** — 用 `demand vendors-by-brand`（先 `--help` 查看参数），**不要用 --all**，默认只返回主要供应商

**方式 C：逐个产品维度搜** — 用 `product spu vendors`（先 `--help` 查看参数），查同 SPU 下已绑供应商

**方式 B：网上搜索**（系统找不到时必须上网搜，不能跳过）

**搜索优先级（按顺序尝试）：**

**优先：品牌官网查找经销商**
1. 先找到该品牌的官方网站（搜索引擎搜"品牌名 官网"）
2. 用 `web_fetch` 打开官网，找"经销商查询"、"授权代理"、"联系我们"等页面
3. 官网通常会列出授权经销商名单及联系方式
4. 如果官网有经销商查询功能，输入所在地区查找

**其次：全网搜索供应商**
| 渠道 | 搜索词示例 |
|------|-----------|
| 搜索引擎 | `<品牌名> 经销商`、`<品牌名> 代理商`、`<产品名> 供应商` |
| B2B 平台 | `1688 <品牌名> <产品名>`、`淘宝 <品牌名> 官方旗舰店` |
| 行业网站 | `<产品分类> 供应商`、`<品牌名> 授权代理` |

企查查核验流程详见 **Step 2** 的企查查核验流程章节。

**不同搜索词多试几次：**
- 搜品牌名找不到 → 搜产品名+供应商
- 搜中文找不到 → 搜英文品牌名+distributor
- 官网找不到联系方式 → 搜"公司名 电话"、"公司名 邮箱"

#### 供应商信息验证标准

找到的供应商必须满足以下条件才能创建：

**必填信息（缺一不可）：**
| 字段 | 说明 | 查找方式 |
|------|------|---------|
| 供应商名称 | 全称，如"深圳市某某科技有限公司" | 网页标题、公司介绍 |
| **邮箱** | 用于系统发送询价通知 | 官网"联系我们"、企查查、天眼查 |
| **手机号/电话** | 用于系统发送短信通知 | 网页联系方式、企查查 |

**辅助验证信息（尽量补充）：**
- 联系人姓名、职位
- 品牌授权证书、代理证明
- **企查查工商信息**（优先用 MCP 工具，其次 web_fetch）：
  - 统一信用代码、法人代表、注册资本
  - 经营状态（存续/注销/吊销）
  - 成立日期、营业期限
  - 注册地址与实际经营地址
  - 司法风险、经营异常记录
- 供应商类型（原厂/授权经销商/代理商）

> **如果只找到公司名但没找到邮箱和电话，不能创建。** 必须继续深挖（找官网→找"联系我们"→找企查查→找1688店铺）。每次点开一个页面都要用 `web_fetch` 实际查看内容，不能只看搜索摘要。

#### 创建供应商并绑定

1. `vendor create` — 创建供应商（先 `--help` 查看参数）
2. `vendor contact add` — 维护联系人（先 `--help` 查看参数）
3. `vendor bind-brand` — 绑定品牌（先 `--help` 查看参数）
4. `vendor bind-category` — 绑定分类（先 `--help` 查看参数）

> **注意**：不得因品牌无供应商就结束流程。必须找到至少一个有邮箱/电话的供应商。

**Step 3：向供应商询价**

```
**Step 3.1** — 用 `demand vendors-by-brand` 查已绑供应商（先 `--help` 查看参数，默认主要供应商）

**Step 3.2** — demand get 结果已含 quotedVendors，无需重复调用

**Step 3.3** — AI 自行对比两个列表，差集 = 需要新发询价的供应商

- 询价结果记录到行项目的 aiRemark 中

### 4.2 供应商报价与价格管理

> 以下为业务参考，不属于全流程处理的固定步骤。AI 根据实际场景按需查阅使用。

#### 查看报价详情
- 用 `demand vendor-quotes`（先 `--help` 查看参数）

#### 报价评估与建议

当业务人员说"看看报价情况"时，AI 应主动整理报价数据并给出建议：

**1. 展示报价对比**
- 用 `demand vendor-quotes` 拉取全部供应商报价
- 将数据整理为对比表格：供应商名称 | 资质 | 单价 | 交期 | 库存 | 发货地 | 报价总金额 | 是否完成
- 同时展示 `demand vendor-quotes` 结果中的供应商资质（tags）、报价有效期等辅助信息

**2. 给出采纳建议**
- 评估标准（按优先级）：价格低 > 交期短 > 已有采购记录 > 供应商资质（专精特新/高新企业等优先）
- 综合排序后推荐 1-2 家最合适的供应商，说明理由
- 明确告知：**所有报价均会保存，业务员只需决定采纳谁**

**3. 等待业务员决策**
- 问"你想采纳哪家供应商？"或直接指定供应商 ID
- 业务员确定后，执行以下操作：

**4. 保存全部报价 + 设置采纳**
- 所有供应商报价 → 用 `demand save-price` 逐条保存（保留全部报价记录）
- 采纳的供应商 → 用 `product supply set-master` 设为主供应商价格

#### 保存供应商报价

**记录报价（不改售价）** — 用 `demand save-price`（先 `--help` 查看参数），isMaster=false

**设为主供应商价格** — 用 `product supply set-master`（先 `--help` 查看参数）
- 改写 SKU 默认售价、采购价、交期，同步更新维嘉替代品折扣
- **智能重定向：** 如果该 SKU 已绑定完全替代品（isFullReplace），价格自动设到替代品上

#### 替代品与停产

以下工具各自独立，按需组合。命令均需先 `--help` 查看参数。

**替代关系** — `product sku replace`
- `add` / `list` / `remove` 子命令。完全替代加 full-replace 参数。只管两个 SKU 关系，不管价格和停产。

**创建替代产品** — `product quick-create` 或 `spu create` + `sku create`

**停产标记** — `product spu update` 的停产参数。值 0=停产无替代，SPU ID=停产有替代。

**操作示例：**
- 供应商说"第N个产品停产，用xxx替代" → items 定位 itemId/skuId → quick-create 建替代品 → sku replace add 绑关系 → set-master 设价（自动走到替代品）→ spu update 停产参数
- 供应商说"可用xxx替代，没停产" → sku replace add 绑关系 → set-master 设价
- 供应商说"停产，没替代" → spu update 停产参数值 0

## 5. 边界情况

- **ES 不可用**（开发环境返回 500）→ 必须走 WebSearch，不能跳过产品匹配
- **品牌无供应商** → 必须网上搜索创建，不能报错退出
- **供应商没有邮箱/电话** → 不能创建，继续深挖找到联系方式
- **多个供应商匹配** → 优先选有授权证书、工商状态正常、有官网的
- **供应商异步报价** → AI 发出询价后等待供应商响应（异步），需轮询检查报价状态
- **一键上架** — 内部自动查重（按品牌+型号），重复则直接绑定
- **主供应商机制** — 品牌可设置主要供应商，自动询价时优先询价
