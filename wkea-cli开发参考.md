# WKEA CLI 开发参考

> 这是 wkea-cli 的开发参考文档，重点描述业务文档如何编写、如何测试。
> 代码层面的规范（API 层、命令层、类型定义等）见项目源码，不在此赘述。

---

## 一、业务文档编写规范

业务文档（`docs/modules/*.md`）是 AI 操作后台的唯一知识来源。AI 不会读源码，只读文档。

### 1.1 结构模板

```markdown
## 1. 业务概念     ← 实体关系、状态枚举、关键字段（只写业务视角，不写 DB 字段）
## 2. 前置条件     ← 操作前必须满足的条件
## 3. 判断依据     ← AI 如何自主决策（如 ES 可用性、品牌匹配标准）
## 4. 操作流程     ← AI 执行步骤（固定步骤严格标序号，非固定用"业务参考"标注）
## 5. 边界情况     ← 异常场景、常见陷阱
```

### 1.2 四个核心原则

**① 不要写完整命令参数 — 引导 AI 用 `--help`**

这是最容易犯的错。文档写了 `demand create --customer-id X --topic "xxx" --channel-source "淘宝-维嘉"`，AI 就会复制粘贴，不会再去 `--help` 看有没有新增的参数。

```
错误：
  demand save-price --sku X --vendor-id Y --price Z --gross-margin N

正确：
  用 `demand save-price`（先 `--help` 查看参数）
```

这条原则**全篇适用**：任何命令引用都只写命令名 + 做什么，不写参数。确有需要单独解释的个别参数才提一句（如 `--full-replace` 的含义）。

**② 流程是流程，业务参考是业务参考 — 不混在一起**

AI 按流程执行时会把所有内容都当成"必须做的步骤"。所以价格管理、替代品等不应嵌在流程章节里，要单独标注为业务参考：

```markdown
### 4.1 全流程处理      ← 固定步骤，AI 必须逐项执行
### 4.2 供应商报价与价格管理  ← 业务参考，AI 按需查阅
> 不属于全流程的固定步骤，AI 根据实际场景按需查阅。
```

**③ 写业务逻辑，不写实现细节**

文档解释"是什么"和"为什么"，不是"怎么调用 API"。比如：
- ✅ "设置主供应商后，SKU 售价自动更新为 采购价/(1-毛利率)"
- ❌ "调用 POST /saveTableData 传入 tableInfoList，isMaster=true"

**④ 场景驱动，不是菜单列表**

不要让 AI 自己去猜用哪个命令。描述场景→对应命令：

```markdown
- 供应商说"第 N 个产品停产，用 xxx 替代" → items 定位 → quick-create 建替代品 → sku replace add 绑关系 → set-master 设价 → spu update 停产参数
- 仅停产无替代 → spu update 停产参数值 0
```

### 1.3 编写检查清单

- [ ] 全文无写死完整命令参数（每个命令引用后都有"先 --help 查看参数"）
- [ ] 每个命令引用后都有"（先 `--help` 查看参数）"
- [ ] 流程步骤和业务参考有明确分界
- [ ] 有场景 → 命令的映射说明
- [ ] 枚举值含义已描述（不需要 AI 额外查 enum）

## 二、设计审查

> 开始编码前必须检查。核心风险就三类：

### 2.1 一对多字段的编辑安全性

编辑接口 DTO **不能包含**一对多关联字段（vendorsId、items 等），关联关系走独立绑定/解绑命令。

```
❌ brand update --name "新名"  → 底层传了 vendorsId: undefined → 清空所有供应商
✅ brand update --name "新名"  → 只传 name
✅ brand bind-vendor / unbind-vendor  → 独立管理关联
```

### 2.2 状态流转安全性

如果实体有状态机（订单、需求等），确认后端已做状态校验，文档写明流转路径。

### 2.3 参数完整性

List 命令别只做 keyword 搜索。常见遗漏：时间范围、负责人筛选、关联实体筛选、空值筛选（`manageIdIsNull`）。

## 三、测试流程

> 每次新增或修改命令后，必须按三阶段完成。

```
Phase 1 — API 自测
  → 直测后端接口（数据结构 / 状态码 / 边界）
Phase 2 — CLI 自测
  → 编译 CLI，核心 CRUD 正常走通，覆盖缺参 / 异常数据
Phase 3 — 子任务模拟测试（核心）
  → 真实场景模拟，AI 仅有 skills 文档 + CLI 工具，自主完成任务
```

### Phase 1 — API 自测

启动本地后端，用 curl 或 Postman 直测所有新接口：

- 正常数据 → 200，返回结构正确
- 缺必填参数 → 400，错误信息清晰
- 异常数据（不存在 ID、越界值）→ 合适的错误码
- 级联影响 → 关联数据正确写入/清理

### Phase 2 — CLI 自测

编译 CLI 后逐条测试：

- 每个命令的 `--help` 输出正确
- 核心链路（创建→查询→更新→删除）正常走通
- 缺参数时报错清晰
- 异常数据时错误提示明确
- 输出格式（JSON + 字段说明表）完整

### Phase 3 — 子任务模拟测试

> **这是核心环节。** 模拟真实使用场景，验证 AI 能否仅凭 skills 文档 + CLI 工具完成任务。

#### 测试环境

AI 只能依赖：
1. SKILL.md 和 docs/modules/ 下的业务文档
2. `node dist/index.js --help` 和各命令的 `--help`
3. CLI 工具本身

AI 不能看源码、不能看 API 文档、不能访问后台页面。

#### 测试方法

1. **准备场景** — 构造 3~5 个真实业务场景（含正常+异常+边界）
2. **编写子任务提示词** — 用自然语言描述目标，不给具体命令
3. **AI 独立执行** — 给 AI 完整的 skills 文档（system + business + enum 说明）
4. **结果评估** — 评估 AI 的选择和操作是否合理：
   - 命令选择是否正确（有没有用错命令）
   - 参数是否完整（有没有漏必填）
   - 流程顺序是否合理（有没有跳步）
   - 错误处理是否恰当（API 报错后有没有继续硬来）

#### 场景要求

**简单场景（覆盖核心链路）：**
- 正常创建、查询、更新、删除
- 每个参数至少在一个场景中被用到

**复杂场景（覆盖边界）：**
- 关联实体操作（绑品牌、绑供应商）
- 状态流转操作（按状态筛选）
- 级联删除影响
- 时间范围、多值筛选等高级查询
- 一对一/一对多字段的编辑安全性
- 每个参数单独验证

#### 完成标准

- [ ] Phase 1：API 所有接口返回 200，异常场景返回合适错误码
- [ ] Phase 2：CLI 所有命令正常执行，错误场景合理提示
- [ ] Phase 3：简单场景全部通过，复杂场景全部通过
- [ ] Phase 3：AI 仅凭 skills 文档即可自主完成，无需人工提示具体命令
- [ ] Phase 3：所有参数经验证有效，没有遗漏的筛选维度

任何 Phase 失败 → 修 bug → 从 Phase 1 重新开始。
