# Workflow 04：品牌开发供应商

**触发条件**：
- 用户说"开发品牌 X" / "X 品牌找代理商" / 提供品牌 Logo 或网址
- 用户说"X 品牌没有供应商" / "X 品牌补全供应商" / "X 品牌供应商少于 2 家"
- 需求处理中发现某品牌的候选供应商不足

**视角**：品牌（vendor-expert 主导，brand-expert 协作）

**与 02 的区别**：
- 04 是**开发视角**（品牌 → 找供应商 → 写库）
- 02 是**上架视角**（产品 → 配套供应商 → 写库）

---

## 流程边界（必读）

**主流程**（同步完成、本轮结束点）：
- Phase 1 品牌识别 + 官网搜索 + 代理商提取
- Phase 2 资质核验 + 报告
- Phase 3 品牌处理（按需新建）
- Phase 4 供应商入库 + 绑定
- Phase 5 汇编

**禁止**把"等待人工确认"塞进主流程挂起；每完成一 Phase 立即汇编给用户。

---

## Phase 1：品牌识别 + 官网搜索 + 代理商提取

由 `wkea-vendor-expert` 主导：

### Step 1.1 品牌识别入口

| 输入类型 | 识别方法 |
|---------|---------|
| **品牌名称**（中文/英文）| 直接进入 Step 1.2 |
| **品牌 Logo 图片** | 搜索 `{品牌名+logo描述}` 反查；或用图片搜索找官网 |
| **品牌网址**（如 `https://www.festo.com/cn`）| 直接打开 → 提取品牌名 |
| **品牌图片**（产品图、宣传图）| 反向搜图找到品牌名 |
| **品牌型号 / 产品图** | 用产品名反查品牌，再回到本流程 |

**官网找不到时**：用 Google 按关键信息（产品类别、应用场景）直接推荐代理商名单，**必须写明推荐理由**。Google 搜不到再换 Bing。

### Step 1.2 官网搜索 + 代理商提取

**搜索优先级**：

1. **品牌官网授权渠道**（最可靠）
   - 搜 `{品牌名} 官网` / `{品牌英文名} official website`
   - 找子页面：「合作伙伴」「渠道」「Where to Buy」「Find a Distributor」「Find a Reseller」
   - 搜 `{品牌英文名} site:官网域名 distributor`
   - PDF 名录：搜 `{品牌名} 授权代理商名录 filetype:pdf`
2. **Google + B2B 平台**
   - Google（优先）、Bing（备选）、1688、百度爱采购、icspec、allchips
3. **行业垂直平台**
   - 工业自动化：工控网(gongkong.com)、智能制造网(gkzhan.com)
   - 电子元器件：icspec.com、allchips.com
   - 通用：made-in-china.com、gys.cn、黄页88

**提取的字段**（按重要性）：
- 邮箱（最优先，发询价用）
- 公司全称、电话、地址
- 联系人姓名及职位
- 授权说明/证书描述
- 官网 URL（必填，记录来源）

**完整性自检（必做）**：
- [ ] 页面是否有滚动加载？用浏览器滚到底
- [ ] 是否有「查看更多 / Load More」按钮？翻完才停
- [ ] 是否有分页？最后一页要点开
- [ ] 是否按区域/省份分类？每个子分类逐个翻
- [ ] 对比页面显示的「合作伙伴总数」与实际提取数量
- [ ] 页面有 PDF 文件链接？单独下载并解析

### Step 1.3 工具选择（不限制死）

| 场景 | 推荐能力 |
|------|---------|
| 简单 HTML 页面、静态列表 | `web_fetch` 抓取 Google/Bing 搜索结果页，或直接访问官网 |
| 表格 + 分页 + 动态加载 | 浏览器自动化（agent-browser / Kimi WebBridge，按当前会话可用工具选择）|
| 需要登录的代理商后台 | 浏览器自动化 + 复用登录会话 |
| PDF 名录 | 浏览器下载 → 转文本 → 解析 |

> **关键原则**：`web_fetch` 拿不到数据时，**升级到直接操作浏览器**，不要死磕 fetch。

**搜索关键词组合**（多试几种）：
- `{品牌名} 中国 授权代理商 经销商 官网`
- `{品牌名} 授权经销商 {地域}`（如"上海 江苏 浙江"）
- `{品牌英文名} China authorized distributor dealer`
- `{品牌英文名} distributor list` / `where to buy China`
- `{品牌名} 代理商名录` / `{品牌名} 在哪买`

## Phase 2：交叉验证 + 企查查核验

由 `wkea-vendor-expert` 执行：

### Step 2.1 交叉验证

- 同一家供应商在 **2 个以上来源**出现 → 可信度高
- 只有 **1 个来源** → 标记"待核实"
- 官网明确列出的 → 最可信

### Step 2.2 企查查核验（每家必做）

对每家潜在供应商**必须调用以下 MCP 工具**：

1. **工商信息** — `mcp__qcc-company__get_company_registration_info(searchKey: "公司全称")`
   → 确认登记状态为「存续」。注销/吊销 → 直接排除
2. **荣誉资质** — `mcp__qcc-operation__get_honor_info(searchKey: "公司全称")`
   → 专精特新、高新技术企业等，记录到 tags
3. **联系方式** — `mcp__qcc-company__get_contact_info(searchKey: "公司全称")`
   → 获取电话、邮箱、官网。必须至少有一个邮箱或电话

**企查查 MCP 不可用时的兜底**：

| 原 MCP 调用 | 替换为 Google/Bing 搜索 | 最低可接受证据 |
|-------------|------------------|---------------|
| 工商信息 | 搜 `{公司全称} 企查查` | 摘要显示"存续/在业"即视为通过；完全搜不到 → 标记"工商状态未知"，不创建 |
| 荣誉资质 | 搜 `{公司全称} 高新技术企业` | 搜到相关页面 → 记录；搜不到 → 标"未查获资质" |
| 联系方式 | 搜 `{公司全称} 联系方式` | 至少找到电话或邮箱之一；都没有 → 排除 |

**退出条件**：qcc MCP 不可用 + Google/Bing 搜索无工商/联系方式 → 标记"核验未通过"，不创建该供应商。

### Step 2.3 供应商四项核验标准

| 核验项 | 要求 | 不通过怎么办 |
|--------|------|------------|
| ① 工商状态 | 存续/在业 | 直接排除 |
| ② 是否有同类产品在售 | 官网/B2B 确实在卖 | 只有公司名 + 搜索摘要 → 标记待核实 |
| ③ 库存能力 | 有现货或可订货 | 无任何产品信息 → 警惕 |
| ④ 价格信息 | 可获取到报价或参考价 | 完全无价格信息 → 仅作备选 |

**三项全无 → 判定为空壳供应商，不创建。**

### Step 2.4 生成 HTML 报告

按「报告输出规范」章节输出。

## Phase 3：品牌处理（按需）

由 `wkea-brand-expert` 执行（**只有需要新建品牌时触发**）：

- **品牌已存在** → 跳过本 Phase，直接进入 Phase 4
- **品牌不存在** → `brand create`，传 `name` 等必填字段

**Phase 间数据传递**：
- Phase 2 完成后，主理人确认品牌存在性
- 不存在时，主理人派发 brand-expert 任务，附：
  - 品牌名
  - 品牌类型（628=自有品牌 629=代理，默认 629）
  - 标志位 `vendorsId` 留空（待 Phase 4 填入）
- brand-expert 完成后回传 brandId 给主理人

## Phase 4：供应商入库 + 绑定

由 `wkea-vendor-expert` 执行：

### Step 4.1 vendor create（每家核验通过的供应商）

对每家通过核验的供应商：

```bash
node dist/index.js vendor create \
  --name "公司全称" \
  --type 107 \    # 107=授权经销商
  --contact "联系人" \
  --phone "电话" \
  --email "邮箱" \
  --address "地址" \
  --website "官网" \
  --tags "高新技术企业" \
  --brand-id-list <brandId>  # Phase 3 拿到的 brandId
```

> 详细参数通过 `node dist/index.js vendor create --help` 查看。

**系统无对应字段的信息**（法人、注册资本、品牌代理等级等）用 `--extra-columns` 保存。

### Step 4.2 vendor bind-all（绑定品牌 + 分类，**优先用 bind-all**）

```bash
node dist/index.js vendor bind-all \
  --vendor-id <vendorId> \
  --brand-ids <brandId> \
  --category-ids <分类ID列表>
```

**为什么用 bind-all 而非 bind-brands + bind-categories 两次调用**：
- 1 次网络往返 vs 2 次
- 原子性更好
- 返回结构更清晰（一个合并结果）

### Step 4.3 vendor superior-category（优势分类，可选）

如代理商在官网明确标注了"主攻/擅长"某分类：

```bash
node dist/index.js vendor superior-category add \
  --vendor-id <vendorId> \
  --name "气动元件" \
  --system-category-path "1/5/23" \
  --priority 1
```

### Step 4.4 子集合维护

供应商创建后，按需补充：
- `vendor contact add` — 添加联系人
- `vendor address add` — 添加收货地址
- `vendor bank add` — 添加银行账户
- `vendor invoice add` — 添加开票信息
- `vendor-url add` — 添加官网/店铺链接

### Step 4.5 验证

每家供应商创建后立即 `vendor get` 验证：
- 基本信息已写入
- 品牌绑定已生效
- 分类绑定已生效

## Phase 5：主理人汇编

汇总所有 Phase 产出：

- **供应商 ID 列表**（每个 vendorId + 跳转链接）
- **品牌 ID**（如 Phase 3 新建）
- **HTML 报告路径**（如 `/tmp/brand-{name}-dev-report-{date}.html`）
- **失败原因**（核验未通过的代理商清单）
- **汇总跳转链接**（按主理人 P10 规则）

## 派单契约（防 #662 复现，主理人必须按此派单）

主理人调度 `wkea-vendor-expert` 处理 Phase 1-2 时，必须在 dispatch prompt 中附：

1. **该 expert 负责的报告内容**（HTML 报告的供应商列表部分）
2. **内容红线**（不写入 WKEA 系统的原始搜索 URL / 不确定的邮箱 / 未核验的联系方式）
3. **4 步写入流程**：
   - 读：`vendor list --keyword <名>` 查重
   - 改：构造 DTO
   - 写：`vendor create` + `vendor bind-all`
   - 验：`vendor get` 确认绑定关系
4. **涉及 Phase 4 时明确**：
   - 是 4.1 批量创建
   - 还是 4.2 bind-all 绑定
   - 还是 4.3 superior-category 优势分类

## 报告输出规范

- **官网 / 代理商页链接**：必须可点击跳转（`target="_blank"`）
- **电话链接（`tel:` 协议）**：桌面端不可用，**需提供备用联系方式**（邮箱、地址、官网联系页）
- **代理商名单按区域 / 省份分类**：便于用户查找
- **每家包含**：公司全称、地址、联系电话、邮箱（如有）、授权来源类型（官网授权 / 平台收录 / Google 搜索 / 待核实）
- **数据来源标注**：每个数据点写明出处
- **报告日期**：标注生成时间
- **未验证信息**：明确标"待核实"
- **格式**：HTML 报告（复用 `wkea-cli/docs/report-template.html` 模板）

## 经验教训

1. **只读一遍页面就停 = 大概率遗漏** — 必须检查滚动加载/分页/查看更多
2. **提取完成 ≠ 提取完整** — 对比页面显示的「合作伙伴总数」验证
3. **按区域分类的名单** — 每个子分类（如省份）都要逐个翻
4. **PDF 名录** — 是隐藏的金矿，单独下载并解析
5. **链接必须可点击** — `target="_blank"`
6. **电话 `tel:` 协议** — 桌面端不可用，必须备用联系方式
7. **不限制死工具** — `web_fetch` 能用就用，不行升级浏览器自动化
8. **fetch 拿不到 ≠ 没有** — 升级到浏览器自动化（agent-browser）后大概率能拿到

## 异常处理

| 场景 | 处理 |
|---|---|
| 品牌官网打不开/找不到 | 用 Google 搜索兜底推荐，写明推荐理由；Google 搜不到再换 Bing |
| 企查查 MCP 不可用 | 用 Google 搜索替代（见 Phase 2.2 兜底方案），Google 搜不到换 Bing |
| 同一家公司在 5+ 来源出现 | 按公司全称去重，合并信息 |
| 核验通过率 < 50% | 继续搜至少 3 轮不同关键词，不放弃 |
| 品牌无任何官网和授权信息 | 标记"待人工确认"，不强行创建 |
| 品牌已存在 | Phase 3 跳过，直接进 Phase 4 |
| 品牌不存在但需新建 | Phase 3 派 brand-expert，拿到 brandId 后回传 |
| 创建部分失败 | 已成功的保留，失败的标红给用户确认 |
| 链接打不开 | 强制 `target="_blank"`，提供电话备用联系方式 |
