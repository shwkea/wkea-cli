# Workflow 02：产品开发供应商

**触发条件**：
- 用户有一批产品要上架，需要同时搞定产品入库和供应商绑定
- 用户说"上架这批产品，供应商也一起找了"
- 多个品牌、多个产品、批量操作场景

**视角**：产品（product-expert 主导，vendor-expert 配合）

**前置条件**：如果用户还没做过产品研究（品牌不确定、型号没核实、datasheet 没找到），先走 **workflow 05（产品开发）** 搞清楚再上架。

**与 04 / 05 的区别**：
- **02**：批量上架（多个产品，品牌可能不同）→ product-expert + vendor-expert 并行
- **04**：品牌开发（品牌 → 找代理商 → 写库）→ vendor-expert 主导
- **05**：单系列配置化（选型手册 → 规格建模 → SKU 变型）→ product-expert 主导，适合规格复杂的单个产品系列

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 准备：产品模板 + 供应商候选（并行）
- Phase 2 入库：供应商 → 产品（串行）
- Phase 3 汇编与报告

**主流程结束点** = Phase 3 HTML 报告输出，含所有产品 SKU ID、供应商 ID、跳转链接、失败项。

---

## Phase 1：准备（并行）

由 `wkea-product-expert` 和 `wkea-vendor-expert` **并行**执行。

### product-expert：产品模板

如果已经做过 03 研究，直接用研究成果。否则按 03 的 SOP 补一轮。

输出每个产品的上架信息：

```markdown
| 产品 | 品牌 | 型号 | 分类 | SPU 名 | 单位 | 参考价 | 已有 SKU？|
|------|------|------|------|--------|------|--------|----------|
| 1 | SMC | AW20-N02 | 气动元件 | 过滤器 AW 系列 | 个 | ¥150 | 否 |
| 2 | ... | ... | ... | ... | ... | ... | ... |
```

**命名规则**：SPU 名不含品牌（品牌是绑定的），分类必须真实存在。

### vendor-expert：供应商候选

按用户给的品牌列表，逐品牌搜索授权代理商：

1. 查询品牌已有供应商：CLI `demand vendors-by-brand`
2. 已有 ≥ 2 家 → 跳过搜索
3. 不足 → 品牌官网搜代理商 → 企查查核验（工商状态、联系方式、荣誉资质）
4. 输出供应商候选列表

```markdown
| 供应商 | 代理品牌 | 资质 | 联系人 | 电话 |
|--------|---------|------|--------|------|
| 上海XX气动 | SMC | 官网授权 | 张三 | 138... |
```

**输出**：产品上架清单 + 供应商候选列表

---

## Phase 2：入库（串行）

**先 vendor 后 product**，因为产品需要供应商 ID 来绑定。

### Step 2.1 供应商入库

由 `wkea-vendor-expert` 执行。对每家通过核验的供应商：

```bash
# 创建供应商
node dist/index.js vendor create \
  --name "公司全称" \
  --type 107 \
  --contact "联系人" \
  --phone "电话" \
  --brand-ids "1,2,3"

# 绑定分类
node dist/index.js vendor bind-all \
  --vendor-id <vendorId> \
  --brand-ids "1,2,3" \
  --category-ids "10,20"
```

创建后立即 `vendor get --vendor-id <id>` 验证。

### Step 2.2 产品上架

由 `wkea-product-expert` 执行。按 Phase 1 的产品清单逐个创建：

**单 SKU 简单产品**：

```bash
# 创建 SPU
node dist/index.js product spu create \
  --name "<SPU 名称>" \
  --brand-id <id> \
  --category-id <id>

# 创建 SKU
node dist/index.js product sku create \
  --spu-id <id> \
  --name "<品牌名> <SPU名> <型号>" \
  --model "<型号>" \
  --sales-price <价格>

# 绑定供应商
node dist/index.js product spu bind-vendor --spu-id <id> --vendor-id <id>
node dist/index.js product sku supply set \
  --sku-id <id> \
  --vendor-id <id> \
  --purchase-price <价格>
```

**多 SKU 产品** → 用 `product quick-create` 一次性创建。

每创建一个立即 `product get` 验证，输出跳转链接。

### Step 2.3 产品资料补充

```bash
node dist/index.js product spu update \
  --spu-id <id> \
  --pdf-link "<datasheet URL>" \
  --images "url1,url2" \
  --description "<描述>"
```

**输出**：每步的结果（成功/失败）+ ID + 跳转链接

---

## Phase 3：汇编与报告

由主理人执行。

生成 HTML 报告，使用 `docs/report-style.html` 的 CSS。内容：

1. **任务摘要** — 一共上架几个产品、开发几个供应商
2. **产品清单** — 每个产品的 SPU ID / SKU ID / 价格 / 供应商 / 跳转链接
3. **供应商清单** — 每个供应商的 ID / 代理品牌 / 联系方式 / 跳转链接
4. **失败项** — 哪些产品/供应商没上架成功，原因是什么
5. **待补充** — 缺 datasheet、图片、价格的产品

文件命名：`/tmp/wkea-product-supplier-{日期}.html`

---

## 异常处理

| 场景 | 处理 |
|------|------|
| 供应商核验不通过 | 跳过该供应商，报告里标注原因 |
| 产品缺规格数据 | 退回到 03 补充研究 |
| 批量创建部分失败 | 已成功的保留，失败的标红给用户确认 |
| 品牌在系统中不存在 | 先创建品牌（派 brand-expert），再继续 |
| 分类不存在 | 让用户确认分类，或提示先创建分类 |
| 用户要做的是品牌开发（没有产品） | 转 workflow 04 |
| 只有一个复杂产品系列 | 转 workflow 05 |
