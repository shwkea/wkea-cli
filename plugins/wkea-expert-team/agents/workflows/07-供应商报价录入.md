# Workflow 07：供应商报价录入（维护供应商给回的报价字段到询价单）

**触发条件**：
- 业务人员拿到供应商的报价回复（一段文字，含型号、价格、货期等）
- 想把这些报价字段维护到询价单上，不需要去后台手填
- 例："帮我把这段报价录到 D20260701-001 / 供应商 S00925 上"

**视角**：需求（demand-expert 主导，本流程纯机械转换，不做产品研究）

**与其他流程的区别**：
- 04 / 02：找新供应商、写新供应商、产品上架
- 01 主流程：研究产品并发起询价（不涉及维护报价）
- **07**：供应商已经回复，业务人员做的是**纯字段录入**（拆解文字、按规则换算、写入系统）

**关键特性**：
- 区别于 `demand save-price`：save-price 是把价格作为产品供应信息写到 product_vendor 表（用于后续采购）。这里是把报价字段写到 demand_quotation_doc_info_data（询价单状态记录）。
- 区别于 `demand vendor-quotes`：那是只读查询。这里是写入。

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 解析报价文本，提取结构化字段
- Phase 2 字段换算（现货/周转天数/范围货期等）
- Phase 3 反问业务人员（特别是现货货期天数）
- Phase 4 写入询价单
- Phase 5 总结报告

**主流程结束点** = Phase 5 输出 HTML 报告 + 维护了多少条数据 / 哪些需要二次判断。

---

## 字段换算规则（核心 AI 行为）

供应商的报价文字经常不是直接能写入系统的字段，必须做换算：

### 1. 价格

| 供应商说法 | 写入字段 | 备注 |
|-----------|---------|------|
| `220` | price=220 | 直接数字 |
| `220.00` | price=220.00 | 直接数字 |
| `220 元` / `220/个` | price=220 | 去掉单位 |
| `约 200-220` / `200-220` | **反问业务人员** | 范围值不能瞎猜 |
| `(面议)` / `电议` | 不写价格，留空 | 价格字段不写 |

### 2. 货期（delivery 字段，单位天）

| 供应商说法 | 换算 |
|-----------|------|
| `现货` / `有货` / `库存` / `现货 100 个` | **必须反问业务人员按"本次"规则换算**（见下文"现货货期天"） |
| `现货 N 个` | delivery=现货换算天数 + stock=N |
| `1 周` / `7 天` / `一周` | 7 |
| `2 周` / `两周` | 14 |
| `3 周` | 21 |
| `1 个月` | 30 |
| `半个月` / `2 周` | 15 |
| `1 季度` / `3 个月` | 90 |
| `半年` | 180 |
| `1 年` | 365 |
| `1-2 周` / `3-5 天` | **反问**：不在范围内填写 |
| `3-5 天` | **反问**：取中位数（4 天）or 反问确认 |
| `(依量定)` / `(订单定)` | **反问**：依量多少 |
| `随时发` / `立即发` | 1 |

### 3. 现货货期天（反问规则）

供应商写了"现货"但**没说具体天数**，每次都要问。这次问的答案**只对本次有效**，下次同样问。

**为什么每次都问**：不同供应商对"现货"定义不同，有的 1 天有的 3-7 天。**不能默认**为某天数（之前案例：WKEA 一家供应商把"现货"维护成 2 天，结果实际是 1 天可发，导致与库存承诺不符，业务被供应商投诉）。

**反问方式**（用提问工具）：
> 该供应商说"现货"，请问按多少天维护？
> 1. 1 天（今天下单明天到）
> 2. 2 天
> 3. 3 天
> 4. 5 天
> 5. 7 天
> 6. 其他（请说明）

> ⚠️ 如果同一对话中**多个产品**都标"现货"，**只问一次**，后续产品沿用同样答案（在同一次询问的会话上下文中记住，跨任务不记忆）。

### 4. 库存

| 供应商说法 | 换算 |
|-----------|------|
| `现货 100 个` / `有 50` | stock=数字 |
| `充足` / `大量` / `有库存` | **反问**大致数量（>100 还是 <100） |
| `需备货` / `按订单备货` | stock=0，备注写"按订单备货" |
| 没说库存 | stock=0（默认），不主动猜测 |

### 5. 产地

| 供应商说法 | 写入 place 字段 |
|-----------|-----------------|
| `中国` / `国产` | place="中国" |
| `日本` / `Japan` | place="日本" |
| `美国` / `USA` | place="美国" |
| `德国` | place="德国" |
| 没说 | 不写 |

### 6. 备注（remark）

- 把供应商说的**关键信息**记下来，比如："含税含运费"、"最低 10 个起订"、"保质期 12 个月"
- 不写 AI 自己加的推测

## Phase 0：前置校验（信息不全时反问）

业务人员给的输入可能有缺漏。先校验**4 要素**，缺任何一个都要反问：

| 缺什么 | 反问 |
|--------|------|
| 需求 ID | "录到哪个需求上？（请提供需求 ID，例如 663 / D20260701-001）" |
| 供应商 ID 或名字 | "是哪家供应商的报价？（请提供供应商 ID 或名字，例如 S00925 / 上海崛内）" |
| 报价原文 | "请把供应商给的报价文字发给我，格式像：型号A 价格 货期" |
| 业务意图确认 | "您是要把供应商报价维护到询价单上吗？（区别于查询、采纳、转报价单）" |

**意图不全（如只说"录报价"）**：必须用提问工具一次性问完 4 要素，不派单。业务人员回答后进入 Phase 1。

### 状态校验

写入前校验需求状态：
- **可录**：274（待处理）、275（处理中）
- **拒绝**：276（已完成）、291（已取消）→ 提示"该需求已 <状态>，不能录入新报价，请联系管理员"

查询命令：
```bash
node dist/index.js demand get --id 663
# 看返回 data.status 字段
```

### 幂等性（写入前必查）

写入前**必须**先查该供应商已有的报价记录：
```bash
node dist/index.js demand vendor-quotes --id 663
# 看返回 quotedVendors 里有没有当前 vendorId
# 如果有，再 get-item 看具体字段值
```

如果有，反问业务人员：
> "供应商 S00925 在需求 663 上已有报价记录（写入时间 <X>），这次要怎么操作？
> A. 完全覆盖（用新值替换所有字段）
> B. 增量更新（只更新本次提供的字段，保留原有其它字段）
> C. 取消写入"

**禁止**：不查就写入，避免悄悄覆盖业务人员之前手工维护的数据。

## Phase 1：解析报价文本

由 `wkea-demand-expert` 执行。

输入：
- 需求 ID（必填）
- 供应商 ID（必填）
- 供应商原始报价文字

提取每行的型号、价格、货期、库存，整理成结构化列表。每行格式预估是：

```
VQC4201-51          220    现货
SY9120-5DZD-03           135     现货
KQ2H12-04S--KQ2H12-04AS         6.5     现货
```

AI 解析输出（每条）：
```
- 型号：VQC4201-51
- 单价：220
- 货期：现货（需要反问）
- 库存：？
```

### 字段类型硬约束（写入时必须用正确类型，否则后端报错）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | **Integer** | 需求行项目 itemId（不是 data 表主键） |
| `price` | **Number（保留 2 位小数）** | 例 220 → 220，不是 "220" |
| `delivery` | **Integer（天）** | 例 3（天），不是 "3天" 或 "现货" |
| `stock` | **Integer（个）** | 默认 0，不能空 |
| `remark` 等字符串 | String | null 转空字符串 "" |

`delivery` 必须传数字天数。**严禁**传 "现货"、"1-2 周"、"依量定"。这些情况必须先反问到具体天数再写入。

### 找行项目 itemId 的命令路径

需求 663 的所有行项目（含 itemId 和原始型号）通过：
```bash
node dist/index.js demand items --demand-id 663
```

返回每条行项目（关键字段）：
- `id`：行项目 itemId（写入 quote-save-info 的 `id` 字段就是这个）
- `productName` / `productBrand` / `productModel`：用于匹配供应商给的型号

匹配逻辑：把供应商给的型号与行项目 `productModel` **精确比对**。完全相同 → 写入该 itemId；有任何不同（连字符、大小写、命名空间、少前缀/后缀）→ **跳过**，不写入、不反问。

**核心原则**：精确匹配才能写入。宁可跳过也不模糊处理。跳过的型号汇总到 HTML 报告的"待二次判断"段，让业务人员自行决定。

## Phase 2：字段换算

由 `wkea-demand-expert` 执行。

按上面"字段换算规则"逐条处理：
- 数字直接转
- 范围值 → 反问
- 现货/有货 → 必须反问 1/2/3/5/7 天（多个产品只问一次）
- 库存充足类 → 必要时反问大致数量
- 模糊的型号 → 跳过，不反问

如果有字段无法判断，统一进入 Phase 3 反问清单。

---

## Phase 3：反问业务人员

把以下疑问一次性用提问工具列出（不要分多次问）：

1. **现货的天数**（如果检测到"现货/有货/库存"且没说具体天数）：用第一题的问法
2. **范围货期**（如"1-2 周"）：取中位数还是其它
3. **价格区间**（如"约 200-220"）：取中位、取上下界、还是反问供应商
4. **必填字段缺**（如某些价格没报）：是默认空还是反问供应商

**型号不匹配不问**：型号不匹配直接跳过，不反问。由报告列出跳过的型号，业务人员自行决定处理。

**业务人员批量指令合法**：业务人员说"3 个都按 1 天发"这种批量答复算合法输入，直接采纳并在报告里标注来源。

**反问原则**：把所有不确定一次性列，避免来回反问。同一次提问的现货天数在本次后续都按这个值复用。

---

## Phase 4：写入询价单

调用 `node dist/index.js demand quote-save-info`：
- `--demand-id <id>`
- `--vendor-id <id>`
- `--info-list <JSON>` 数组，每个对象是 `{id: <itemId>, price, delivery, stock, remark, ...}`

**字段类型必须用 Phase 1 的类型表**：`delivery` 是数字不是字符串；`stock` 默认 0；`id` 是 itemId 不是 data 表主键。

**只写精确匹配的**：型号与系统行项目 `productModel` 完全相同时才写入。任何不同都跳过，不写入，不反问。跳过的型号汇总到报告"待二次判断"段。

**跨供应商现货答案**：不同供应商必须重新反问现货天数，即使同会话内。

验证：再 `demand get-item` 看字段真的写了。

## 异常处理

| 场景 | 处理 |
|------|------|
| 业务人员给了报价文字但没说需求/供应商 | 反问，给需求 ID 和供应商 ID |
| 供应商 ID 错误（系统查不到） | 反问让用户重新确认 |
| 需求 ID 没找到 | 反问 |
| 所有型号都匹配不上 | 报告里说明哪些型号已跳过，让业务人员确认 |
| 现货货期业务人员选了"其他" | 记下用户输入的数字，本次所有"现货"按这个填 |
| 多行报价文本格式混乱（如多个空格、对齐错乱） | 按列分析优先，按空格分隔其次，每个数字都视为候选 |
| 写入后查询发现字段不对 | 用 `demand update-item` 修正指定字段，不重新走全流程 |

## 经验教训

1. **现货必须反问**：不同供应商含义不同，不存在统一答案
2. **多产品同次反问只问一次现货天数**：避免重复打扰业务人员
3. **未匹配型号宁可跳过**：不匹配的写入等于污染数据
4. **HTML 报告核心是"哪些数据需要二次判断"**：业务人员最关心这个
