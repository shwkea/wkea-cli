# Workflow 01：需求询价处理（13 步主流程）

**触发条件**：用户提供一个新需求（文字/表格/图片/文件），需要从需求分析到生成报告的完整流程。

**铁律**：13 步**禁止删减、合并、重排**。每完成一步立即 `progress step`，不攒到一批。

**根据已有信息直接执行，不反复询问**：用户给了什么信息就直接开工，不追问品牌、数量、价格、渠道等缺失字段。缺失的按默认值处理（如渠道默认"其他"、数量默认"1"），不停下来问用户。

**中文输出铁律**：所有面向业务人员的输出（思考过程、处理记录、HTML 报告）中，字段名、状态描述、说明文字必须使用中文。禁止出现英文字段名（如 `price`、`delivery`、`stock`、`skuId`、`originalText`、`status` 等），一律改为"价格"、"货期"、"库存"、"SKU 编号"、"客户原文"、"状态"等中文。aiRemark 内的 Markdown 标题和说明也必须是中文。

---

## 流程边界（必读）

**主流程**（同步完成、本轮结束点）：
- Phase 0 创建需求
- Phase 1 产品研究
- Phase 2 供应商匹配
- Phase 3 询价发送
- Phase 6 生成报告

**主流程结束点** = Phase 3 询价发送完成 + Phase 6 生成报告 = 本轮 demand-expert 工作完成。

**固定流程，Phase 间自动推进，禁止询问"接下来做什么"**：从 Phase 0 到 Phase 6 是固定路线，每完成一个 Phase 自动进入下一个，**不允许在 Phase 之间问用户"接下来需要做什么"**。唯一可以停下问用户的点是 Phase 6 报告生成后的收尾（此时本轮工作已全部完成）。

**异步后续流程**（不等供应商回复，不阻塞主流程）：
- Phase 4 报价采纳 — 等供应商回复后由**新会话**触发，不在本轮同步处理
- Phase 5 转报价单 — 采纳完成后由 `wkea-quotation-expert` 接手

**禁止**把"等供应商回复"塞进主流程挂起；发完询价、生成报告后就该收尾，**轮询/新事件**在外部触发。

## Phase 0：创建需求

由 `wkea-demand-expert` 执行需求登记部分，但涉及**系统搜索必须派给 product-expert**。

### ⚠️ 创建需求铁律：不问缺失字段

用户给了什么信息就登记什么，**禁止反问缺失字段**（数量、价格、交期、品牌、渠道等）。缺失的按默认值处理：
- 数量 → 默认 `1`
- 渠道（channelSource）→ 默认 `"其他"`
- 价格 → 留空，等供应商报价
- 品牌/分类 → 留空，在产品研究中确认

这条规则**优先级高于所有"先确认再执行"规则**。需求登记的唯一目的是"把用户说的记下来"，不是"把信息补全"。

### 快速搜索（≤10 个产品时先搜系统）

当产品数量**≤10 个**时，不走完整流程，先快速搜系统确认已有产品：

1. **demand-expert 解析需求** — 用 MCP `parse_demand` 或 CLI `demand parse` 拆解出所有产品
2. **leader 派 product-expert 做系统搜索** — product-expert 按 workflow 03 的 **2.1 系统内搜索**，对所有产品逐一搜系统，返回每个产品的匹配结果（有无 SKU、SKU ID、链接）
3. **leader 根据结果走分支**：

**分支 A：全部搜到（10/10 都有 SKU）**
- 不给业务人员建需求
- 直接给每个产品的后台链接 + 简单网页总结
- 问一句"系统都有，还需要做其他操作吗？"
- **结束**

**分支 B：有至少一个没搜到**
- **全部原文登记**，不能缺少任何一个产品（搜到的和没搜到的一起登）
- demand-expert 执行：
  - 创建需求：CLI `demand create`，传 channel-source + items
  - **保留客户原文**：从 `demand parse --json` 输出的 items JSON 中，**必须保留 `originalText` 字段传给 `demand create --items`**，不得丢弃
  - **直接绑定已搜到的**：对已匹配 SKU 的产品执行 `demand update-item --sku-id <id>` 绑定
  - 验证：`demand items --demand-id <id>` 确认每条 `originalText` 有值、已绑的 `skuId` 不为空
  - 进入 Phase 1

> 注意：Phase 0 以前的"不跨 expert"规则在此场景不适用。系统搜索是 product-expert 的专业范畴（workflow 03 2.1），demand-expert 不做搜索。

## Phase 1：产品研究

由 `wkea-product-expert` 主导（**不要在 demand-expert 里跑产品研究**，跨权）。

产品研究本身按 **workflow 03（产品开发）** 的 SOP 执行（系统内搜索 → 网上搜品牌 → 官网验证 → 规格核对 → B2B 交叉验证 → 替代品检查）。本 Phase 只写需求特有的后续操作。

### 已绑 SKU 的产品跳过系统搜索

如果 Phase 0 中已经通过 `demand update-item --sku-id` 绑定了 SKU，**该产品直接跳过 workflow 03 的 2.1 系统内搜索步骤**（已经搜过了），但仍可能需要走 2.2-2.6 的网上研究（官网验证、规格核对等），由 product-expert 按需判断。

**怎么判断**：检查 `demand items` 输出中该 item 的 `skuId` 字段，有值则说明已绑定，跳过 2.1。

### 需求特有步骤（product-expert 在研究完成后执行）

- 系统已有匹配 SKU 且未绑定 → `demand update-item` 绑定到行项目
- 系统没有 → `product quick-create --specs`（有选型手册）或 `demand simple-create-product`（单型号）
- 创建/绑定后立即 `product get` / `demand get-item` 验证
- 立即输出跳转链接：`{manageMainUrl}#/main/product-edit/{skuId}`

### 产品配置器页面（新增标准步骤）

每个被研究/创建的产品，必须生成产品配置器页面：

- **已有系统 SKU 但规格不全** → 按官网资料提取规格 → 生成**研究模式**配置器页面
- **新创建的产品** → 系统规格数据完整 → 生成**完整模式**配置器页面
- **纯研究未创建** → 研究数据 → 生成**研究模式**配置器页面

页面路径记录到 progress step 的 summary 中，格式：`产品配置器页面: /tmp/wkea-product-page-{产品名slug}.html`

### 写入 aiRemark

将研究结果写入 aiRemark 区域 5（`## 产品研究`），按 workflow 03 Phase 2 的输出格式（品牌确认、型号核实、规格对比、datasheet/B2B 来源 URL）。**禁止**在区域 5 写供应商信息。

**输出**：每个行项目的 aiRemark（8 区域结构）+ SKU 绑定结果 + 产品研究链接

---

## ⚠️ aiRemark 跨阶段写入铁律（必读，违反会导致数据丢失）

**问题背景**：`demand update-item --ai-remark` 是整体覆盖写入（`src/commands/demand/crud.ts:273`），多 expert 跨阶段各写自己区域时后写覆盖前写，导致前期研究丢失。

### 区域标题（精确匹配，不允许改/加序号）

8 个区域用以下标题作锚点（与原模板一致）：

```
## 风险评估      ## 客户原文       ## 产品研究
## 二次确认      ## 供应商匹配     ## 处理时间线
## 供应商确认清单
```

**区域 7 拆两个子节**（与"主流程 vs 异步后续"边界对齐）：

```
### 询价发送    # 主流程 Phase 3 写
### 报价到达    # 异步 Phase 4 写
```

### 写入流程（4 步：读→改→写→验）

1. **读**：`node dist/index.js demand items --demand-id <id>` 提取 `aiRemark` 完整内容
2. **改**：用 `## {区域名}` 定位自己的区域 → 替换；找不到就 append（按 1→8 顺序）
3. **写**：`node dist/index.js demand update-item --item-id <id> --ai-remark "<合并后>"`
   - 长内容用 Node 脚本 + `execFileSync` 传参，避开 shell 引号坑
4. **验**：再跑一次 `demand items`，确认 8 区标题都在、其他区域未被覆盖

### 各 expert 负责区域（修正时机 + 内容红线）

| Expert | 负责区域 | 何时写入 | 内容红线 |
|--------|---------|---------|---------|
| demand-expert | 4 客户原文 | Phase 0 后 | 客户原话、产品名/品牌/型号/数量 |
| demand-expert | 1 风险评估 + 2 二次确认 + 3 供应商确认清单 | Phase 1+ 研究完成后 | 不写产品研究本身 |
| product-expert | 5 产品研究（5a/5b/5c） | Phase 1 后 | **不写供应商**（#662 根因） |
| vendor-expert | 6 供应商匹配 | Phase 2 后 | 不写产品规格 |
| demand-expert | 7.1 询价发送 + 8 时间线 | Phase 3 后 | 不写报价内容 |
| demand-expert（新会话） | 7.2 报价到达 + 8 时间线 | 异步 Phase 4 触发 | 不写产品研究 |

**红线铁律**：禁止在负责区域写其他 expert 的素材；跨区信息用"参见 X 区域"传递，不展开。

### 禁止行为

- ❌ `--ai-remark "只有自己区域的内容"`（整体覆盖）
- ❌ 不读就写 / 改别人的区域 / 跨区写素材
- ❌ 区域标题改名 / 加序号
- ❌ 向 `--to-vendor-remark`（供应商可见）或 `--remark`（客户可见）自行写入任何内容。这些字段是客户/供应商直接看到的，AI 的研究分析、产品来源、推测等只能写入 `--ai-remark`。除非用户明确要求，否则绝不碰这两个字段。

## Phase 2：供应商匹配

由 `wkea-vendor-expert` 主导：

1. 查询品牌已有供应商：CLI `demand vendors-by-brand`
2. 已有 ≥ 2 家 → 跳过 3-4，直接进 Phase 3
3. 不足 2 家 → Google 搜索 + 企查查核验
4. 创建供应商并绑定品牌（4.1 vendor create + 4.2 绑定）

**核验通过标准**：工商状态=存续/在业 + 有联系方式（电话/邮箱至少一个）+ 荣誉资质正常。

**输出**：每个行项目对应品牌的供应商候选列表（含联系方式、资质、来源 URL）

---

## Phase 3：询价

由 `wkea-demand-expert` 执行（核心在 demand-expert 内部，跨 expert 协作已经走完）：

1. 对比已询价供应商：CLI `demand get` 的 quotedVendors
2. 向新供应商发送询价：CLI `demand quote-to-vendor`（每次只对一个供应商）
3. 询价发出 → 本 Phase 结束；**不挂起等供应商**（等回复属于异步后续流程 Phase 4）

**输出**：询价记录（已发送/待发送）

---

## Phase 4：报价采纳与价格管理（异步后续，**不阻塞主流程**）

⚠️ **本 Phase 不属于主流程**。等供应商回复后由**新会话**触发，不由本轮 demand-expert 同步挂起等待。

由 `wkea-demand-expert` 执行（业务判断在 demand-expert 内部）：

1. 查看报价详情：CLI `demand vendor-quotes`
2. 评估与建议（按优先级排序）：价格低 > 交期短 > 已有采购记录 > 供应商资质
3. 用户确认采纳哪家后：
   - 全部报价 → `demand save-price` 逐条保存（保留全部记录，isMaster=false）
   - 采纳的供应商 → `product supply set-master` 设为主供应商价格

**输出**：采纳决策 + 保存的报价记录

---

## Phase 5：转报价单（异步后续，**不阻塞主流程**）

⚠️ **本 Phase 不属于主流程**。采纳完成后由 `wkea-quotation-expert` 接手处理，不由本轮 demand-expert 同步处理。

由 `wkea-quotation-expert` 执行：

1. 从需求行项目生成报价单：CLI `demand share-order`
2. 验证：CLI `quotation get`

**输出**：shareId + 后台查看链接 + 客户查看链接

---

## Phase 6：生成处理报告（强制，不可跳过）

由 `wkea-demand-expert` 执行（**最后一步必做**，不生成报告 = 流程未完成）：

1. 读取报告规范：`docs/report-spec.md`（信息规范 + 样式规范 + **链接规范**）
2. 读取报告模板：`docs/report-template.html`
3. **跑 `node dist/index.js urls` 获取 manageMainUrl**，所有后台链接用这个返回值拼接（**禁止猜测URL**）
4. 收集数据：demand get / demand items / progress get / aiRemark + 用户原文 + 图片
5. 填充模板：HTML 结构和 CSS 不动，只替换 `{{占位符}}`
6. aiRemark 用 marked 库渲染为 HTML，填入每个产品行下方的**折叠行**（`ai-detail-row`）中
7. **补全头部**：生成时间精确到秒、用户原文回放、图片附件嵌入
8. **每个产品行**：附上对应配置器页面链接（🔗列）+ 折叠 AI 详情触发器
9. **所有 `<a>` 链接必须加 `target="_blank"`**
10. **补全页脚**：生成时间 + 免责声明
11. 写入文件（如 `/tmp/wkea-demand-{ID}-report.html`），图片附件放 `/tmp/uploads/`
9. 输出报告路径 + 进度最后一步的 summary 记录报告路径

### 客户原文突出原则

报告表格中 `客户原文` 列必须填充，**每行都展示该行项目的客户原文**（即 `originalText` 字段），用对比方式呈现——左边是"客户说了什么"，右边是"系统拆解后是什么"（产品名/品牌/型号等）：

| 客户原文（原话） | 系统拆解后字段 |
|----------------|--------------|
| "SMC 电磁阀 VQC4201 220 元 要 5 个" | 产品名=电磁阀、品牌=SMC、型号=VQC4201、价格=220、数量=5 |

如果该行的 `originalText` 为空，在报告中用黄色标记 ⚠️ "客户原文缺失"，提示业务人员补录。
**禁止**在报告中使用英文字段名展示数据。

**输出**：报告文件路径 + 进度最后一步的 summary 记录报告路径

## 进度跟踪（贯穿全程）

**创建进度时机**：Phase 0 完成后立即创建，由 demand-expert 执行。

`progress create` 一次性创建所有 13 个步骤（**禁止删减合并**）：
```
1. 获取需求详情
2.1 系统内搜索产品
2.2 网上搜索生产商品牌（优先 Google）
2.3 逐个品牌官网验证
2.4 查找技术文档和规格书
2.5 B2B平台交叉验证
2.6 规格对比并记录研究结果
2.7 检查客户疑点（有疑点则暂停通知）
2.8 产品上架或绑定
3.1 查询品牌已有供应商
3.2 网上搜索新供应商（Google 优先，Bing 备选）
3.3 企查查核验供应商资质
3.4 创建供应商并绑定品牌
4.1 对比已询价供应商
4.2 向新供应商发送询价
5. 生成处理报告
```

**progress step 调用规则**：
- 每完成一小步立即 `progress step` 更新，不攒到一批
- summary 必须包含**验证证据**：搜到了什么（具体型号/参数）、搜到了几个（数字）、来源 URL
- 链接必须来自搜索工具或 web_fetch 的实际返回（**禁止编造 URL**）
- 步骤 2.7（二次确认）标记为 `--abnormal` 时，是有疑点通知客户，**不阻塞流程**——无论客户是否回复，都继续往下走

## 异常处理

| 场景 | 处理 |
|------|------|
| ES 搜索 500（开发环境）| 走 WebSearch |
| 品牌无供应商 | 转 vendor-expert 继续开发 |
| 客户疑点 | 标记 2.7 abnormal，通知企业微信，继续往下走 |
| 进度某 step 卡住 | 标记 `--abnormal`，跳过该阶段 |
| 全部供应商核验失败 | 标记异常并通知，不创建 |
