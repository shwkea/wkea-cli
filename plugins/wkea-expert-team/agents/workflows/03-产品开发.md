# Workflow 03：产品开发

**触发条件**：
- 用户给了产品型号、图片、参数描述，想了解这个产品
- 用户说"帮我查一下这个产品""这个是什么""有没有这个型号"
- 用户丢了一个或多个产品，没有明确说要创建还是询价
- 需求询价流程（workflow 01）中需要做产品研究时，也走本流程

**视角**：产品（`wkea-product-expert` 主导，纯研究不写库）

**本流程不做什么**：不创建 SPU/SKU、不绑定需求行项目、不写 aiRemark、不开发供应商。这些是调用方（如 workflow 01）的事。

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 信息整理
- Phase 2 逐个产品深度研究
- Phase 3 汇编研究报告

**主流程结束点** = Phase 3 输出研究报告。是否继续（登记需求/上架/深挖）由调用方或用户决定。

---

## Phase 1：信息整理

由 `wkea-product-expert` 执行。

### Step 1.1 提取产品线索

从用户输入中提取每个产品的已知线索，列出已有信息和缺失字段：

| 产品 | 已知线索 | 待补全 |
|------|---------|--------|
| 产品 1 | 型号=XXX、图片=有 | 品牌未知、规格参数缺失 |
| 产品 2 | ... | ... |

### Step 1.2 判断场景（关键，决定后续是否启用快速停止）

根据**调用来源**和**用户意图**判断走哪种模式：

| 调用来源 | 用户意图 | 模式 | Phase 2 行为 |
|---------|---------|------|------------|
| **直接调用（用户在主理人路由进 03）** | 纯了解 | **快速模式** | Phase 2.1 搜到系统已有 → 反问是否还要网上搜，可跳过 2.2-2.6 |
| **被 workflow 01 需求询价调用** | 询价前研究 | **完整模式** | 不跳过任何 Phase，按完整 SOP 执行 |
| **被 workflow 02/05 调用**（建议前置研究）| 上架前研究 | **完整模式** | 不跳过任何 Phase，按完整 SOP 执行 |
| **被 product-expert 内部流程 0 调用** | 纯了解 | **快速模式** | 同直接调用 |

**快速模式铁律**：仅在 Phase 2.1 搜到**完全匹配** SKU 且用户意图是"只是想了解"时启用。如果是"完全一致或等价"的规格匹配（✅）可启用；规格匹配是 ⚠️ 或 ❓ 时仍走完整 6 步。

**完整模式铁律**：永远走完所有 6 步，不允许在 Phase 2.1 反问业务人员。任何写操作（创建/上架/绑定/询价）必须基于完整研究结果。

主理人或 product-expert 在调度本流程时，必须先判断场景并选定模式，作为 Phase 1 输出的一部分传递给 Phase 2。

---

## Phase 2：逐个产品深度研究

由 `wkea-product-expert` 执行。**每个产品独立研究，不要合并**。

### 搜索关键词方法论（铁律）

**核心原则：先原样搜、再理解搜、禁止第一步就加自己的猜测。**

| 轮次 | 搜索内容 | 目的 | 禁止行为 |
|------|---------|------|---------|
| **第一轮：原样搜索** | 只用用户给的原始文本（型号、关键词），**不加任何AI理解的品牌名/产品名/分类**。如用户只给"FKC-G02"，第一轮就只搜 `FKC-G02` | 看全网怎么描述这个型号，让数据告诉你品牌是什么 | ❌ 自己猜品牌然后搜"欣三立 FKC-G02"（先入为主导致后续全偏）|
| **第二轮：品牌验证** | 根据第一轮结果中出现的品牌/厂商信息，搜 `{型号} + {品牌}`、`{型号} + datasheet`、`{型号} + 规格书` | 验证品牌真实性，锁定官网 | ❌ 跳过第一轮直接搜品牌 |
| **第三轮：深挖** | 搜到品牌后，`{品牌} + {系列} + 文档`、官网产品中心确认 | 提取完整规格参数 | ❌ 用一个B2B页面就确认全部信息 |

**搜索引擎选择**（网上搜索一律用 Google/Bing，辅以工业品网站）：

| 搜索目标 | 推荐渠道 |
|---------|---------|
| 系统内已有产品 | ES搜索（CLI `product spu list --keyword` / `product sku list --keyword`）|
| 一切网上搜索（品牌、官网、datasheet、技术文档）| **Google**（首选，中英文均覆盖，外资品牌用英文搜一轮）→ **Bing**（Google 搜不到时换）|
| B2B价格、供应商数量 | 1688、淘宝、爱采购、icspec、allchips 等工业品网站 |

### 研究步骤（每个产品都走一遍）

```
2.1 系统内搜索
    - ES 搜索型号、缩短型号、去掉分隔符的变体
    - 搜到了 → 记录已有 SKU ID、价格、品牌、规格参数
    - 没搜到 → 标记"系统无此产品"

    ⚠️ 系统已有 + Phase 1 判定为「快速模式」时 → 立即停下反问业务人员
        → 展示系统内的 SKU 给业务人员看
        → 询问"系统已找到 X 个匹配，供参考。还需要去网上搜索更多信息吗？"
            - 用户说"不用" → 跳过 2.2-2.6，直接进 Phase 3 出报告
            - 用户说"要" → 继续走 2.2-2.6

    ⚠️ 系统已有 + Phase 1 判定为「完整模式」时（需求询价/上架前研究）
        → 标记"系统已有，规格匹配 = ✅/⚠️/❓"
        → 不反问、继续走 2.2-2.6 完整研究
        → 因为后续要写 aiRemark / 创建 SKU / 找供应商，必须有完整研究依据

2.2 网上搜索品牌（Google 优先，搜不到换 Bing）
    - 搜 {型号} + 品牌/厂商/producer/manufacturer
    - 搜 {型号} + datasheet/规格书/选型手册
    - 中英文双关键词（外资品牌用英文搜一轮）

2.3 官网验证
    - 找到品牌后 → 进官网产品中心确认这个型号真实存在
    - 找到 datasheet → 提取关键规格参数

2.4 规格核对
    - 用户给了规格 → 逐条对比，标 ✅/⚠️/❓
    - 用户没给规格 → 列出 datasheet 里的关键参数

2.5 B2B 平台交叉验证
    - 1688/淘宝/爱采购/icspec/allchips 搜同型号
    - 记录价格区间、是否有现货、供应商数量
    - 多个平台都搜不到 → 标记"全网信息稀少"

2.6 替代品检查
    - 系统内有没有同规格的其他品牌产品
    - datasheet 里有没有写替代型号
```

### 规格匹配标记

| 标记 | 含义 |
|------|------|
| ✅ | 完全一致或等价 |
| ⚠️ | 不完全一致（用词不同但可能同义、数值偏差、术语不统一）|
| ❓ | 无法确认（找不到规格、官网未列出）|

**⚠️ 标记铁律**：客户说和搜到的不一样——哪怕只差一个字——必须标 ⚠️，不能标 ✅。每个 ⚠️ 必带三要素：差异解释 + 影响评估 + 判断依据。

### 输出格式（每个产品）

```markdown
## 产品 N：{产品名称/型号}

| 项目 | 内容 |
|------|------|
| 确定品牌 | XXX（来源：官网）或 未知 |
| 确定型号 | XXX |
| 系统已有 | 是（SKU ID: XXX）/ 否 |
| 规格匹配 | ✅ 完全一致 / ⚠️ 差异项 / ❓ 待核实 |
| 价格参考 | ¥XXX-XXX（来源：1688/淘宝）|
| 供应商数量 | 系统 X 家 / 平台约 X 家 |
| 替代品 | 有（XXX）/ 无 |
| 信息来源 | [来源URL列表] |
```

## Phase 3：汇编研究报告 + 产品配置器页面

由 `wkea-product-expert` 或主理人执行。

### 3.1 生成 HTML 研究报告（已有）

生成 HTML 报告，风格统一使用 `docs/report-style.html` 的 CSS，内容包含：

1. **研究摘要** — 一共几个产品、研究到什么程度、关键发现
2. **逐个产品详情** — Phase 2 的每个产品研究结果
3. **信息完整度** — 哪些产品信息充分、哪些缺关键数据
4. **下一步建议** — 可以做什么（附产品配置器页面路径）

### 3.2 每个产品独立生成产品配置器页面（新增标准步骤）

这是**标准输出**，每个产品研究完成后必须生成。页面使用 `docs/product-page-template.html` 模板。

**数据来源对应关系**（Phase 2 研究成果 → 配置器数据）：

| 配置器区块 | 数据来源 | Phase 2 步骤 |
|-----------|---------|-------------|
| 产品头部（品牌、名称）| 品牌确认结果 | 2.2 网上搜索、2.3 官网验证 |
| 型号结构解析 | 型号拆解 + 规格参数提取 | 2.3 官网提取、2.4 规格核对 |
| 交互式配置器 | 型号结构 → 位置/类型/值 | 2.3-2.4 |
| 价格参考 | B2B 价格区间 + 供应商数量 | 2.5 B2B交叉验证 |
| SKU 清单 | 系统已有的 SKU（2.1） + 研究发现的新型号 | 2.1 系统搜索、2.3-2.4 官网提取 |
| 资料附件 | Datasheet URL、官网链接、B2B链接 | 2.2-2.3、2.5 |
| 操作入口 | "按此规格维护到系统" | 研究完成后的下一步 |

**关键规则**：
- 从 Phase 2 的研究结果中提取型号结构（逐位置分析），转换为 positions 数据
- 价格统一用"B2B参考价"区间标注（如 `¥200-350`），不写死具体价格
- SKU 实体标记为"研究发现"（`isSystem: false`），与系统已有 SKU 区分
- 如果系统已有该产品（2.1 搜到），切换到完整模式显示系统价格和链接

**文件命名**：
```
/tmp/wkea-product-page-{产品名slug}.html    # 每个产品独立页面
```

### 3.3 产出物总览

研究完成后，产出以下文件：

```
/tmp/wkea-product-dev-{日期}.html                    # 研究报告（汇总所有产品）
/tmp/wkea-product-page-{产品A}.html                   # 产品A 配置器页面
/tmp/wkea-product-page-{产品B}.html                   # 产品B 配置器页面
...
```

在研究报告的"下一步建议"中附每个产品的配置器页面路径。

### 如果是从 workflow 01 调用

研究完成后，把结果交给 demand-expert，由 demand-expert 负责：
- 将研究结果写入 aiRemark 区域 5（产品研究）
- 系统已有的 SKU → `demand update-item` 绑定
- 系统没有的 → `product quick-create` 或 `demand simple-create-product`
- 产品配置器页面路径记录到 progress step 的 summary 中

**输出**：每个产品的品牌确认、型号核实、规格对比、datasheet/B2B 来源 URL + 产品配置器页面路径

### 如果是独立使用

报告生成后，**反问用户下一步**：

> 以上是研究结果。请问接下来需要：
> 1. 登记为需求，帮你找供应商询价（推荐，走 workflow 01）
> 2. 把这些产品上架到系统（走 workflow 05 产品配置与上架）
> 3. 继续深挖某个产品
> 4. 仅了解，不需要后续操作

---

## 异常处理

| 场景 | 处理 |
|------|------|
| 型号全网搜不到 | 标记"信息稀少"，给用户看搜了哪些关键词，问是否换关键词 |
| 搜到多个品牌同名型号 | 列出所有品牌，让用户确认是哪个 |
| 图片里型号模糊 | 列出可能解读，让用户确认 |
| 没有 datasheet | 标记"无官方规格书"，用 B2B 平台参数兜底 |
| 产品系统内已存在 | 直接给 SKU ID + 价格 + 后台链接 |
