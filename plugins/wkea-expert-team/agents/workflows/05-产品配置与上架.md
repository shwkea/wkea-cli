# Workflow 05：产品配置与上架

**触发条件**：
- 用户提供厂商选型手册/型号对照表/选型配置表，要"配置化上架"
- 用户说"上架一个产品系列"且提供了多个型号 SKU 数据
- 需求询价处理中发现产品规格复杂，需要完整建模后再上架

**视角**：产品（`wkea-product-expert` 主导，需求/供应商视情况协作）

**前置条件**：用户说"上架"且有产品资料（PDF、选型手册、网址、文字、图片等任意形式），**直接走本流程**。Phase 1 负责解析资料、提取信息。只有在解析过程中发现**品牌和分类都无法确定**时，才暂停问用户或转 workflow 03 补研究。不要一看到 PDF 就默认"先研究再上架"——大部分 PDF 里信息是完整的。

**与 03 的分工**：
- **03**：用户说"了解/查一下" → 纯研究，不建库
- **05**：用户说"上架" → 资料解析 + 建库，资料不全时在 Phase 1 暂停确认

**与 02 的区别**：
- **02**：上架视角（已有产品模板 → 批量套用）—— Phase 1 vendor + product **并行**
- **05**：配置视角（选型资料 → 规格建模 → SKU 变型）—— Phase 1-2 product **主导**

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 选型资料解析
- Phase 2 SPU + 规格创建
- Phase 3 SKU 上架
- Phase 4 供应绑定
- Phase 5 产品资料完善
- Phase 6 验证与交付

**主流程结束点** = Phase 6 输出"已上架"清单和"待补充"清单。

**异步后续**：无（本 workflow 不涉及等供应商回复等异步场景）。

## Phase 1：选型资料接收 + 型号解析

由 `wkea-product-expert` 主导。

### Step 1.1 接收资料

接收用户提供的任何形式的产品资料：
- **PDF 文件**、选型手册、型号对照表、规格定义、网址链接
- Markdown 表格、Excel 截图、纯文本、图片
- kimi-webBridge 打开 PDF/网页直接查看内容

从资料中提取：产品名、品牌、分类、型号结构（如 `AMS[1]X-[2][3]-[4]L[5]-X2044`）。

**缺失处理**：先尝试从资料本身提取全部信息；仅当**品牌和分类都无法确定**时暂停问用户；型号结构/规格参数始终从资料解析，不反问用户。

### Step 1.2 型号结构解析（核心）

按"位置"逐位分析型号，每个位置判定为以下三类之一：

| 类型 | 标识 | 示例 |
|------|------|------|
| **可变规格**（`is_fixed=false`）| 存在多个可选值 | 主体尺寸 20/30/40/60 |
| **固定规格**（`is_fixed=true`）| 仅一个固定值 | AMS、X、L、X2044 |
| **分隔符** | 位置之间的连接符 | `-`、`.`、`/`、空格 |

输出**结构化规格表**（位置 / 类型 / 规格值列表 / 名称），作为 Phase 2 规格建模的输入。

### Step 1.3 关键判定原则

| 场景 | 判定 |
|------|------|
| 有完整选型配置表 | 型号中**所有内容**都是规格（含固定规格和分隔符）|
| 参数参与型号拼接 | → 规格 |
| 参数不参与型号拼接 | → 属性 |
| **值变了型号会变** | → 规格（可变或固定）|
| **值变了型号不变** | → 属性 |
| 型号中有固定值片段（如 AMS、X2044）| → 固定规格（`is_fixed=true`）|

### Step 1.4 工具

选型资料获取统一使用 kimi-webBridge 操作浏览器访问厂商网站，直接查看页面内容、下载 PDF、提取型号结构。

## Phase 2：SPU + 规格创建

由 `wkea-product-expert` 执行。

### Step 2.1 创建 SPU

**前置条件**：品牌和分类必须已存在

```bash
node dist/index.js product spu create \
  --name "<SPU 名称，不含品牌>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --series "<系列>" \
  --description "<描述>" \
  --pdf-link "<datasheet 链接>" \
  --details "<详情介绍>"
```

**命名规则（铁律）**：SPU 名称**不能包含品牌名**（品牌是绑定的）

### Step 2.2 规格创建：三种类型 + 完整 CLI 命令

型号由三部分组成：**可变规格 + 固定规格 + 分隔符**。创建时必须区分处理。

#### 概念对照表

| 类型 | 定义 | 例子 | 命令行标记 | 选择器 |
|------|------|------|-----------|--------|
| **可变规格** | 多个可选值，用户在不同场景选不同值 | 凸轮方向(A/B/R)、流量(2/4/6) | 不加 `--fixed` | ✅ 可切换 |
| **固定规格** | 型号中唯一值、不会变的片段，也是规格 | FKC系列前缀、G02口径 | `--fixed` / `--is-fixed` | 🔒 锁定不可选 |
| **分隔符** | 位置之间的连接符 | `-`、`/`、空格 | `separator set` | 纯显示 |

#### 推荐方式：`product spec add`（V1，一步完成创建+绑定+标记固定）

**可变规格示例**（凸轮方向，多个可选值）：
```bash
node dist/index.js product spec add \
  --spu-id WK999304 \
  --name "凸轮方向" \
  --tag "DIR" \
  --sort 2 \
  --param '[{"name":"A (推轮在左)","tag":"A","sort":1},{"name":"B (推轮在右)","tag":"B","sort":2},{"name":"R (滚轮)","tag":"R","sort":3}]'
```

**固定规格示例**（系列 FKC，只有一个值）：
```bash
node dist/index.js product spec add \
  --spu-id WK999304 \
  --name "系列" \
  --tag "SERIES" \
  --sort 1 \
  --fixed \
  --param '[{"name":"FKC","tag":"FKC","sort":1}]'
```

⚠️ **关键区别**：固定规格必须加 `--fixed` 标记，且 `--param` 只有一个值。不加 `--fixed` 则 AI 会创建成可变规格。

#### 备选方式：`product spu spec create` + `param add`（V2，两步操作）

如果使用 V2 API，创建和绑定分两步：

```bash
# 1. 创建规格（固定规格加 --is-fixed）
node dist/index.js product spu spec create --spu-id WK999304 --name "系列" --manage-name "SERIES_WK999304" --sort 1 --is-fixed
# 2. 添加规格值
node dist/index.js product spec param add --spec-id <返回的specId> --name "FKC" --tag "FKC" --sort 1
# 3. 绑定到 SPU
node dist/index.js product spu spec bind --spu-id WK999304 --spec-id <返回的specId>
```

⚠️ `product spu spec bind` **不支持** `--is-fixed` 参数。如果规格创建时没标记固定，绑定后也无法补标记（需用 `product spu spec update --is-fixed` 事后修正）。

### Step 2.2.5 规格值 name/tag 合规校验（必做）

创建完规格值后，**必须**逐条校验：

| 校验项 | 通过条件 | 若违规 |
|-------|---------|-------|
| `tag` 不含空格 | `"G02"` ✅ / `"G02 1/4\""` ❌ | 截断到第一个空格前 |
| `tag` 不含中文 | `"R"` ✅ / `"R 滚轮"` ❌ | 提取纯代码部分 |
| `tag` 不含单位/描述文字 | `"001"` ✅ / `"001 0.1~1 l/min"` ❌ | 只取型号码部分 |
| `name` 含描述（有额外信息时）| `"G02 (1/4\")"` ✅ / `"G02"` ❌ | 追加括号描述 |
| 固定规格有 `isFixed=true` | 用 `spec list` 验证 | 补调 `spec update --is-fixed` |

**自检命令：**
```bash
node dist/index.js product spu spec list --spu-id <id>
```
人肉看 specParams 中每个 `tag` 字段是否干净，固定规格的 `isFixed` 是否为 true。

**正确 vs 错误对照：**
```
✅ name="G02 (1/4″)",     tag="G02"       ← 代码+描述型（推荐）
✅ name="红色",            tag="RED"       ← 纯描述型（tag 抽象为英文）
✅ name="20",             tag="20"        ← 纯代码型
❌ name="G02",            tag="G02 1/4″"  ← tag 混入描述（最常见错误）
❌ name="001",            tag="001 0.1~1" ← tag 混入范围值
```

### Step 2.3 设置分隔符

分隔符存储在 SPU 级别的 JSON 配置中，**独立于规格**。创建完所有规格后设置。

**分隔符枚举 ID**（`enum --type 分隔符`）：
| ID | 字符 | 说明 |
|----|------|------|
| 101 | `-` | 短横线（最常用） |
| 102 | `*` | 星号 |
| 103 | `/` | 斜杠 |
| 104 | ` ` | 空格 |
| 463 | `+` | 加号 |
| null | (无) | 位置间无分隔符 |

**设置示例**（4 个规格之间有 3 个分隔符）：
```bash
node dist/index.js product spu separator set \
  --spu-id WK999304 \
  --spec-fg "-,null,-,-" \
  --spec-fg-ids "101,null,101,101"
```
含义：位置1-2之间=横杠，2-3之间=无分隔符，3-4之间=横杠。
注意：`spec-fg` 数组长度 = `spec-fg-ids` 数组长度 = 规格数量（每个规格后面跟一个分隔符）。

**验证：**
```bash
node dist/index.js product spu separator get --spu-id <id>
```

### Step 2.4 验证（四项必查）

```bash
# 1. 规格绑定列表（确认数量和排序）
node dist/index.js product spu spec list --spu-id <id>
# → 检查：可变规格数量、固定规格 isFixed 标记、排序序号

# 2. 型号结构预览（确认拼接效果）
node dist/index.js product spu spec model --spu-id <id>
# → 检查：型号模板字符串是否符合预期

# 3. 分隔符配置
node dist/index.js product spu separator get --spu-id <id>
# → 检查：specFg 数组、specFgIds 数组

# 4. SPU 完整信息
node dist/index.js product spu get --spu-id <id>
# → 检查：skus 是否已创建、specs 是否正确
```

## Phase 3：SKU 上架

由 `wkea-product-expert` 执行。

### Step 3.1 选择创建方式

| 场景 | 推荐方式 | 关键命令 |
|------|---------|---------|
| 有完整数据（名称+品牌+分类+型号+价格+库存）| **quick-create**（具体 SKU 模式）| `product quick-create` |
| 只有规格定义，无具体 SKU 数据 | quick-create 变型模式（只建 SPU + 规格）| `product quick-create --specs` |
| 已有 SPU，新增 SKU | 单独创建 | `product sku create` |
| 枚举所有规格值组合批量生成 | 4×3×2=24 个有限组合一次性铺 | 脚本 + `product spu spec model` 模板 |

**quick-create 两种模式**：
- **变型模式**：只创建 SPU + 规格定义，**不生成 SKU**
- **具体 SKU 模式**：创建 SPU + 规格 + 直接生成有型号/价格/库存的 SKU

### Step 3.2 创建 SKU（以具体 SKU 模式为例）

```bash
node dist/index.js product quick-create \
  --spu-name "<SPU 名称>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --vendor-id <vendorId> \
  --specs '{"主体尺寸":["20","30","40"],"螺纹":["M"]}' \
  -s '{"name":"SKU1","model":"AMS-20-M-X","salesPrice":100,"purchasePrice":50,"stock":10}' \
  -s '{"name":"SKU2","model":"AMS-30-M-X","salesPrice":120,"purchasePrice":60,"stock":5}'
```

### Step 3.3 SKU 扩展字段（30+ 字段）

通过 `-s` JSON 传递，关键字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| `name` | SKU 名称（必填）| `品牌名 + SPU名 + 型号` |
| `model` | 型号 | `AMS-20-M-X` |
| `specs` | 规格值 | `{"主体尺寸":["20"]}` |
| `attributes` | 属性列表 | `[{"name":"材质","value":"不锈钢"}]` |
| `salesPrice` / `purchasePrice` | 售价/采购价 | 100 / 50 |
| `stock` | 库存 | 10 |
| **`unit`** | **单位（必填）** | **469（pcs，工业品默认）** |
| `images` | 图片集合 | `url1,url2,url3` |
| `purchaseLink` | 采购链接 | `https://...` |
| `manufacturerModel` | 制造商型号 | `AMS-20` |
| `minOrderQuantity` | 最小起订量 | 1 |
| `lengthWidthHeight` | 长宽高 | `100x50x30cm` |
| `weight` | 重量(kg) | 0.5 |
| `deliveryMethod` | 发货方式（枚举）| 1 |
| `purchaseDeliver` | 采购交期（枚举）| 7 |
| `barcode` | 条码 | `6901234567890` |
| `itemNumber` | 货号 | `AMS-001` |
| `life` | 质保期(天) | 365 |
| `safetyStock` / `ceilingStock` | 库存上下限 | 5 / 100 |
| `taxRate` / `purchaseTaxRate` | 销售/采购税率（枚举）| 13 |
| `templateId` | 运费模板 | 1 |
| `esKeyword` | ES 搜索关键词 | `AMS 气缸` |

### Step 3.4 验证

```bash
node dist/index.js product sku get --sku-id <id>      # 单 SKU 验证
node dist/index.js product sku list --spu-id <id>     # SPU 下所有 SKU
```

## Phase 4：供应绑定

由 `wkea-product-expert` 主导（`wkea-vendor-expert` 协作）。

### Step 4.1 SPU-供应商绑定（先）

```bash
node dist/index.js product spu bind-vendor --spu-id <id> --vendor-id <id>
```

### Step 4.2 SKU-供应信息

```bash
node dist/index.js product sku supply set \
  --sku-id <id> \
  --vendor-id <id> \
  --purchase-price <price> \
  --min-order <quantity> \
  --delivery-days <days>
```

### Step 4.3 优势分类（可选）

如该供应商擅长该分类：

```bash
node dist/index.js vendor superior-category add \
  --vendor-id <id> \
  --name "气动元件" \
  --system-category-path "1/5/23" \
  --priority 1
```

### Step 4.4 解绑级联

⚠️ 解绑 SPU-供应商会**级联清空**该 SPU 下所有 SKU 的供应信息，**务必提前备份**。

## Phase 5：产品资料完善

由 `wkea-product-expert` 执行。

### Step 5.1 SPU 级资料字段

| 字段 | CLI 参数 | 说明 |
|------|---------|------|
| 描述 | `--description` | 简短描述 |
| 详情介绍 | `--details` | 富文本 |
| PDF 链接（datasheet）| `--pdf-link` | datasheet URL |
| 系列 | `--series` | 系列产品名 |
| 图片 | `--images` | 逗号分隔多图 URL |
| 型号备注 | `--model-remark` | 选型备注 |
| 资质文件 | `--qualification-path` | 资质文件路径 |
| 信息文件 | `--information-files` | 多个信息文件 |
| 标签 | `--labels` | 标签 ID 列表 |
| ES 搜索关键词 | `--es-keyword` | 搜索优化 |
| 维嘉折扣 | `--wkea-discount` | 折扣率 |
| 是否可退货 | `--can-be-returned` | bool |

```bash
node dist/index.js product spu update \
  --spu-id <id> \
  --description "<描述>" \
  --pdf-link "<datasheet URL>" \
  --details "<富文本>" \
  --images "url1,url2" \
  --es-keyword "AMS 气缸"
```

### Step 5.2 属性管理

不参与型号拼接的参数维护为属性：

| 类型 | 例子 | 原因 |
|------|------|------|
| 产地 | 江苏苏州 | 值变了型号不变 |
| 材质 | 不锈钢 | 值变了型号不变 |
| 保修年限 | 1 年 | 值变了型号不变 |
| 适用场景 | 工业自动化 | 值变了型号不变 |

```bash
node dist/index.js product attribute set \
  --spu-id <id> \
  --attributes '[{"name":"产地","value":"江苏苏州"},{"name":"材质","value":"不锈钢"}]'
```

### Step 5.3 标签管理

```bash
node dist/index.js product spu update --spu-id <id> --labels "1,2,3"
```

## Phase 6：验证与交付

由 `wkea-product-expert` 主导，主理人汇编。

### Step 6.1 验证清单

- [ ] SPU ID 已生成
- [ ] 规格已绑定（`product spu spec list`）
- [ ] 型号结构正确（`product spu spec model`）
- [ ] SKU 列表完整（`product sku list --spu-id <id>`）
- [ ] 供应信息已绑定（`product spu vendors` + `product sku supply list`）
- [ ] 优势分类已设置（如适用）
- [ ] 产品资料完整（描述、PDF、图片、标签）
- [ ] 属性已维护

### Step 6.2 交付输出（HTML 报告）

生成 HTML 报告，使用 `docs/report-style.html` 的 CSS，内容包含：

- SPU ID + SKU 列表（含型号/价格/库存）
- 规格结构（可变/固定/分隔符）
- 供应商绑定情况
- 已上架清单 + 待补充清单
- 跳转链接

文件命名：`/tmp/wkea-product-listing-{spuId或日期}.html`

### Step 6.3 "待补充"清单

输出**未完成**的项目：
- 缺 PDF 链接
- 缺图片
- 缺属性
- 缺替代品
- 缺优势分类
- 库存为 0

## 分支流程

### 分支 A：仅建规格体系，不生成 SKU

> **触发条件**：用户只想先建立 SPU 的规格框架，后续再分批补 SKU。

- 走 Phase 1（选型资料解析）→ Phase 2（SPU+规格创建）
- **跳过失** Phase 3-4-5（不创建 SKU、不绑供应、不完善资料）
- Phase 2 创建规格后，输出 SPU 规格结构供用户确认
- 后续可在任意时刻调用 `product sku create` 添加具体 SKU

### 分支 B：SKU 已上架，仅绑定供应信息

> **触发条件**：SPU 和 SKU 已存在，用户要补供应商或调整价格。

- **跳过失** Phase 1-2-3（不解析选型资料、不创建 SPU/SKU）
- 直接走 Phase 4（供应绑定）：
  1. 先查 `product spu vendors` 确认供应商绑定情况
  2. 未绑 SPU-供应商则补 `product spu bind-vendor`
  3. 调用 `product sku supply set` 维护每个 SKU 的供应价
  4. 验证：`product sku supply list` / `product sku supply summary`

### 分支 C：停产替代管理

> **触发条件**：用户反馈某产品停产或被新系列替代。

**SPU 级停产替代**：
```bash
# 停产且有替代
node dist/index.js product spu update --spu-id <id> --stop-production <替代SPU_ID>
# 停产且无替代
node dist/index.js product spu update --spu-id <id> --stop-production 0
# 取消停产标记（不传或传空）
node dist/index.js product spu update --spu-id <id> --stop-production ""
```
设置后前端详情页展示停产标识和替代品链接。

**SKU 级替代关系**：
```bash
node dist/index.js product sku replace add --sku <SKU_ID> --replace-sku <SKU_ID> [--full-replace]
node dist/index.js product sku replace list --sku <SKU_ID>
node dist/index.js product sku replace remove --sku <SKU_ID> --replace-sku <SKU_ID>
```
`--full-replace` 标记功能、性能完全等同的完全替代。替代关系是**双向展示**的。

## Phase 7：产品配置器页面输出（标准）

> 本 Phase 是**标准输出步骤**（不是可选的）。所有产品开发完成后，必须生成产品配置器页面，展示产品全部结果。无论产品是"系统已上架"还是"研究阶段"都要生成。

### 7.1 数据收集

### 7.4 填充方法

模板使用 `{{PLACEHOLDER}}` 占位符，按以下规则填充：

**简单替换字段**（直接文本替换）：
- `{{PRODUCT_NAME}}` — SPU名称
- `{{BRAND_NAME}}` — 品牌名
- `{{CATEGORY_NAME}}` — 分类名
- `{{SERIES}}` — 系列

  - `prices{}` — 价格参考
  - `attributes[]` — 属性列表

**条件区块**（根据数据完整性填充不同内容）：

- `{{STRUCTURE_ROWS}}` — 型号结构表行（`<tr>` 列表）
- `{{SPEC_SELECTORS}}` — 规格选择按钮组（服务端预渲染或JS动态渲染）
- `{{SEPARATOR_STRIP}}` — 分隔符序列（服务端预渲染或JS动态渲染）
- `{{PRICE_CONTENT}}` — 价格区块HTML
- `{{SKU_TABLE}}` — SKU列表表格HTML
- `{{SUPPLY_CONTENT}}` — 供应信息区块HTML
- `{{ATTACHMENT_CONTENT}}` — 资料附件区块HTML
- `{{ACTION_CONTENT}}` — 操作入口区块HTML

**模式标记**：
- `{{MODE_CLASS}}` = `"research"` 或 `"full"`
- `{{MODE_LABEL}}` = `"研究模式"` 或 `"完整模式"`
- `{{SOURCE_CLASS}}` = `"research"` 或 `"system"`

### 7.5 分隔符呈现规则（参考实现）

型号 = 固定/可变规格值 + 分隔符，页面中必须显式可视化。实现 `renderSeparatorStrip()` 函数：

```javascript
function renderSeparatorStrip(positions, separators) {
  // 按位置顺序渲染，每个位置显示为：
  //   [规格值] 或 [分隔符]
  // 规格值用蓝色底色（#e3f2fd），分隔符用灰色高亮（#eef1f4 + #6c8cff 文字）
  // 固定规格值用灰色底色（#f7f8fa）表示不可变
  //
  // 处理规则：
  // 1. 连续空规格 → 分隔符合并为单个 '-'
  // 2. 开头/结尾的 '-' 删除
  // 3. 分隔符字符用独立的 .sep-char 包裹，视觉上与规格值区分
}
```

**分隔符 HTML 结构**：
```html
<div class="separator-strip">
  <span class="spec-val">DSBC</span>
  <span class="sep-char">━</span>
  <span class="spec-val">32</span>
  <span class="sep-char">━</span>
  <span class="spec-val">100</span>
  ...
</div>
```

### 7.6 JS 核心组件表（参考实现）

配置器页面的交互逻辑由以下 JS 函数实现：

| 函数 | 职责 | 触发时机 |
|------|------|---------|
| `buildModel(selectedValues, positions, separators)` | 按位置顺序 + 分隔符拼装完整型号 | 规格值变化时 |
| `initSpecForm(positions, selectedValues)` | 渲染规格选择器按钮组，绑定事件 | 页面加载 |
| `updateModel()` | 调用 buildModel + 触发匹配 + 更新UI | 规格值变化时 |
| `matchSku(model, skus)` | 精确匹配 → 模糊匹配（忽略分隔符）→ 展示匹配结果 | 型号更新时 |
| `renderSeparatorStrip(positions, separators)` | 可视化分隔符序列 | 页面加载 |
| `renderSkuTable(skus, container)` | 渲染 SKU 完整表格（型号/价格/库存/状态/来源） | 页面加载 |
| `matchCustomerModel(inputModel, positions, skus)` | 逐位置拆解客户输入型号 → 标记 ✅/❌ | 用户点击匹配按钮 |
| `initPage()` | 整体初始化，调用上述各函数 | DOMContentLoaded |

**`buildModel()` 参考实现**：
```javascript
function buildModel(selectedValues, positions, separators) {
  let result = '';
  let prevNonSep = null;
  for (const pos of positions) {
    if (pos.type === 'separator') {
      if (prevNonSep !== null && selectedValues[prevNonSep]) {
        result += separators[`${prevNonSep}-${pos.pos}`] || '-';
      }
      continue;
    }
    const code = selectedValues[pos.pos];
    if (code) {
      result += code;
      prevNonSep = pos.pos;
    }
  }
  // 清理：连续分隔符合并、头尾删除
  return result.replace(/-+/g, '-').replace(/^-|-$/g, '');
}
```

### 7.7 客户型号匹配

页面上提供客户型号输入框。实现 `matchCustomerModel()`：

```javascript
function matchCustomerModel(inputModel, positions) {
  // 1. 从输入型号中逐位置匹配规格值代码
  // 2. 每个位置渲染为独立卡片
  //    - 匹配到 → 绿色边框 ✅，显示匹配的代码和名称
  //    - 匹配不到 → 红色边框 ❌
  // 3. 输出结论："完全匹配" / "部分匹配（差异位置：3、5）"
  //
  // 匹配策略：
  // - 优先精确匹配规格值 code
  // - 如果精确匹配失败，尝试从已选值中匹配
  // - 如果仍未匹配，标记为"无法匹配"
}
```

### 7.9 生成步骤

```

Step 2  读取模板 → Read docs/product-page-template.html

Step 5  填充模板 → 替换 {{PLACEHOLDER}} + 内嵌 PRODUCT_DATA_JSON

Step 7  验证 → 在浏览器中打开确认渲染正常
```

## 异常处理

| 场景 | 处理 |
|---|---|
| 品牌/分类不存在 | 提示先创建（转 brand-expert/category）|
| 选型资料格式无法解析 | 提示用户提供更清晰的资料（截图/Excel）|
| 规格组合数超 100 | 拆分批次上架，避免单次 `quick-create` 超时 |
| quick-create 失败 | 事务回滚，SPU/SKU 都不创建，可重试 |
| 已有同名 SPU | 用 `product spu list --name <名>` 查重，展示给用户决定复用还是改名 |
| 解绑 SPU-供应商误操作 | 警告"会级联清空所有 SKU 供应信息"，需用户明确确认 |
| 必填字段缺失 | 提示用户补充（型号、价格、库存）|
