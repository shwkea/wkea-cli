# Workflow 05：产品开发（研究→配置器→上架）

**触发条件**：
- 用户提供任何产品信息（型号/图片/PDF/网址/选型手册/文字），说"了解""查一下""上架""开发"
- 需求询价流程（workflow 01）中需要做产品研究时，也走本流程的前半段

**视角**：产品（`wkea-product-expert` 主导，供应商阶段 `wkea-vendor-expert` 协作）

**核心原则**：
> 产品开发 = **研究 + 配置器页面**（标配）。上架是可选后续。
> **唯一交付物就是产品配置器页面**（`/tmp/wkea-product-page-{slug}.html`），无论"了解"还是"上架"。
> 不要生成"上架报告"、"listing 报告"等额外格式——配置器页面本身已包含全部信息：
> 产品头部、型号结构、交互式配置器、价格参考、SKU 列表、供应信息、资料附件（含 PDF 链接）、操作入口。

**与 02 的区别**：
- **02**：已有产品模板 → 批量套用上架（多品牌多供应商并行）
- **05**：从零开始 → 研究 → 配置器 → 上架（本 workflow）

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 信息收集（系统搜索 + 资料提取 + 网上补全）
- Phase 2 型号结构解析（可变/固定/分隔符）
- Phase 3 完整性检查（对照数据库必填字段）
- Phase 4 产品配置器页面（标配，不论是否上架）
- Phase 5 按意图分流

**"了解/查一下"结束点** = Phase 4 完成 + Phase 5 反问下一步。

**"上架"继续**：
- Phase 6 SPU + 规格创建
- Phase 7 SKU 上架
- Phase 8 供应绑定
- Phase 9 产品资料完善
- Phase 10 验证 + 更新配置器页面为完整模式

**异步后续**：无。

## Phase 1：信息收集（多源搜索）

由 `wkea-product-expert` 主导。**每个产品独立处理，不合并。**

### Step 1.1 系统内搜索

```bash
# ES 搜索型号、缩短型号、去分隔符变体
node dist/index.js product sku list --keyword <型号>
node dist/index.js product spu list --keyword <关键词>
```

- 搜到了 → 记录已有 SKU ID、价格、品牌、规格参数
- 没搜到 → 标记"系统无此产品"

### Step 1.2 提取用户提供的资料

从用户给的 PDF、图片、文字、网址中提取：品牌、分类、型号、规格参数。

所有资料统一用 kimi-webBridge 操作浏览器打开查看。

### Step 1.3 网上补全（kimi-webBridge → Google）

**搜索铁律（三轮搜索）**：

| 轮次 | 搜什么 | 禁止 |
|------|--------|------|
| **第一轮：原样搜** | 只用用户给的原始型号，不加 AI 猜测的品牌/分类 | ❌ 猜品牌后搜"欣三立 FKC-G02" |
| **第二轮：品牌验证** | 第一轮结果中出现的品牌 → `{型号} + {品牌}` + datasheet | ❌ 跳过第一轮 |
| **第三轮：深挖** | 进官网产品中心确认型号存在，提取 datasheet 完整规格参数 | ❌ 一个 B2B 页面就下结论 |

搜索引擎：**Google**（首选，中英文双关键词）→ 搜不到换 **Bing**。所有操作通过 kimi-webBridge 浏览器完成。

### Step 1.4 B2B 交叉验证

用 kimi-webBridge 打开 1688、淘宝、爱采购、icspec、allchips 搜同型号：
- 记录价格区间、是否有现货、供应商数量
- 多个平台都搜不到 → 标记"全网信息稀少"

### Step 1.5 图片收集

从官网、datasheet、B2B 平台收集产品图片 URL。

## Phase 2：型号结构解析（核心）

由 `wkea-product-expert` 执行。

### Step 2.1 判定产品类型

```
有选型配置表 / 型号含多段可变代码？
├─ 是 → 多规格产品 → 继续 Phase 2.2 解析型号结构
└─ 否 → 单产品（无规格体系）→ 跳到 Phase 3
```

### Step 2.2 逐位置解析

按"位置"逐位分析型号，每个位置判定为以下三类之一：

| 类型 | 标识 | 示例 |
|------|------|------|
| **可变规格**（`is_fixed=false`）| 存在多个可选值 | 主体尺寸 20/30/40/60 |
| **固定规格**（`is_fixed=true`）| 仅一个固定值 | FKC、AMS、X2044 |
| **分隔符** | 位置之间的连接符 | `-`、`.`、`/`、空格 |

### Step 2.3 关键判定原则

| 场景 | 判定 |
|------|------|
| 有完整选型配置表 | 型号中**所有内容**都是规格（含固定规格和分隔符）|
| 参数参与型号拼接 | → 规格 |
| 参数不参与型号拼接 | → 属性 |
| **值变了型号会变** | → 规格（可变或固定）|
| **值变了型号不变** | → 属性 |
| 型号中有固定值片段（如 AMS、X2044）| → 固定规格（`is_fixed=true`）|

### Step 2.4 输出结构化规格表

```
位置 / 类型 / 名称 / 代码 / 可选值列表
作为后续配置器页面 + 系统上架的共同输入
```

## Phase 3：完整性检查

由 `wkea-product-expert` 执行。对照系统必填字段，列出"已有"和"缺什么"。

### Step 3.1 检查清单

| 维度 | 检查项 | 已有 | 缺失 |
|------|--------|------|------|
| 品牌 | 品牌名 + 品牌 ID | | |
| 分类 | 分类名 + 分类 ID | | |
| 型号结构 | 位置表（2.4 输出）| | |
| datasheet | PDF 链接 | | |
| 图片 | 产品图片 URL | | |
| 价格 | B2B 参考价或系统售价 | | |
| 规格值 | 每个可变规格的所有可选值 | | |

### Step 3.2 判定

- ✅ 资料齐全 → 产品开发成功，准备上架 / 生成配置器页面
- ⚠️ 部分缺失 → 标注缺失项，继续（不影响生成配置器页面）
- ❌ 品牌+分类都缺 → 暂停问用户

## Phase 6：系统上架 — SPU + 规格创建

> 以下 Phase 6-10 仅当用户意图为"上架"时执行。

由 `wkea-product-expert` 执行。

### Step 6.1 创建 SPU

**前置条件**：品牌和分类必须已存在

```bash
node dist/index.js product spu create \
  --name "<SPU 名称，不含品牌>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --series "<系列>" \
  --description "<描述>" \
  --pdf-link "<datasheet 链接>" \
  --details "<详情介绍>"
```

**命名规则（铁律）**：SPU 名称**不能包含品牌名**（品牌是绑定的）

### Step 6.2 单产品 vs 多规格产品

```
Phase 2 判定的产品类型？
├─ 单产品（无规格）→ 直接跳到 Phase 7 创建 SKU
└─ 多规格产品 → 继续 Step 6.3 创建规格体系
```

### Step 6.3 规格创建：三种类型 + 完整 CLI 命令

型号由三部分组成：**可变规格 + 固定规格 + 分隔符**。创建时必须区分处理。

#### 概念对照表

| 类型 | 定义 | 命令行标记 | 选择器 |
|------|------|-----------|--------|
| **可变规格** | 多个可选值 | 不加 `--fixed` | ✅ 可切换 |
| **固定规格** | 型号中唯一值、不会变的片段，也是规格 | `--fixed` / `--is-fixed` | 🔒 锁定不可选 |
| **分隔符** | 位置之间的连接符，独立于规格 | `separator set` | 纯显示 |

#### 推荐方式：`product spec add`（V1，一步完成创建+绑定+标记固定）

**可变规格示例**（多个可选值）：
```bash
node dist/index.js product spec add \
  --spu-id <spuId> \
  --name "凸轮方向" \
  --tag "DIR" \
  --sort 2 \
  --param '[{"name":"A (推轮在左)","tag":"A","sort":1},{"name":"B (推轮在右)","tag":"B","sort":2},{"name":"R (滚轮)","tag":"R","sort":3}]'
```

**固定规格示例**（只有一个值，必须加 `--fixed`）：
```bash
node dist/index.js product spec add \
  --spu-id <spuId> \
  --name "系列" \
  --tag "SERIES" \
  --sort 1 \
  --fixed \
  --param '[{"name":"FKC","tag":"FKC","sort":1}]'
```

⚠️ **固定规格不加 `--fixed` = 普通可变规格**。这是最常见的上架错误。

#### 备选方式：`product spu spec create` + `param add`（V2，两步）

```bash
# 1. 创建规格（固定规格加 --is-fixed）
node dist/index.js product spu spec create --spu-id <id> --name "系列" --manage-name "SERIES_<spuId>" --sort 1 --is-fixed
# 2. 添加规格值
node dist/index.js product spec param add --spec-id <specId> --name "FKC" --tag "FKC" --sort 1
# 3. 绑定到 SPU（注意：spec bind 不支持 --is-fixed）
node dist/index.js product spu spec bind --spu-id <id> --spec-id <specId>
```

### Step 6.4 规格值 name/tag 合规校验（必做）

| 校验项 | 通过条件 | 违规示例 |
|-------|---------|---------|
| `tag` 不含空格 | `"G02"` | ❌ `"G02 1/4\""` |
| `tag` 不含中文 | `"R"` | ❌ `"R 滚轮"` |
| `tag` 不含单位/描述 | `"001"` | ❌ `"001 0.1~1 l/min"` |
| `name` 含描述 | `"G02 (1/4\")"` | ❌ `"G02"`（缺描述）|
| 固定规格 isFixed=true | 用 spec list 验证 | 补调 `spec update --is-fixed` |

**自检**：`node dist/index.js product spu spec list --spu-id <id>` → 检查每个 specParams 的 tag + isFixed

### Step 6.5 设置分隔符

分隔符枚举 ID（`enum --type 分隔符`）：101=`-`, 102=`*`, 103=`/`, 104=` `, 463=`+`, null=无

```bash
node dist/index.js product spu separator set \
  --spu-id <id> \
  --spec-fg "-,null,-,-" \
  --spec-fg-ids "101,null,101,101"
```
`spec-fg` 数组长度 = 规格数量（每个规格后面跟一个分隔符）。

### Step 6.6 验证

```bash
node dist/index.js product spu spec list --spu-id <id>    # 规格绑定列表
node dist/index.js product spu spec model --spu-id <id>   # 型号结构预览
node dist/index.js product spu separator get --spu-id <id> # 分隔符配置
node dist/index.js product spu get --spu-id <id>           # SPU 完整信息
```

## Phase 7：SKU 上架

由 `wkea-product-expert` 执行。

### Step 7.1 选择创建方式

| 场景 | 推荐方式 | 关键命令 |
|------|---------|---------|
| 有完整数据（名称+品牌+分类+型号+价格+库存）| **quick-create**（具体 SKU 模式）| `product quick-create` |
| 只有规格定义，无具体 SKU 数据 | quick-create 变型模式（只建 SPU + 规格）| `product quick-create --specs` |
| 已有 SPU，新增 SKU | 单独创建 | `product sku create` |
| 枚举所有规格值组合批量生成 | 有限组合一次性铺 | 逐 SKU 创建 |

### Step 7.2 quick-create 两种模式

- **变型模式**：只创建 SPU + 规格定义，**不生成 SKU**
- **具体 SKU 模式**：创建 SPU + 规格 + 直接生成有型号/价格/库存的 SKU

```bash
node dist/index.js product quick-create \
  --spu-name "<SPU 名称>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --specs '{"主体尺寸":["20","30","40"],"螺纹":["M"]}' \
  -s '{"name":"SKU1","model":"AMS-20-M-X","salesPrice":100,"purchasePrice":50,"stock":10}' \
  -s '{"name":"SKU2","model":"AMS-30-M-X","salesPrice":120,"purchasePrice":60,"stock":5}'
```

### Step 7.3 SKU 关键字段

| 字段 | 说明 | 示例 |
|------|------|------|
| `name` | SKU 名称（必填）| `品牌名 + SPU名 + 型号` |
| `model` | 型号 | `AMS-20-M-X` |
| `salesPrice` / `purchasePrice` | 售价/采购价 | 100 / 50 |
| `stock` | 库存 | 10 |
| **`unit`** | **单位（必填，默认 469=pcs）** | 469 |
| `images` | 图片集合 | `url1,url2` |
| `purchaseLink` | 采购链接 | `https://...` |

### Step 7.4 验证

```bash
node dist/index.js product sku get --sku-id <id>
node dist/index.js product sku list --spu-id <id>
```

## Phase 8：供应绑定

由 `wkea-product-expert` 主导（`wkea-vendor-expert` 协作）。

### Step 8.1 SPU-供应商绑定（先）

```bash
node dist/index.js product spu bind-vendor --spu-id <id> --vendor-id <id>
```

### Step 8.2 SKU-供应信息（再）

```bash
node dist/index.js product sku supply set \
  --sku-id <id> \
  --vendor-id <id> \
  --purchase-price <price> \
  --min-order <quantity> \
  --delivery-days <days>
```

⚠️ 解绑 SPU-供应商会**级联清空**该 SPU 下所有 SKU 的供应信息，操作前务必备份。

## Phase 10：验证与交付

由 `wkea-product-expert` 执行，主理人汇编。

### Step 10.1 验证清单

- [ ] SPU ID 已生成
- [ ] 规格已绑定，固定规格 isFixed=true（`product spu spec list`）
- [ ] 型号结构正确（`product spu spec model`）
- [ ] 分隔符已设置（`product spu separator get`）
- [ ] SKU 列表完整（`product sku list --spu-id <id>`）
- [ ] 供应信息已绑定（`product spu vendors` + `product sku supply list`）
- [ ] 产品资料完整（描述、PDF、图片）
- [ ] 属性已维护

### Step 10.2 更新配置器页面为完整模式

Phase 4 生成的是研究模式页面（无系统 SKU 数据）。上架完成后，**必须重新生成**配置器页面，更新为完整模式：
- 补充系统 SKU ID、价格、库存、状态
- 补充供应商绑定信息
- 补充后台跳转链接（用 `node dist/index.js urls` 获取 manageMainUrl 拼接）
- 更新 `{{MODE_CLASS}}` 为 `full`、`{{MODE_LABEL}}` 为 `完整模式`

**唯一交付物就是产品配置器页面**：`/tmp/wkea-product-page-{slug}.html`。
不要生成其他格式的上架报告，配置器页面本身已经包含所有信息（头部、规格体系、配置器、SKU 列表、供应商、资料附件、操作入口）。
如果有"待补充"项，直接写在配置器页面的操作入口区块中。

## 分支流程

### 分支 A：仅建规格体系

> **触发**：用户只想先建 SPU 规格框架，后续分批补 SKU。

- 走 Phase 1→6（建到规格，跳 Phase 7-9）
- 输出 SPU 规格结构 + 配置器页面
- 后续任意时刻调用 `product sku create` 补 SKU

### 分支 B：仅绑定供应信息

> **触发**：SPU+SKU 已存在，仅补供应商/价格。

- 跳 Phase 1-7，直接走 Phase 8（供应绑定）

### 分支 C：停产替代

> **触发**：产品停产或被新系列替代。

```bash
# SPU 级停产
node dist/index.js product spu update --spu-id <id> --stop-production <替代SPU_ID>
# SKU 级替代
node dist/index.js product sku replace add --sku <SKU> --replace-sku <SKU> [--full-replace]
```

## 异常处理

| 场景 | 处理 |
|------|------|
| 品牌/分类不存在 | 转 brand-expert 创建 |
| 选型资料无法解析 | 提示用户提供更清晰资料 |
| 规格组合数超 100 | 拆分批次上架 |
| quick-create 失败 | 事务回滚，可重试 |
| 已有同名 SPU | 查重展示，用户决定复用/改名 |
