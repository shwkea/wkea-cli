# Workflow 05：产品配置与上架

**触发条件**：
- 用户提供厂商选型手册/型号对照表/选型配置表，要"配置化上架"
- 用户说"上架一个产品系列"且提供了多个型号 SKU 数据
- 需求询价处理中发现产品规格复杂，需要完整建模后再上架

**视角**：产品（`wkea-product-expert` 主导，需求/供应商视情况协作）

**与 02 的区别**：
- **02**：上架视角（已有产品模板 → 批量套用）—— Phase 1 vendor + product **并行**
- **05**：配置视角（选型资料 → 规格建模 → SKU 变型）—— Phase 1-2 product **主导**

---

## 流程边界

**主流程**（同步完成）：
- Phase 1 选型资料解析
- Phase 2 SPU + 规格创建
- Phase 3 SKU 上架
- Phase 4 供应绑定
- Phase 5 产品资料完善
- Phase 6 验证与交付

**主流程结束点** = Phase 6 输出"已上架"清单和"待补充"清单。

**异步后续**：无（本 workflow 不涉及等供应商回复等异步场景）。

---

## Phase 1：选型资料接收 + 型号解析

由 `wkea-product-expert` 主导。

### Step 1.1 接收资料

接收用户提供的**选型手册/型号对照表/SPU+规格定义**：
- 支持 Markdown 表格、Excel 截图、纯文本、PDF 链接
- 识别产品名、品牌、分类、型号结构（如 `AMS[1]X-[2][3]-[4]L[5]-X2044`）
- 若资料缺失品牌/分类/型号结构之一，**先反问用户确认再继续**

### Step 1.2 型号结构解析（核心）

按"位置"逐位分析型号，每个位置判定为以下三类之一：

| 类型 | 标识 | 示例 |
|------|------|------|
| **可变规格**（`is_fixed=false`）| 存在多个可选值 | 主体尺寸 20/30/40/60 |
| **固定规格**（`is_fixed=true`）| 仅一个固定值 | AMS、X、L、X2044 |
| **分隔符** | 位置之间的连接符 | `-`、`.`、`/`、空格 |

输出**结构化规格表**（位置 / 类型 / 规格值列表 / 名称），作为 Phase 2 规格建模的输入。

### Step 1.3 关键判定原则

| 场景 | 判定 |
|------|------|
| 有完整选型配置表 | 型号中**所有内容**都是规格（含固定规格和分隔符）|
| 参数参与型号拼接 | → 规格 |
| 参数不参与型号拼接 | → 属性 |
| **值变了型号会变** | → 规格（可变或固定）|
| **值变了型号不变** | → 属性 |
| 型号中有固定值片段（如 AMS、X2044）| → 固定规格（`is_fixed=true`）|

### Step 1.4 工具选择

- 简单 HTML/静态列表 → `web_fetch` 足够
- 复杂表格 + 分页 + 动态加载 → 浏览器自动化
- PDF 名录 → 浏览器下载 → 转文本 → 解析

`web_fetch` 拿不到时升级浏览器（不限制死工具）。

## Phase 2：SPU + 规格创建

由 `wkea-product-expert` 执行。

### Step 2.1 创建 SPU

**前置条件**：品牌和分类必须已存在

```bash
node dist/index.js product spu create \
  --name "<SPU 名称，不含品牌>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --series "<系列>" \
  --description "<描述>" \
  --pdf-link "<datasheet 链接>" \
  --details "<详情介绍>"
```

**命名规则（铁律）**：SPU 名称**不能包含品牌名**（品牌是绑定的）

### Step 2.2 创建规格值 + 绑定到 SPU

- 为每个**可变规格**创建多个规格值
- 为每个**固定规格**创建一个规格值（`is_fixed=true`）
- 按位置顺序绑定到 SPU

```bash
# 创建规格值
node dist/index.js product spec value create

# 绑定规格到 SPU
node dist/index.js product spu spec bind
```

### Step 2.3 设置分隔符

```bash
node dist/index.js product spu separator set
```

分隔符存储在 SPU 级别 JSON 配置中，独立管理。

### Step 2.4 验证

```bash
node dist/index.js product spu spec list     # 查看规格
node dist/index.js product spu spec model    # 查看型号结构
```

## Phase 3：SKU 上架

由 `wkea-product-expert` 执行。

### Step 3.1 选择创建方式

| 场景 | 推荐方式 | 关键命令 |
|------|---------|---------|
| 有完整数据（名称+品牌+分类+型号+价格+库存）| **quick-create**（具体 SKU 模式）| `product quick-create` |
| 只有规格定义，无具体 SKU 数据 | quick-create 变型模式（只建 SPU + 规格）| `product quick-create --specs` |
| 已有 SPU，新增 SKU | 单独创建 | `product sku create` |
| 枚举所有规格值组合批量生成 | 4×3×2=24 个有限组合一次性铺 | 脚本 + `product spu spec model` 模板 |

**quick-create 两种模式**：
- **变型模式**：只创建 SPU + 规格定义，**不生成 SKU**
- **具体 SKU 模式**：创建 SPU + 规格 + 直接生成有型号/价格/库存的 SKU

### Step 3.2 创建 SKU（以具体 SKU 模式为例）

```bash
node dist/index.js product quick-create \
  --spu-name "<SPU 名称>" \
  --brand-id <brandId> \
  --category-id <categoryId> \
  --vendor-id <vendorId> \
  --specs '{"主体尺寸":["20","30","40"],"螺纹":["M"]}' \
  -s '{"name":"SKU1","model":"AMS-20-M-X","salesPrice":100,"purchasePrice":50,"stock":10}' \
  -s '{"name":"SKU2","model":"AMS-30-M-X","salesPrice":120,"purchasePrice":60,"stock":5}'
```

### Step 3.3 SKU 扩展字段（30+ 字段）

通过 `-s` JSON 传递，关键字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| `name` | SKU 名称（必填）| `品牌名 + SPU名 + 型号` |
| `model` | 型号 | `AMS-20-M-X` |
| `specs` | 规格值 | `{"主体尺寸":["20"]}` |
| `attributes` | 属性列表 | `[{"name":"材质","value":"不锈钢"}]` |
| `salesPrice` / `purchasePrice` | 售价/采购价 | 100 / 50 |
| `stock` | 库存 | 10 |
| `images` | 图片集合 | `url1,url2,url3` |
| `purchaseLink` | 采购链接 | `https://...` |
| `manufacturerModel` | 制造商型号 | `AMS-20` |
| `minOrderQuantity` | 最小起订量 | 1 |
| `lengthWidthHeight` | 长宽高 | `100x50x30cm` |
| `weight` | 重量(kg) | 0.5 |
| `deliveryMethod` | 发货方式（枚举）| 1 |
| `purchaseDeliver` | 采购交期（枚举）| 7 |
| `barcode` | 条码 | `6901234567890` |
| `itemNumber` | 货号 | `AMS-001` |
| `life` | 质保期(天) | 365 |
| `safetyStock` / `ceilingStock` | 库存上下限 | 5 / 100 |
| `taxRate` / `purchaseTaxRate` | 销售/采购税率（枚举）| 13 |
| `templateId` | 运费模板 | 1 |
| `esKeyword` | ES 搜索关键词 | `AMS 气缸` |

### Step 3.4 验证

```bash
node dist/index.js product sku get --sku-id <id>      # 单 SKU 验证
node dist/index.js product sku list --spu-id <id>     # SPU 下所有 SKU
```

## Phase 4：供应绑定

由 `wkea-product-expert` 主导（`wkea-vendor-expert` 协作）。

### Step 4.1 SPU-供应商绑定（先）

```bash
node dist/index.js product spu bind-vendor --spu-id <id> --vendor-id <id>
```

### Step 4.2 SKU-供应信息

```bash
node dist/index.js product sku supply set \
  --sku-id <id> \
  --vendor-id <id> \
  --purchase-price <price> \
  --min-order <quantity> \
  --delivery-days <days>
```

### Step 4.3 优势分类（可选）

如该供应商擅长该分类：

```bash
node dist/index.js vendor superior-category add \
  --vendor-id <id> \
  --name "气动元件" \
  --system-category-path "1/5/23" \
  --priority 1
```

### Step 4.4 解绑级联

⚠️ 解绑 SPU-供应商会**级联清空**该 SPU 下所有 SKU 的供应信息，**务必提前备份**。

## Phase 5：产品资料完善

由 `wkea-product-expert` 执行。

### Step 5.1 SPU 级资料字段

| 字段 | CLI 参数 | 说明 |
|------|---------|------|
| 描述 | `--description` | 简短描述 |
| 详情介绍 | `--details` | 富文本 |
| PDF 链接（datasheet）| `--pdf-link` | datasheet URL |
| 系列 | `--series` | 系列产品名 |
| 图片 | `--images` | 逗号分隔多图 URL |
| 型号备注 | `--model-remark` | 选型备注 |
| 资质文件 | `--qualification-path` | 资质文件路径 |
| 信息文件 | `--information-files` | 多个信息文件 |
| 标签 | `--labels` | 标签 ID 列表 |
| ES 搜索关键词 | `--es-keyword` | 搜索优化 |
| 维嘉折扣 | `--wkea-discount` | 折扣率 |
| 是否可退货 | `--can-be-returned` | bool |

```bash
node dist/index.js product spu update \
  --spu-id <id> \
  --description "<描述>" \
  --pdf-link "<datasheet URL>" \
  --details "<富文本>" \
  --images "url1,url2" \
  --es-keyword "AMS 气缸"
```

### Step 5.2 属性管理

不参与型号拼接的参数维护为属性：

| 类型 | 例子 | 原因 |
|------|------|------|
| 产地 | 江苏苏州 | 值变了型号不变 |
| 材质 | 不锈钢 | 值变了型号不变 |
| 保修年限 | 1 年 | 值变了型号不变 |
| 适用场景 | 工业自动化 | 值变了型号不变 |

```bash
node dist/index.js product attribute set \
  --spu-id <id> \
  --attributes '[{"name":"产地","value":"江苏苏州"},{"name":"材质","value":"不锈钢"}]'
```

### Step 5.3 标签管理

```bash
node dist/index.js product spu update --spu-id <id> --labels "1,2,3"
```

## Phase 6：验证与交付

由 `wkea-product-expert` 主导，主理人汇编。

### Step 6.1 验证清单

- [ ] SPU ID 已生成
- [ ] 规格已绑定（`product spu spec list`）
- [ ] 型号结构正确（`product spu spec model`）
- [ ] SKU 列表完整（`product sku list --spu-id <id>`）
- [ ] 供应信息已绑定（`product spu vendors` + `product sku supply list`）
- [ ] 优势分类已设置（如适用）
- [ ] 产品资料完整（描述、PDF、图片、标签）
- [ ] 属性已维护

### Step 6.2 交付输出

```
- SPU ID: <id>
- SKU 列表: <id1, id2, id3>（含型号/价格/库存）
- 规格结构: 22 位置（可变/固定/分隔符）
- 供应商: <vendorId>（已绑定）
- 跳转链接: {manageMainUrl}#/main/product-group-list?id=<spuId>
```

### Step 6.3 "待补充"清单

输出**未完成**的项目：
- 缺 PDF 链接
- 缺图片
- 缺属性
- 缺替代品
- 缺优势分类
- 库存为 0

---

## 派单契约

主理人调度本 workflow 时，**必须**在 dispatch prompt 中附：

1. **该 expert 负责的 Phase**（product-expert 全程；vendor-expert 在 Phase 4 协作；demand-expert 选品阶段可选）
2. **资料类型**（选型手册/型号对照表/纯文本）
3. **预期 SKU 数量**（影响是否走批量枚举）
4. **是否包含配置器预览**（默认不包含，WKEA 有自己的产品 UI）

---

## 异常处理

| 场景 | 处理 |
|---|---|
| 品牌/分类不存在 | 提示先创建（转 brand-expert/category）|
| 选型资料格式无法解析 | 提示用户提供更清晰的资料（截图/Excel）|
| 规格组合数超 100 | 拆分批次上架，避免单次 `quick-create` 超时 |
| quick-create 失败 | 事务回滚，SPU/SKU 都不创建，可重试 |
| 已有同名 SPU | 用 `product spu list --name <名>` 查重，展示给用户决定复用还是改名 |
| 解绑 SPU-供应商误操作 | 警告"会级联清空所有 SKU 供应信息"，需用户明确确认 |
| 必填字段缺失 | 提示用户补充（型号、价格、库存）|

---

## 与同事 product-configurator 的差异

**吸收**：
- 规格建模方法论（值变了型号变 → 规格）
- 选型资料解析 SOP（Phase 1）
- 三种创建方式（quick-create / 单独 / 批量）
- 三层绑定（SPU-供应商 → SKU-供应信息 → 优势分类）
- 30+ 产品资料字段

**简化**（不在 WKEA 范围）：
- ❌ HTML 配置器预览生成（22 位置 UI + 18 约束规则 + JS 拼装）
- ❌ CSS 设计令牌 / 部署到外网（cloudstudio_deploy）
- ❌ 5 件套输出物结构（HTML + 4 个 JSON）

WKEA 已有自己的产品配置 UI，不需要再生成独立的 HTML 配置器。
